# KV Cache Eviction 정확도 벤치마크 리포트

**날짜**: 2026-03-10
**목적**: Base(무제거) vs Sliding Window vs H2O vs H2O+ 네 정책의 정확도 비교
**모델**: Llama 3.2 1B, Llama 3.2 3B (Q4_0 양자화)
**환경**: 호스트 PC (x86_64 AVX2, CPU 백엔드)

---

## 1. 실험 설계

| 파라미터 | 값 |
|----------|-----|
| 디코드 토큰 | 256 |
| Max seq len | 512 |
| Eviction 트리거 | 디코드 토큰 128 (memory_critical) |
| Eviction ratio | 0.50 (캐시의 50% 유지) |
| H2O/H2O+ keep_ratio | 0.5 |
| H2O/H2O+ decay | 0.0 |
| H2O/H2O+ scoring | 시간 정규화 (기본값, `score / steps_since_entry`) |
| 샘플링 | greedy (temperature=0) |
| 프롬프트 | PPL-01 (Literary), PPL-03 (Technical) |

**정책 차이점:**
- **Sliding Window**: 최신 N 토큰만 유지, 가장 오래된 토큰 제거
- **H2O**: attention score 기반 heavy-hitter 선별 (query head 집계)
- **H2O+**: GQA-aware per-KV-head 독립 score 추적 (query head → KV head 매핑 후 KV head별 집계)

**Eviction 흐름**: 프롬프트 (~50 토큰) prefill → 128 토큰 디코드 → memory_critical 신호 → 캐시 50% 제거 → 128 토큰 추가 디코드

## 3. 분석

### 3.1 정책 간 정확도 순위

**1B 모델 — H2O+가 확실한 우위:**

| 메트릭 | Sliding | H2O | H2O+ | H2O+ vs Sliding |
|--------|--------:|----:|-----:|----------------:|
| EMR (PPL01) | 0.514 | 0.529 | **0.612** | **+0.098** |
| Suffix EMR (PPL01) | 0.016 | 0.008 | **0.189** | **+0.173** |
| ROUGE-L (PPL01) | 0.602 | 0.653 | **0.718** | **+0.116** |
| BLEU-4 (PPL01) | 0.544 | 0.607 | **0.667** | **+0.123** |
| EMR (PPL03) | 0.510 | 0.510 | **0.529** | +0.019 |
| ROUGE-L (PPL03) | 0.550 | 0.551 | **0.639** | **+0.089** |

- H2O+가 PPL01(Literary)에서 압도적 우위. EMR이 0.514→0.612로 **+19%p** 향상.
- 특히 Suffix EMR 0.189는 eviction 후에도 부분적 복구가 가능함을 의미. Sliding/H2O는 0.008~0.016으로 사실상 복구 불가.
- PPL03(Technical)에서도 H2O+가 일관되게 우위. FDT 134로 5토큰 더 오래 baseline 유지.

**3B 모델 — 정책 간 차이 미미:**

| 메트릭 | Sliding | H2O | H2O+ |
|--------|--------:|----:|-----:|
| EMR (PPL01) | 0.510 | 0.510 | 0.514 |
| EMR (PPL03) | 0.514 | 0.522 | 0.522 |
| ROUGE-L 평균 | 0.624 | 0.612 | 0.607 |
| Top-K 평균 | 0.618 | 0.607 | 0.609 |

- 3B에서는 세 정책 모두 비슷한 수준. 3B 모델의 높은 eviction 내성으로 인해 정책 차이가 상쇄됨.
- Sliding이 ROUGE-L/Top-K에서 미세하게 우위인 것은, 3B에서 최근 토큰의 정보량이 충분하여 score 기반 선별의 이점이 줄어들기 때문으로 보임.

### 3.2 H2O+ vs H2O: Per-Head 집계의 효과

H2O+는 GQA 구조에서 KV head별로 독립 score를 추적하여, 같은 KV head를 공유하는 query head들의 서로 다른 attention 패턴을 더 정확히 반영한다.

| 비교 (1B 평균) | H2O | H2O+ | 차이 |
|---------------|----:|-----:|-----:|
| EMR | 0.520 | **0.571** | **+0.051** |
| Suffix EMR | 0.008 | **0.099** | **+0.091** |
| ROUGE-L | 0.602 | **0.679** | **+0.077** |
| BLEU-4 | 0.553 | **0.626** | **+0.073** |
| Top-K Overlap | 0.549 | **0.594** | **+0.045** |

1B 모델에서 H2O+ 대비 H2O의 정확도 격차가 크다. 이는 **1B의 GQA 구조(32 query heads → 8 KV heads, 4:1 그룹)에서 query head를 일괄 집계하면 개별 KV head의 중요 토큰이 가려지는 현상**이 H2O+의 per-head 추적으로 해소됨을 의미한다.

### 3.3 모델 크기별 Eviction 내성

| 메트릭 (4정책 평균) | 1B | 3B | 차이 |
|---------------------|----:|----:|-----:|
| EMR | 0.527 | 0.516 | -0.011 |
| ROUGE-L | 0.613 | 0.618 | +0.005 |
| Post-FDT Top-K | 0.108 | 0.215 | **+0.107** |
| Suffix EMR | 0.055 | 0.019 | -0.036 |

3B는 post-FDT Top-K에서 1B보다 2배 높은 값을 보이며, eviction 후에도 유사한 토큰 분포를 유지한다. 단, 1B에서 H2O+의 Suffix EMR이 높아 전체 평균이 올라간 것에 주의.

### 3.4 속도 오버헤드 비교

| Model | Sliding | H2O | H2O+ |
|-------|--------:|----:|-----:|
| 1B | +1.0~1.5% | +18~25% | **+28~49%** |
| 3B | -1.2~-1.3% | +14~16% | **+15~18%** |

- H2O+는 H2O보다 per-head score 추적이 추가되어 1B에서 최대 49% 오버헤드 발생.
- 3B에서는 matmul 비중이 커져 상대적 오버헤드가 줄어듦 (15-18%).
- Sliding은 score 추적이 없어 사실상 오버헤드 없음.

### 3.5 시간 정규화 효과 (Round 15 대비)

| | Round 15 (raw SUM) | 이번 벤치마크 (time-normalized) |
|---|---|---|
| H2O vs Sliding (1B EMR) | -0.013 (열위) | +0.010 (우위) |
| 원인 | 누적 바이어스: 오래된 토큰 과대평가 | 단위 시간당 중요도로 공정 비교 |

**시간 정규화가 H2O/H2O+ 모두의 토큰 선택 품질을 실질적으로 개선**했다.

## 4. 결론

1. **H2O+가 1B에서 최고 성능** — EMR 0.612 (Sliding 0.514 대비 +19%p), Suffix EMR 0.189로 eviction 후 유일하게 부분 복구 가능. Per-head GQA-aware 집계가 핵심 기여.
2. **3B에서는 정책 간 차이 미미** — 큰 모델은 eviction에 견고하여 단순한 Sliding으로도 충분. 복잡한 score 기반 정책의 이점이 상쇄됨.
3. **H2O+는 정확도 최우선 시 권장, Sliding은 속도 최우선 시 권장**:
   - 정확도 우선: H2O+ (EMR +19%p, 단 +28~49% 오버헤드)
   - 속도 우선: Sliding (오버헤드 없음, 기본적 정확도)
   - 균형: H2O (+18~25% 오버헤드로 약간의 정확도 개선)
4. **시간 정규화가 score 기반 정책의 전제 조건** — 미적용 시 H2O가 Sliding보다 열위한 역전 현상 발생.
5. **Eviction 후 발산은 공통 과제** — Suffix EMR이 H2O+를 제외하면 0.008~0.032으로, 한번 발산하면 복구가 어려움.

## 6. 부수 성과

이번 벤치마크 수행 과정에서 다음 코드 변경이 이루어짐:

- **Multi-shard safetensors 로딩** (`engine/src/models/llama/llama_model.rs`): `model.safetensors.index.json`을 파싱하여 여러 shard 파일에서 텐서를 로드. 3B 이상 모델 지원.
- **k_dim 계산 버그 수정** (`engine/src/bin/generate.rs`): `hidden_size / 4` 하드코딩 → `num_kv_heads * head_dim`. 1B에서는 우연히 맞았으나 3B에서 panic 발생 원인이었음.
