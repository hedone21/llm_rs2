# Round 8: 대규모 시퀀스 H2O vs Sliding 비교 리포트

## 실험 설계

- **시퀀스 길이**: 2048, 4096, 8192
- **Eviction 위치**: 각 길이의 50% 지점 (Memory Critical, target_ratio=0.50)
- **비교 정책**: H2O (per-layer MAX, kr=0.5, decay=0.0, all layers) vs Sliding Window
- **조건**: greedy decoding, CPU backend, Q4_0 KV cache

## 핵심 발견

### 1. H2O와 Sliding은 대규모 시퀀스에서 동일한 품질

4096과 8192 토큰에서 EMR, FDT, ROUGE-L, BLEU-4가 **완전히 동일**.
동일한 수의 토큰을 evict하고 동일한 최종 cache 크기를 유지.

**의미**: 대규모 시퀀스에서 50% eviction 시, HH 토큰 선택이 sliding window와
동일한 토큰 집합을 보존함. 즉 "가장 중요한 토큰"이 곧 "가장 최근 토큰"과 일치.

### 2. 1024 토큰에서만 H2O가 극적으로 우위

| | H2O | Sliding |
|---|-----|---------|
| EMR | 0.990 | 0.526 |
| Δ | — | **-0.464** |

1024 토큰에서 H2O가 Sliding보다 EMR +46.4%p 높음.
이 길이에서는 HH 토큰 선택이 실제로 차별화된 가치를 제공.

### 3. 시퀀스 길이에 따른 EMR 비선형 패턴

| SeqLen | EMR | 해석 |
|--------|-----|------|
| 1024 | 0.990 | eviction 후 500토큰만 생성 → diverge 시간 부족 |
| 2048 | 0.800~0.833 | 중간 수준 |
| 4096 | 0.522 | 최저 — eviction 후 2000+ 토큰 생성 → 충분히 diverge |
| 8192 | 0.939 | 높음 — FDT=7691(94% 지점), 대부분 일치 후 말미에 diverge |

4096에서 EMR이 최저인 이유: eviction 후 남은 토큰이 충분히 많아 diverge가
누적되지만, 8192에서는 eviction 비율(50%)은 동일하나 절대적 recent 토큰 수가
더 많아(~2050 vs ~1030) 컨텍스트 유지가 더 양호.

### 4. H2O 성능 스케일링 문제

전체 레이어 추적의 비용이 시퀀스 길이에 따라 급격히 증가:

| SeqLen | H2O overhead |
|--------|-------------|
| 2048 | -1.3% (동등) |
| 4096 | +10% |
| 8192 | +192% (3x 느림) |

원인: `accumulate_layer()`에서 `scores[h * stride + t]` 접근 패턴이
stride=max_seq_len일 때 CPU L1 캐시 미스를 유발. 전체 16레이어 × 32헤드를
매 스텝 순회.

## 결론

### 품질 측면

1. **대규모 시퀀스 (4k+)에서 H2O와 Sliding은 동일한 품질**을 제공
2. **소규모 시퀀스 (1k)에서 H2O가 극적으로 우수** (EMR +46%p)
3. 50% 일괄 eviction에서 두 방식 모두 ROUGE-L ≥ 0.995, BLEU-4 ≥ 0.990 유지

### 성능 측면

1. **전체 레이어 추적은 8k+ 시퀀스에서 3x 성능 저하** 유발
2. 최적화 방향:
   - `accumulate_layer()` 메모리 접근 패턴 개선 (stride → sequential)
   - 또는 `tracked_layers=4~8`로 추적 레이어 제한 (품질 동등 시)

### 운영 권장

| 시퀀스 길이 | 권장 정책 | 이유 |
|------------|----------|------|
| ≤ 1024 | **H2O** | 극적 품질 우위 (EMR +46%p) |
| 2048~4096 | Sliding | 동등 품질 + 성능 우위 |
| 8192+ | Sliding | 동등 품질 + H2O 3x 느림 |
