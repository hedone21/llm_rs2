0. 이 챕터를 읽는 법

이 코드베이스의 PR 리뷰에서 가장 흔한 지적은 버그가 아니라 용어 오용이다 — "Layer"를 저장 형태에 쓰거나, Stage가 버퍼를 직접 변형하거나, 측정 공간이 다른 QCF 두 값을 직접 비교하는 식이다. 이 챕터는 그런 오용을 막기 위한 용어 사전이며, 각 용어마다 (1) 정의, (2) 실제 코드 심볼 위치, (3) 안티패턴 문장(이렇게 쓰면 틀림)을 단다. 모든 사실은 기준 커밋의 코드를 직접 읽고 확인했다. 아키텍처 전반은 01. 시스템 개요, 모듈 배치는 03. 코드베이스 지도 참조.

1. 세 직교 축 — format ⊥ stage ⊥ hardware

KV·weight 데이터 관리의 모든 용어는 세 축 위의 좌표로 읽으면 정리된다. 한 축에 멤버를 추가해도 다른 축 코드를 건드리지 않도록(비용이 M×N×K 곱이 아니라 M+N+K 합이 되도록) 설계되어 있고, 각 축마다 정식 trait이 코드에 존재한다.

직교성의 코드 증거: 단일 OpenCL backend가 여러 precision 커널을 갖고, q4_0은 CPU·OpenCL·CUDA 세 backend에 모두 존재한다 — precision은 backend의 함수가 아니므로 format과 hardware는 독립 축이다. 단, (format × hardware) paired kernel 행렬 자체는 환원 불가능한 M×N이며, 이 행렬은 Backend trait의 dispatch 내부에 격리되어 호출자·Stage는 어느 커널이 뜨는지 모른다.

복합 동작은 단일축 primitive의 합성으로 읽는다. 예: KIVI는 "KV를 Q2로 다시 까는" stage 동작 + "Q2 표현으로 attention하는" format의 합성이고, weight swap은 stage(로드/언로드) + format(precision 변경)의 합성이다. partition은 format × hardware 곱 — 슬라이스마다 (f16, GPU) / (q4, NPU) 다른 좌표를 가질 수 있다.

2. Format — 저장 형태 (noun)

2.1 KVCacheFormat trait

KV 캐시의 state 책임을 storage-format-agnostic하게 노출하는 base trait. 메서드가 정확히 6개로 고정되어 있다 — geometry 3개(idx / current_pos / capacity), mutation 2개(write_kv / write_kv_batch), attention 1개(attention_into). dtype·codebook·rotation 같은 내부는 의도적으로 표면에 없다 — trait 정의 말미 주석이 as_any()(downcast)·dtype()·needs_attn_scores()의 부재를 명시적 설계 결정으로 박아 두었다.

// engine/src/format/kv_cache_format.rs
pub trait KVCacheFormat: Send + Sync {
    fn idx(&self) -> usize;          // 어느 decoder layer의 KV인가
    fn current_pos(&self) -> usize;   // 유효 토큰 수
    fn capacity(&self) -> usize;      // 물리 버퍼 용량(토큰)
    fn write_kv(&self, new_k: &Tensor, new_v: &Tensor, backend: &dyn Backend) -> Result<()>;
    fn write_kv_batch(&self, new_k: &Tensor, new_v: &Tensor, backend: &dyn Backend) -> Result<()>;
    fn attention_into(&self, q: &Tensor, backend: &dyn Backend, out: &mut Tensor,
                      dims: AttnDims, scores: Option<&mut [f32]>) -> Result<()>;
    // as_any() 없음 — downcast 의도적 차단.
    // dtype() / KVCacheView 없음 — Stage가 어느 Format인지 모름.
}

구현체는 둘 다 pressure/에 산다(guard rail: trait은 format/, impl은 감싸는 캐시 옆): StandardFormat(F32/F16/Q4_0 KVCache를 Mutex로 감싼 wrapper)과 KIVIFormat(KiviCache wrapper). AttnDims는 per-call 파라미터로 n_heads_q(GQA 쿼리 헤드 수)와 window(Gemma3 local SWA)만 싣는다 — kv_heads/head_dim/scale은 전부 format 내부 지식이다. 한때 trait에 있던 compact는 폐기되었고, plan 실행 정본은 execute_kv_plan이다(§3.2).

근거: engine/src/format/kv_cache_format.rs::KVCacheFormat(L58)·::AttnDims, engine/src/pressure/standard_format.rs::StandardFormat(L41), engine/src/pressure/kivi_format.rs::KIVIFormat(L27), engine/src/pressure/eviction/stage_registry.rs::execute_kv_plan(L94)

2.2 WeightFormat trait

KV 쪽과 대칭인 weight 쪽 base trait. idx() + apply_dispatch(d: LayerDispatch, hw: &Hardware) 두 메서드뿐이다. dispatch 모드 타입(LayerDispatch/PartitionShare)은 plugin 결정 표면이라 technique-api crate에 살고 여기서 re-export된다. read용 view()와 apply_storage(spec)은 의도적으로 없다 — precision swap 등 format mutation은 concrete-handle을 든 Stage가 직접 수행한다.

근거: engine/src/format/weight_format.rs::WeightFormat(L21), pub use technique_api::{LayerDispatch, PartitionShare}(같은 파일 L18)

안티패턴:
  • "KV를 Q4 Layer로 저장한다" — ✗. 저장 형태는 Layer가 아니라 Format이다(§4).
  • "이 Stage에서 dtype 분기가 필요하니 KVCacheFormat에 dtype()을 추가하자" — ✗. trait 주석이 부재를 설계 결정으로 명시한다. dtype 분기는 impl 내부에 흡수하거나, 정말 구체 타입이 필요하면 concrete-handle(§5)로 바꾼다.
  • "precision과 레이아웃을 별도 파라미터로 분리하자" — ✗. q4_0은 4bit 블록 레이아웃이자 precision인 한 좌표다. q4와 f16은 이미 다른 paired kernel을 쓰므로 precision이 곧 표현을 가른다.

3. Stage — 관리 동작 (verb)

3.1 KVCacheStage: plan-returning trait

stage 축의 정식 plugin trait. 핵심 계약은 "결정은 plugin, 버퍼 변형 실행은 엔진 독점"이다 — 기법은 StageCtx 읽기 추상을 통해 캐시 상태를 읽고 KVCachePlan을 반환할 뿐, &mut KVCache 같은 엔진 타입을 받지 않는다. 그래서 technique crate는 엔진 내부에 결합하지 않고, 미래 .so C-ABI 승격과도 정합한다(같은 파일에 StageCtxAbi fn-ptr 평탄화가 이미 존재).

// crates/technique-api/src/lib.rs
pub trait KVCacheStage: Send + Sync {
    fn name(&self) -> &str;   // CLI --eviction-policy <name> 매칭
    fn plan(&self, ctx: &dyn StageCtx) -> Option<KVCachePlan>;  // None = no-op
}

pub struct KVCachePlan {
    pub keep: KeepSpec,                // 보존 토큰 모양 (배타)
    pub merges: Vec<WeightedMerge>,    // 가중 병합 지시 (직교, 없으면 빈 Vec)
}
pub enum KeepSpec {
    LayerWide(Vec<usize>),       // sliding/h2o/streaming/d2o — prefix 포함 ascending
    PerHead(Vec<Vec<usize>>),    // h2o_plus — [n_kv_heads][keep], 전 head 길이 동일
}
pub struct WeightedMerge {
    pub into: usize,                   // 가중 합이 누적될 retained 위치
    pub into_weight: f32,              // D2O Eq.11의 w_c
    pub from: Vec<(usize, f32)>,       // evicted (위치, 가중치) — Σfrom + into_weight ≈ 1
}

등록은 linkme distributed_slice로 한다: technique crate가 KV_CACHE_STAGES 슬라이스에 KVCacheStageReg { name, make: fn(StageParams) -> Box<dyn KVCacheStage> }를 제출하고, 엔진이 construction 시 find_stage(name)으로 찾는다 — 엔진 코어의 closed match arm 수정 0으로 새 기법이 추가된다. 실제 plugin 예: crates/techniques/caote/의 static CAOTE: KVCacheStageReg. 상세 확장 절차는 10. 확장 How-to.

StageCtx가 노출하는 읽기 표면: current_pos / target_len(ratio→len 환산은 엔진 책임 — plugin은 환산된 절대 토큰 수만 읽음) / layer_idx / importance()(flat per-token score, None이면 score-free) / n_kv_heads / head_dim, 그리고 단일 텐서 접근 메커니즘 tensor(kind: TensorKind) -> Option<&dyn TensorHandle>(kind = Key/Value/AttnWeights/Scores). dequant_k/dequant_v/head_score/attn_weight는 전부 tensor() 위의 default sugar라 엔진은 tensor() 하나만 구현하면 된다. 호출 간 누적 상태(D2O EMA 등)는 ctx로 thread하지 않고 plugin struct가 interior-mutability로 보유한다.

근거: crates/technique-api/src/lib.rs::KVCacheStage(L220)·::KVCachePlan(L206)·::KeepSpec·::WeightedMerge·::StageCtx(L93)·::KV_CACHE_STAGES(L264), crates/techniques/caote/src/lib.rs(L103 등록 예)

3.2 plan의 실행 — 엔진 독점

sequenceDiagram
    participant E as 엔진 (eviction 트리거 시점)
    participant S as plugin: KVCacheStage
    participant C as StageCtx (엔진이 KVCache 위로 구현)
    participant K as KVCache 버퍼
    E->>S: plan(ctx)
    S->>C: current_pos() / target_len() / importance()
    S->>C: tensor(Key).read_row(pos, head, out) — D2O cosine 등
    S-->>E: Some(KVCachePlan { keep, merges }) 또는 None(no-op)
    E->>K: execute_kv_plan(cache, plan) — merge 적용 후 keep compaction
    Note over E,K: 버퍼 변형은 엔진만 수행. new_pos는 keep.len()에서 도출

plan의 좌표계 규약 두 가지가 리뷰 포인트다: (1) keep은 prefix 포함 ascending이어야 하고(compaction 재배치가 in-place evict와 bit-identical이 되는 조건), (2) merges의 into/from은 compact 적용 직전(pre-compact)의 논리 위치로 해석한다 — merge를 in-place 적용한 뒤 keep compaction을 수행하는 순서(D2O Step 5 → Step 6)다.

근거: crates/technique-api/src/lib.rs::KeepSpec doc("prefix 포함 ascending"), engine/src/format/kv_cache_format.rs::Merge doc(pre-compact 좌표), engine/src/pressure/eviction/stage_registry.rs::execute_kv_plan

4. Layer — transformer 디코더 블록 전용

"Layer"는 이 코드베이스에서 transformer 디코더 블록만 가리킨다(Llama 3.2 1B 기준 16개). 코드의 TransformerLayer는 한 블록의 weight 묶음이다: attention 4종(wq/wk/wv/wo), MLP 3종(w_gate/w_up/w_down), norm 2종(attention_norm/ffn_norm), 그리고 모델별 optional 필드(Qwen qkv_bias, Gemma3 q_norm/k_norm/pre_ffn_norm/post_ffn_norm, tensor partition용 partition_ctx).

KV·weight가 어떤 표현으로 저장되는가는 Layer가 아니라 Format이다(§2). 이 구분이 깨지면 "layer 16개" (디코더 블록)와 "format 멤버" (f16/q4_0/KIVI)가 한 문장에서 충돌한다. KVCacheFormat::idx()가 "이 format이 어느 decoder layer의 KV인가"를 돌려주는 데서 보듯, 둘은 직교 개념이다 — layer 하나당 format 인스턴스 하나가 붙는다.

근거: engine/src/layers/transformer_layer.rs::TransformerLayer(L29), engine/src/format/kv_cache_format.rs::KVCacheFormat::idx doc

안티패턴: "KV cache를 KIVI layer로 바꾸자" — ✗. "KIVI format으로 바꾸자"가 맞다. / "layer 단위 quantization"이 디코더 블록별 weight precision을 뜻하는지(QCF_weight 영역, §6), 저장 표현 변경(format)을 뜻하는지 항상 명시하라.

5. Handle 3종 — Stage가 데이터를 잡는 방식

Stage가 자신이 조절할 데이터를 보유하는 참조의 정적 타입은 정확히 3가지 메뉴다. 4번째는 없다.

안티패턴:
  • "Sliding이 fmt를 StandardFormat으로 downcast해서 빠르게 처리" — ✗. base-trait-handle을 든 Stage는 format을 몰라야 하고, downcast 자체가 trait 설계로 차단되어 있다. 구체 접근이 필요하면 처음부터 concrete-handle을 들어라.
  • "미래에 다른 backend도 쓸 수 있으니 capability trait을 미리 만들어 두자" — ✗. 소비자 1개면 만들지 않는다. 실제로 ScoreCollector/TierMovable은 "첫 소비자 등장 전까지 만들지 않는다"고 capability.rs 모듈 doc에 명시되어 있다.
  • CapabilityRegistry 함정: register/get의 타입 키가 다르면 panic이 아니라 silent miss(None)다. "capability가 안 잡히는데 에러도 없다"면 양쪽 trait bound가 같은지부터 확인하라.

6. QCF 두 패밀리 — QCF_kv vs QCF_weight

QCF(Quality Cost Function)는 lossy action의 품질 열화를 추정하는 메트릭인데, 측정 공간이 다른 두 패밀리가 있고 이름이 비슷해 가장 자주 오용된다.

QCF_kvQCF_weight
측정 공간attention output: QCF = ‖O_before − O_after‖ / ‖O_before‖, O = Σ_t α_t · V_tforward path의 weight/layer 단위: QCF_swap(S) = Σ_{i∈S} imp_i / Σ_{j valid} imp_j (0..1 clamp)
포함 액션sliding/H2O/streaming eviction, D2O merge (QcfActionType 4종), KIVI quant flush(NMSE 등)weight swap(F16→Q4), layer skip(SWIFT) — ImportanceTable 공유
주 코드 위치engine/src/qcf/qcf_kv.rs::compute_qcf_kv, qcf/quant_qcf.rs::compute_flush_nmseengine/src/pressure/weights/decider.rs::compute_qcf_weight_swap, qcf/skip_qcf.rs::SkipQcfTracker
운반 타입qcf_types::QcfMetric { action, raw_value, normalized_value, per_head, tokens_affected } — HandlerContext.qcf_sink로 수집

QCF_kv 쪽 세부: QcfActionType은 EvictSliding/EvictH2o/EvictStreaming/MergeD2o를 simulate하며, D2O merge 시뮬레이션은 actuator(pressure/d2o_handler.rs)와 lockstep으로 유지된다 — Eq.11 그룹 정규화(w_c = e/D, e = MERGE_E = 0.1, u는 exp 전 [-10,10] clamp), nearest 매핑은 per-head K cosine. 단 EMA threshold filter는 simulator에 적용하지 않으므로 estimator 값은 실제 QCF의 상한이다.

QCF_weight 쪽 세부: compute_qcf_swap_internal은 NaN-ε layer를 분자·분모 양쪽에서 제외하고(애초에 유효 swap 후보가 아니었던 layer), importance 테이블 부재 시 1.0을 기본값으로 쓴다. layer skip이 weight 패밀리에 속하는 이유는 "그 layer의 weight를 쓰지 않는" 동작이라서다.

직접 비교 불가: 두 패밀리는 측정 공간이 다르므로 raw 값끼리 비교할 수 없다. cross-family 비교는 DegradationEstimator(offline-calibrated piecewise-linear 곡선)로 ΔPPL로 환산한 뒤에만 가능하다.

근거: engine/src/qcf/qcf_kv.rs 모듈 doc(핵심 수식)·::QcfActionType, engine/src/pressure/weights/decider.rs::compute_qcf_weight_swap(L267)·::compute_qcf_swap_internal(L278), engine/src/qcf.rs 모듈 doc(DegradationEstimator ΔPPL 환산), engine/src/qcf_types.rs::QcfMetric(L110)

안티패턴: "H2O eviction의 QCF가 0.3이고 weight swap의 QCF가 0.2니까 swap이 덜 해롭다" — ✗. 앞은 attention output 공간, 뒤는 weight importance 비율 공간이다. 비교하려면 둘 다 DegradationEstimator로 ΔPPL 환산 후 비교하라(estimator에 액션별 곡선이 등록되어 있어야 함).

7. KVLayout — HeadMajor / SeqMajor

KV 캐시 버퍼의 메모리 배치 enum. element offset 공식까지 외워 둘 가치가 있다 — eviction·QCF·GPU scatter가 전부 이 공식 위에 서 있다.

// engine/src/kv_cache_ops.rs
pub enum KVLayout { SeqMajor, HeadMajor }

// engine/src/pressure/kv_cache.rs::offset(pos, head)
SeqMajor:  pos * kv_heads * head_dim + head * head_dim   // [batch, seq_pos, kv_heads, head_dim]
HeadMajor: head * capacity * head_dim + pos * head_dim    // [batch, kv_heads, seq_pos, head_dim]

// pos_stride() — 같은 head에서 인접 position 간 element 간격
SeqMajor:  kv_heads * head_dim
HeadMajor: head_dim

HeadMajor의 offset에 capacity(물리 용량)가 들어가는 점에 주의 — head별 구획 크기가 current_pos가 아니라 capacity로 고정되므로, 캐시 grow 시 head 구획 전체의 재배치가 필요하다. GPU 커널은 kv_pos_stride/kv_head_stride 인자로 stride 기반 접근을 하므로 flash attention 자체는 layout-agnostic이다.

근거: engine/src/kv_cache_ops.rs::KVLayout(L18), engine/src/pressure/kv_cache.rs::offset(L183)·::pos_stride

실측 교훈 (CLAUDE.md/프로젝트 기록): production은 HeadMajor 고정이다. flash attention과 attention_scores는 stride-agnostic이지만, OpenCL Plan 경로의 kernel_kv_scatter_f32_to_f16은 HeadMajor offset이 하드코딩되어 있고 qcf/d2o/eviction 경로도 HeadMajor를 전제한다. SeqMajor를 선택하면 에러 없이 silent garbage가 나온다(2026-04-14 실측). 예외: KiviCache는 자체 residual 레이아웃 [kv_heads, res_cap, head_dim]을 쓰며 doc상 "SeqMajor layout only (Phase 1)"다 — KIVI 경로 리뷰 시 KVCache의 HeadMajor 가정과 섞지 말 것.

8.2 Pressure / PressureSource (α-W additive — 미배선)

Pressure(u8)는 memory·thermal·energy 등 여러 system 압력을 0~100 단일 scalar로 융합한 graded 입력이다(new()가 100으로 clamp). 의도적 lossy 단일화 — 소비 Stage는 압력의 출처를 구분하지 않는다. scalar로 환원 불가한 mode 동작(switch/suspend — "73만큼 switch"는 없다)은 이 값이 아니라 별도 이산 명령 채널로 흐른다. PressureSource trait(fn pressure(&self) -> Pressure) 뒤에 Manager 통합값/로컬 자율 소스가 숨는다. band()가 4-level로 강등하는 cutoff(0–49 Normal / 50–74 Warning / 75–89 Critical / 90+ Emergency)는 코드 주석이 명시한 day-1 placeholder이며, band()는 현재 어떤 live 경로에도 배선되어 있지 않다.

근거: engine/src/pipeline.rs::Pressure(L24)·::Pressure::band(L44, placeholder 명시)·::PressureSource(L59)

2-tier 추상화의 상위가 CachePressureHandler다 — eviction을 넘어 swap·quantize·merge를 일반화한 trait로, handle(&self, ctx: &mut HandlerContext) -> Result<ActionResult> 하나가 계약이다. CachePressurePipeline::execute는 min_level 오름차순으로 정렬된 stage들 중 ctx.pressure_level >= stage.min_level인 것을 순차 실행하고, 각 핸들러는 직전 핸들러가 남긴 캐시 상태를 본다. 같은 핸들러를 레벨별 다른 계수로 여러 슬롯에 등록할 수도 있다.

ActionResult는 핸들러 실행 결과 enum이며 현재 코드에 정확히 4개 variant가 있다:

// engine/src/pressure.rs (L81)
pub enum ActionResult {
    NoOp,
    Evicted { tokens_removed: usize, new_pos: usize },
    Swapped { tokens_swapped: usize },
    WeightSwapped { layers_changed: usize, freed_bytes: usize, duration_ms: f64 },

과거 문서·메모리에 기록된 Quantized/Recalled variant(생산자 없는 dead variant)는 기준 커밋에서 이미 삭제되었다. 현재 살아있는 생산자: EvictionHandler·D2OHandler → Evicted, SwapHandler → Swapped, WeightSwapHandler::execute_swap → WeightSwapped(파이프라인 밖 직접 dispatch). QuantizeHandler는 등록은 되지만 항상 NoOp를 반환하는 stub이다 — 실제 KIVI quantization은 이 핸들러가 아니라 KiviCache 자체 경로로 동작한다. 리뷰에서 "quantize 핸들러가 처리하겠지"라고 가정하지 말 것.

8.4 EvictionPolicy (legacy, 흡수 중)

2-tier의 하위 trait — 영구 토큰 삭제 전용이다. should_evict / evict(cache, target_len) / evict_with_scores / evict_with_head_scores(GQA per-head)로 &mut KVCache를 in-place 변형한다. 구현: NoEvictionPolicy, SlidingWindowPolicy, H2OPolicy, H2OPlusPolicy, StreamingLLMPolicy. EvictionHandler가 이를 래핑해 상위 파이프라인에 연결하며 target_len = current_pos × target_ratio(ratio는 0.1~0.99로 clamp)를 계산해 넘긴다. in-place 방식은 merge(D2O)·per-head keep을 못 담는 한계가 있어, plan-returning으로의 다리인 plan_keep()(default None = 미지원)이 trait에 추가되어 KVCacheStage로 흡수 중이다 — 마이그레이션 동안 두 경로가 branch-by-abstraction으로 공존한다.

근거: engine/src/pressure/eviction.rs::EvictionPolicy(L13)·::plan_keep(L68, default None), engine/src/pressure/eviction_handler.rs::EvictionHandler(L23)

안티패턴:
  • "PressureLevel이 50을 넘으면…" — ✗. PressureLevel은 4-variant enum이고, 0~100 scalar는 별개 타입 Pressure다. 그리고 Pressure는 아직 live 경로에 미배선이다 — production 게이트는 PressureLevel 비교(ctx.pressure_level >= min_level)다.
  • "ActionResult::Quantized에 match를 추가" — ✗. 그 variant는 존재하지 않는다(컴파일 에러). 옛 문서를 보고 코드를 추정하지 말 것.
  • "새 캐시 관리 기법이니 EvictionPolicy를 구현하자" — △. 영구 삭제 외의 동작(merge·per-head·quantize)은 legacy trait에 안 담긴다. 신규 기법은 KVCacheStage(§3)가 정식 표면이다.

9. 추론 단계·메모리 용어

9.1 Prefill / Decode

Prefill은 프롬프트의 배치 토큰(seq_len > 1)을 한 번에 forward하여 KV 캐시를 구축하는 1단계, Decode는 토큰 단위(seq_len = 1)로 반복 생성하는 2단계다. layer 레벨 진입점이 갈린다 — prefill은 forward_prefill_fmt, decode는 forward_gen(legacy generic) 또는 forward_gen_fmt(base-trait-handle fork). 세션 레벨에서는 session/decode_loop.rs::DecodeLoop가 루프를 소유하고, eviction 등 stage 동작은 forward 호출 사이에 caller 책임으로 끼어든다(모델은 eviction을 모른다). 전체 흐름은 04. 추론 파이프라인.

근거: engine/src/layers/transformer_layer/forward_prefill_fmt.rs::forward_prefill_fmt, engine/src/layers/transformer_layer/forward_gen_fmt.rs::forward_gen_fmt("decode, seq_len=1"), engine/src/session/decode_loop.rs::DecodeLoop(L30)

9.2 KIVI — format이자 stage 합성

KV를 quantized 블록 + F32 residual로 양자화하는 format 멤버. KiviCache의 자료구조: 양자화 저장(qk/qv, Q2/Q4/Q8 — bits 필드 2/4/8, transition_bits()로 동적 전환), residual 버퍼 res_k/res_v(F32, [kv_heads][res_cap][head_dim], res_cap은 group size 32의 배수), 카운터 q2_tokens(양자화 토큰 수, 항상 res_cap 배수)·res_pos(residual 유효 토큰 수). 양자화 방향이 K/V 비대칭이다 — Key는 per-channel(head별 head_dim 채널 각각을 토큰 방향 32개 그룹으로), Value는 per-token(토큰별 head_dim 값을 32개 그룹으로). GPU 모드에서는 attention_gen_kivi 커널이 dequant 없이 KiviRawBuffers(양자화 블록 + residual 참조 + bits)를 직접 읽는다. "KV를 Q2로 다시 까는" 변환이 상주 데이터를 수정하므로 stage와의 합성이다(§1 note).

근거: engine/src/pressure/kivi_cache.rs::KiviCache(L205, struct doc에 per-channel/per-token 명세), engine/src/kv_cache_ops.rs::KiviRawBuffers(L27)

9.3 Zero-copy — UnifiedBuffer vs rpcmem

CPU↔GPU memcpy 제거 기법의 총칭인데 두 메커니즘을 구분해야 한다. UnifiedBuffer는 CL_MEM_ALLOC_HOST_PTR로 GPU 버퍼를 CPU 포인터로 매핑하는 경로(engine/src/memory/opencl/unified.rs), rpcmem은 DMA-BUF heap 할당 + CL_MEM_USE_HOST_PTR alias 경로(engine/src/memory/rpcmem/)다.

실측 교훈 (CLAUDE.md): Adreno(Galaxy S25)에서 CL_MEM_ALLOC_HOST_PTR(UnifiedBuffer / --zero-copy)는 driver internal pool alloc이라 zero-copy 효과가 없다. 진짜 zero-copy 원천은 rpcmem DMA-BUF interop(--opencl-rpcmem)이다. 또한 ARM UMA 캐시 비일관성 때문에 as_ptr 직접 접근은 stale cache를 읽는다 — read_buffer/write_buffer 경유가 필수다. 이는 코드만으로 확인 불가한 디바이스 실측 지식이다.

상세는 06. 메모리 & 데이터 포맷.

9.4 Hardware — read-only resolver

런타임 Hardware 객체는 backend 레지스트리({cpu, gpu?, npu?}) ⊥ memory 레지스트리({host, device?})를 분리 보유하고, resolve(target: DeviceTarget) -> Option<(backend, memory)>로 UMA/discrete 분기를 한 곳에 가둔다 — device == None이면 UMA로 보고 host memory를 재사용한다. Cpu는 항상 Some(universal floor), Gpu/Npu는 미컴파일·미보유 시 None이고 부재 시 정책은 호출자 몫이다. 활성 backend를 mutable로 들지 않는다 — "현재 device"는 decode-loop local 상태이고, 연산 실행 위치는 텐서가 든 backend 태그(Tensor = Shape + buffer + Arc<dyn Backend>)에서 얻는다. OpenCL과 CUDA는 feature 배타라 DeviceTarget::Gpu 하나로 충분하며, --opencl-rpcmem은 device가 아니라 memory interop이라 variant가 아니다.

근거: engine/src/hardware.rs::Hardware(L52)·::DeviceTarget(L25)·::resolve(L79)
