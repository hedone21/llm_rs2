08. Resilience & IPC

기준 커밋: 46ee4c2d · 작성일: 2026-06-10 · llm.rs Engine Guide

1. 한눈에 보기 — 두 프로세스, 두 채널

llm.rs의 resilience 시스템은 별도 프로세스인 llm_manager(시스템 리소스 매니저)가 디바이스 상태(메모리·온도·전력·연산 부하)를 감시하고, 추론 엔진에게 IPC로 명령을 내려 추론 품질·속도·메모리 사용을 런타임에 조절하는 구조다. 양쪽은 같은 wire 타입 크레이트 llm_shared(shared/src/lib.rs)를 공유한다.

엔진으로 들어오는 제어 입력은 성격에 따라 두 채널로 나뉜다:

  • 이산(discrete) 명령 채널 — EngineCommand enum. switch/suspend/throttle/evict 같은 mode 동작. Manager가 EngineDirective로 묶어 보내고, 엔진의 CommandExecutor가 토큰마다 poll하여 ExecutionPlan으로 번역한다. 이 챕터의 주제다.
  • graded Pressure 채널 — Pressure(u8) 0~100 scalar. memory 압력처럼 "얼마나"가 중요한 입력은 이산 명령이 아니라 scalar로 흐르도록 설계되었다(§8). 단, 이 채널은 Phase α-W에서 타입만 신설된 additive 상태로 아직 live decode loop에 배선되지 않았다.
용어 주의 — ResilienceAction은 폐기된 이름이다. 과거 설계에서 strategy의 출력이던 ResilienceAction은 현재 코드에 존재하지 않는다. ResilienceStrategy::react()의 출력 타입은 Vec<EngineCommand>로 단일화되었다 (strategy.rs 주석: "구 ResilienceAction 폐기, EngineCommand 단일 어휘"). 마찬가지로 구 MemoryStrategy도 삭제되어 memory 압력은 strategy 경로를 타지 않는다.

2.1 Level — 공통 4단계 심각도

pub enum Level { Normal, Warning, Critical, Emergency }  // PartialOrd: Normal < ... < Emergency

모든 SystemSignal이 공유하는 심각도다. Ord derive 덕에 a.max(b)로 "가장 나쁜 신호" 집계가 가능하고, 이것이 OperatingMode::from_levels()의 기반이다(§6.3). 프로토콜 레벨(Manager↔Engine 통신)에는 별도의 3단계 ResourceLevel { Normal, Warning, Critical }이 존재한다 — 주석에 따르면 내부 모니터는 4단계를 쓰고, PolicyEngine이 Emergency를 Suspend 명령으로 변환하므로 wire에는 3단계만 필요하다.

3. Transport 계층 — D-Bus / UnixSocket / TCP

3.1 Transport trait과 wire format

pub trait Transport: Send + 'static {
    fn connect(&mut self) -> Result<(), TransportError>;
    fn recv(&mut self) -> Result<ManagerMessage, TransportError>;   // blocking
    fn send(&mut self, msg: &EngineMessage) -> Result<(), TransportError>;
    fn name(&self) -> &str;
}

소켓 계열(Unix/TCP)의 wire format은 [4바이트 BE u32 길이][UTF-8 JSON payload]이고, payload 상한은 MAX_PAYLOAD_SIZE = 64 * 1024(64 KB)다 — 초과 시 ParseError. EOF는 Disconnected로, WouldBlock/TimedOut은 Timeout으로 매핑된다.

3.2 MessageLoop — 단일 스레드 양방향 브리지

MessageLoop::spawn(transport)은 transport를 소유하는 스레드 1개를 띄우고 (cmd_rx: Receiver<ManagerMessage>, resp_tx: Sender<EngineMessage>, JoinHandle)을 돌려준다. 루프 본체는: ① resp_rx.try_recv()로 나갈 메시지를 전부 송신 → ② transport.recv() blocking 수신 → ③ Timeout이면 ①로 되돌아감(continue), ParseError는 skip, Disconnected/기타는 루프 종료.

함정: UnixSocket/D-Bus 경로의 상향 메시지 송신 지연. 나가는 메시지(heartbeat 등)는 blocking recv() 사이에만 flush된다. TcpTransport는 set_read_timeout(50ms)를 걸어 50ms마다 Timeout으로 깨어나 drain하지만, UnixSocketTransport::connect()는 read timeout을 설정하지 않는다 — Manager가 directive를 보내지 않는 동안 heartbeat가 resp_rx 큐에 적체될 수 있다. D-Bus도 signal iterator가 blocking이라 같은 구조다. 주기 heartbeat가 중요한 시나리오에서는 tcp: transport가 코드상 가장 안전하다.

근거: engine/src/resilience/transport.rs::MessageLoop::run_loop (L106~150), TcpTransport::connect (L363~379 — 50ms timeout), UnixSocketTransport::connect (L404~418 — timeout 설정 없음)

3.3 DbusTransport — 신·구 두 프로토콜

DbusTransport(resilience feature + zbus 의존)는 System Bus의 org.llm.Manager1(destination/path/interface 모두 동일 문자열)에 proxy를 걸고 receive_all_signals() iterator로 수신한다. 수신 분기는 두 갈래다:

  1. 신형: member가 "Directive"인 signal — body의 JSON 문자열을 ManagerMessage로 역직렬화해 그대로 반환.
  2. legacy: MemoryPressure/ComputeGuidance/ThermalAlert/EnergyConstraint signal — D-Bus 타입 튜플을 SystemSignal로 파싱한 뒤, transport 내부의 하드코딩 매핑 signal_to_manager_message()로 EngineCommand 배치를 합성한다. seq_id는 transport가 로컬 카운터로 발급.

legacy 변환표 (transport 내장 정책):

신호NormalWarningCriticalEmergency
MemoryPressureRestoreDefaultsKvEvictSliding{0.85}KvEvictH2o{0.50}Suspend
ComputeGuidanceRestoreDefaultsThrottle{30} + SwitchHw{dev}Throttle{70} + SwitchHw{dev}Suspend
ThermalAlertRestoreDefaultsThrottle{30} + PrepareComputeUnit{cpu}Throttle{70} + SwitchHw{cpu}Suspend

엔진→Manager 방향(send)은 EngineMessage JSON을 D-Bus signal "EngineMessage"로 emit하는 best-effort다.

D-Bus는 사실상 하향 단방향이다. Manager 쪽 transport 핸들에서 D-Bus는 TransportHandle::EmitterOnly로 분류되고 try_recv_engine_message()가 항상 None을 반환한다 — 엔진이 emit한 heartbeat/QCF/Response는 D-Bus 경로에서는 Manager에 수신되지 않는다. heartbeat 기반 정책(QCF 협상, available_actions 필터)을 쓰려면 unix:/tcp: transport가 필요하다.

4. 엔진 수신측 — CommandExecutor

4.1 역할과 poll() 사이클

CommandExecutor는 "strategy 없음 — 명령을 plan으로 번역만 한다"는 명세의 얇은 변환기다. cmd_rx/resp_tx 채널 양끝을 쥐고, 추론 루프가 토큰마다 1회 poll(&KVSnapshot) -> ExecutionPlan을 호출한다. poll의 순서:

  1. heartbeat 체크 — last_heartbeat.elapsed() >= heartbeat_interval(production 1000ms)이면 EngineStatus 송출.
  2. directive drain — cmd_rx.try_recv()를 비울 때까지 수집. 비어 있으면 sticky 상태(throttle/target_tbt/evict_plan/kv_quant_bits/partition_ratio_sticky)만 실어 즉시 반환.
  3. 명령 적용 — directive별로 apply_command(cmd, &mut plan) 실행, 명령마다 CommandResult를 모아 CommandResponse{seq_id, results}를 directive 단위로 송신.
  4. sticky carry — 이번 배치가 덮어쓰지 않은 evict/kv_quant_bits를 이전 sticky 값으로 채움.
  5. Suspend override — plan.suspended면 evict/switch/prepare/throttle/resumed를 모두 무효화하고 EngineState::Suspended로 전이.

근거: engine/src/resilience/executor.rs::CommandExecutor::poll (L289~358)

4.2 ExecutionPlan과 sticky / transient 구분

ExecutionPlan은 추론 루프가 소비하는 단일 구조체다. 필드의 수명 의미가 셋으로 갈리는데, 리뷰 시 가장 헷갈리는 지점이므로 분류해 둔다:

수명필드동작
sticky (RestoreDefaults까지 유지)throttle_delay_ms, target_tbt_ms(+target_tbt_set), evict, kv_quant_bits, partition_ratioexecutor 내부 필드에 저장되어 매 poll 재게시. partition_ratio_sticky는 prefill 중 도착한 directive가 decode poll에서도 보이게 하는 ISSUE-5 fix
transient (도착한 poll에서만 Some)switch_device, prepare_device, offload_ratio, layer_skip, request_qcf, restore_defaults, recall_offload, prefill_*1회성. 그 poll의 소비자가 처리하지 않으면 소실
sticky-until-consumeswap_weights + executor의 pending_swap_weights아래 warn 참조
LISWAP-6 패턴 — prefill poll이 plan을 버린다. prefill 루프도 heartbeat 유지를 위해 executor.poll()을 호출하지만 반환된 plan을 소비하지 않고 drop한다. SwapWeights가 transient 필드에만 기록되면 prefill 중 도착한 directive가 소실되는 버그가 있었다. 수정: apply_command가 pending_swap_weights에도 저장하고, decode 루프가 매 tick take_pending_swap_weights()로 consume할 때까지 executor 내부에 살려 둔다. 새 명령을 추가할 때 "prefill 중 도착해도 살아남아야 하는가"를 반드시 검토할 것.

근거: engine/src/resilience/executor.rs::ExecutionPlan (L19~65), pending_swap_weights (L162~169 주석), take_pending_swap_weights (L258)

4.3 RestoreDefaults의 미묘한 의미론

RestoreDefaults는 throttle/target_tbt/evict_plan/kv_quant_bits/partition_ratio_sticky/active_actions를 모두 초기화하고 recall_offload = true를 세운다. 단 plan.target_tbt_set은 의도적으로 false로 남긴다 — 엔진이 "Manager가 명시적으로 0을 보냄(pacing 끔)"이 아니라 "restore_defaults 분기(CLI 기본값 복귀)"를 타게 하기 위해서다. SetTargetTbt{0}과 RestoreDefaults는 둘 다 pacing을 멈추지만 엔진의 fallback 동작이 다르다.

근거: engine/src/resilience/executor.rs::apply_command RestoreDefaults arm (L484~502, 주석 L489~491), ExecutionPlan.target_tbt_set doc (L31~35)

4.4 heartbeat와 available_actions 계산

send_heartbeat()는 KVSnapshot(total_bytes/total_tokens/capacity/protected_prefix/kv_dtype/eviction_policy/skip_ratio)에서 파생값을 계산한다: kv_cache_utilization = total_tokens / capacity, memory_lossy_min = max(protected_prefix / total_tokens, 0.01), memory_lossless_min = 1.0(고정 — "현재 무손실 축소 불가"). 처리량은 토큰마다 on_token_generated()가 갱신하는 EMA(α=0.1)다. available_actions는 구성 기반 조건부:

  • 항상: throttle, switch_hw, weight.skip
  • eviction_policy != "none"일 때: kv.evict_h2o / kv.evict_sliding / kv.evict_streaming / kv.merge_d2o
  • kv_dtype가 'q'로 시작할 때: kv.quant_dynamic
  • secondary GGUF/AUF가 로드되어 set_has_secondary(true)일 때: swap_weights

근거: engine/src/resilience/executor.rs::send_heartbeat (L573~643), ::compute_available_actions (L646~672), ::on_token_generated (L268~285)

5. 세션 배선 — ResilienceAdapter와 DecodeLoop

5.1 초기화: build_command_executor

argus_cli는 args.enable_resilience = !args.no_resilience로 default-on이다 (legacy generate 계열은 --enable-resilience opt-in 유지). build_command_executor()가 --resilience-transport 문자열로 분기한다: dbus(feature resilience 필요) / unix:<path> / tcp:<addr>. spawn 실패·unknown transport·feature off는 모두 graceful fallback — "[Resilience] Manager unreachable" 경고 후 Ok(None)으로 Manager 없이 추론을 계속한다 (default-on 정책에서 일반 사용자의 추론이 깨지지 않게 하는 회귀 차단). 이어 heartbeat 1000ms로 executor를 만들고 Capability를 송출한다 (bytes_per_kv_token = num_kv_heads × head_dim × 2(K+V) × 2(F16)).

근거: engine/src/bin/argus_cli.rs (L27), engine/src/session/resilience_init.rs::build_command_executor (L25~132)

5.2 ResilienceAdapter — 1 인스턴스, 3 trait slot

session 파이프라인은 resilience 기능을 3개 trait으로 쪼개 요구한다: CommandSource(poll → ExecutionPlan), EngineReport(capability/qcf/swap report 송신), TokenTickSink(on_token_generated). ResilienceAdapter가 CommandExecutor 하나를 소유한 채 3개를 모두 구현하고, DecodeLoopBuilder::with_resilience()가 이를 Arc<Mutex<_>>로 감싸 newtype wrapper 3개(CmdSrcWrapper/ReportWrapper/TickWrapper)로 각 slot에 주입한다. per-token lock 횟수는 poll 1회 + tick 1회라 contention은 무시 가능하다. resilience 미사용 시에는 defaults.rs의 NoOpCommandSource 등이 들어간다.

근거: engine/src/session/resilience_adapter.rs::ResilienceAdapter (L21~59, wrapper L70~115), engine/src/session/decode_loop.rs::DecodeLoopBuilder::with_resilience (L634~645), engine/src/session/traits.rs::CommandSource (L171)

5.3 DecodeLoop가 plan을 소비하는 순서

DecodeLoop::run()의 step당 resilience 관련 구간 (라벨은 코드 주석의 (a)~(h)와 일치):

  1. (a) poll — cmd_source.poll(ctx, kv_snap). plan.suspended면 StopReason::CommandRequested로 루프 탈출. throttle_delay_ms > 0이면 그만큼 thread::sleep.
  2. (a.5) resilience-driven eviction — plan.evict가 Some이고 evict_applied == false일 때만 forward.try_evict(cache_manager, None, true, target_ratio) 1회 호출(score-free force_evict — H2O는 score 부재 시 recency로 degrade). 성공 시 pos 갱신 + on_kv_prune hook + observer 통지. plan.evict가 None으로 돌아오면(RestoreDefaults) evict_applied를 reset.
  3. (a.6) KvOffload / recall — offload_ratio는 non-sticky라 자연히 1회 발동: forward.try_offload(cm, ratio)로 prefix를 disk swap(pos 감소). recall은 recall_offload && restore_defaults 둘 다 참일 때만 try_recall(cm)(pos 증가).
  4. (f2) tick — sampling 직후 tick_sink.on_token_generated() → 처리량 EMA 갱신.
  5. (h) target TBT pacing — step 시작점 t_iter부터의 wall-clock(throttle sleep 포함)이 target_tbt_ms에 못 미치면 잔여분 sleep. 0이면 no-op.
sticky evict의 재적용 방지(evict_applied)는 의도된 비대칭이다. evict plan은 executor에서 sticky로 매 poll 재게시되지만, DecodeLoop는 active 구간당 1회만 실행한다 — target_ratio를 매 step 반복 적용하면 캐시가 기하급수로 줄기 때문이다. 반대로 throttle/target_tbt는 매 step 효과가 의미라 sticky 그대로 소비한다. eviction 자체의 동작은 07장 CacheManager 참조. 또한 eviction은 cache_manager가 builder로 주입된 경우(현재 argus-bench 경로)에만 발동한다 — happy path/chat은 미주입이라 plan.evict가 와도 무시된다.

근거: engine/src/session/decode_loop.rs::DecodeLoop::run (L124~346, evict guard L168~194, offload L201~223, pacing L325~333), engine/src/session/traits.rs::Forward::try_evict (L108~117)

6. 엔진 내장 자율 정책 — ResilienceStrategy와 ResilienceManager

6.1 trait과 strategy 3종

ResilienceStrategy는 신호 1건을 받아 이산 명령 묶음을 내는 계약이다: 

pub trait ResilienceStrategy: Send + Sync {
    fn react(&mut self, signal: &SystemSignal) -> Vec<EngineCommand>;  // 빈 Vec = 무반응
    fn name(&self) -> &str;
}

구현은 compute/thermal/energy 3종뿐이다. memory strategy는 없다 — memory 압력은 graded Pressure scalar 채널 담당으로 설계가 이동했고(§8), react는 scalar로 환원 불가한 mode 출력(switch/suspend/throttle/restore)만 낸다. Level별 매핑 전체:

strategyNormalWarningCriticalEmergency
ComputeStrategy
(현재 backend 상태 보유)		RestoreDefaults
(current → Any)		빈 Vec — 준비만
(current_backend 갱신)		권고 backend ≠ current면 SwitchHw{dev}
(Any면 switch 생략); 이미 같으면 Throttle{50}		SwitchHw{cpu} + Throttle{100}
ThermalStrategyRestoreDefaultsSwitchHw{cpu}SwitchHw{cpu} + Throttle{(1.0 − throttle_ratio) × 100 ms}Suspend
EnergyStrategyRestoreDefaultsSwitchHw{cpu}SwitchHw{cpu} + Throttle{30}Suspend

구 LimitTokens{64}·RejectNew 액션은 EngineCommand 등가가 없어 drop되었다 (주석: "Emergency stop-intent는 Suspend가 흡수"). backend 역할 문자열은 switch_device()가 Cpu→"cpu", Gpu→"gpu", Any→None(switch 생략)으로 변환한다 — opencl/cuda 같은 구체 device 해석은 Hardware resolver 책임이다.

근거: engine/src/resilience/strategy.rs::ResilienceStrategy (L29), ::switch_device (L15), strategy/compute.rs::ComputeStrategy::react (L25~67), strategy/thermal.rs (L21~48), strategy/energy.rs (L21~41)

6.2 resolve_conflicts — 4규칙 병합

여러 strategy의 출력을 합칠 때 resolve_conflicts(Vec<EngineCommand>)가 충돌을 정리한다:

  1. R1 — Suspend가 하나라도 있으면 결과는 [Suspend] 단독.
  2. R2 — RestoreDefaults는 다른 제약이 전무할 때만 살아남는다.
  3. R3 — SwitchHw: "cpu"가 한 번이라도 나오면 cpu 고정(안전 우선), 아니면 마지막 값.
  4. R4 — Throttle: 가장 큰 delay_ms가 이긴다.

그 외 명령(SetTargetTbt/Kv*/Resume 등)은 원순서 보존 pass-through다. 결과 배열 순서는 SwitchHw → Throttle → passthrough.

근거: engine/src/resilience/strategy.rs::resolve_conflicts (L44~96)

6.3 ResilienceManager와 OperatingMode

ResilienceManager는 mpsc::Receiver<SystemSignal>을 들고 토큰당 1회 poll()로 신호를 drain → 신호 타입별 strategy 위임(memory는 명령 0) → resolve_conflicts를 수행하는 자율 오케스트레이터다. 부수적으로 신호 4종의 최신 Level을 캐시해 OperatingMode를 갱신한다: 

// 최악의 신호가 mode를 결정 (state.rs L19~27)
worst = memory.max(compute).max(thermal).max(energy)
Normal → OperatingMode::Normal      Warning → Degraded
Critical → Minimal                  Emergency → Suspended
이 경로는 현재 production에 배선되어 있지 않다. ResilienceManager::new()의 호출자는 자기 파일의 테스트들뿐이다. 주석은 "manager-less 자율 정책의 원형 — LocalPolicy CommandSource로 발전 예정(ENG-ST-055)"이라 밝힌다. 즉 production 신호 경로는 §3~5의 Manager IPC(CommandExecutor)이고, SystemSignal → ResilienceStrategy.react() 변환이 실제로 작동하는 곳은 (1) 이 미배선 prototype과 (2) DbusTransport의 legacy 변환(§3.3 — 단 strategy를 호출하지 않는 별도의 하드코딩 표) 두 군데로 갈라져 있다. 두 매핑은 값이 다르다 (예: thermal Warning — strategy는 SwitchHw{cpu}, D-Bus legacy는 Throttle{30}+PrepareComputeUnit{cpu}). 리뷰 시 어느 표를 고치는지 명확히 할 것.

근거: engine/src/resilience/manager.rs::ResilienceManager (L53~119, 호출자는 동파일 테스트 L133 등 5곳뿐 — grep ResilienceManager::new로 확인), engine/src/resilience/state.rs::OperatingMode::from_levels (L19~27)

7. Manager 서비스 내부 — 개요 수위

manager/src/main.rs의 구조만 짚는다(상세는 manager 크레이트 영역). 부팅 시 config(TOML)에서 Monitor 4종(memory/thermal/compute/energy — 기본 활성) + external(opt-in)을 만들고 각각 전용 스레드에서 mpsc::Sender<SystemSignal>로 신호를 쏜다. 메인 루프는 50ms recv_timeout tick으로 ① 엔진 상향 메시지 수신(update_engine_state — heartbeat의 available_actions로 엔진이 못 하는 액션을 정책에서 필터), ② QCF timeout 체크(SEQ-098), ③ 신호 수신 → PolicyStrategy::process_signal() → DirectiveDeduplicator(동일 command 배치 + cooldown 내 재방출 억제; 억제 시 cancel_last_observation()으로 학습 관측 취소) → Emitter 송출을 돈다.

PolicyEngine은 PolicyStrategy trait 뒤의 두 구현이다: 내장 HierarchicalPolicy (hierarchical feature, PI 컨트롤러/relief model 포함 — 종료 시 save_model())와 --policy-script로 주입하는 LuaPolicy(lua feature, SIGHUP hot-reload 지원). QCF 협상은 양방향 왕복이다: 정책이 RequestQcf를 보내고, 엔진의 QcfEstimate가 도착하면 complete_qcf_selection()이 보류된 액션 선택을 완료해 후속 directive를 만든다. 1초 내 미도착 시 check_qcf_timeout()이 fallback directive를 낸다.

8. graded Pressure 채널 — memory 압력의 행방

KV 캐시처럼 "얼마나 줄일까"가 연속량인 영역에서, 이산 Level 4단계는 표현력이 부족하다. 그래서 memory 압력은 엔진 내부에서 Pressure(u8)(0~100 clamp) scalar로 흐르도록 재설계되었고, 구 MemoryStrategy(Warning→Evict 0.85 식의 이산 매핑)는 삭제되었다. PressureSource trait 뒤에 Manager 통합값(ManagerPressureSource)이나 로컬 자율 소스 (LocalPressureSource)가 숨는 구도이며, 소비 Stage는 출처를 구분하지 않는다. Pressure::band()는 4단계 강등 placeholder를 제공한다 — 0..=49 Normal / 50..=74 Warning / 75..=89 Critical / 90..=100 Emergency — 하지만 주석이 명시하듯 현재 어떤 live 경로에도 배선되지 않은 additive 타입이다.

그러면 지금 memory 압력은 어떻게 작동하는가? Manager가 이산 KV 명령을 직접 보낸다. 정책(또는 D-Bus legacy 변환표)이 KvEvictSliding{0.85}/KvEvictH2o{0.50} 같은 구체 directive를 만들고, §4~5 경로로 CacheManager에 도달한다. 즉 graded 채널은 미래형이고 현행은 여전히 discrete다 — 이 간극이 이 영역 PR의 단골 혼동 포인트다.

근거: engine/src/pipeline.rs::Pressure (L24, band() L43~50 — "live 경로 미배선" 주석 L38~41), engine/src/resilience/manager.rs::process_signal (L110~115 — MemoryPressure → vec![]), engine/src/resilience/dbus_transport.rs (L36~46)

9. 자기 측정기 — heartbeat에 실리는 self-utilization

heartbeat의 self_cpu_pct/self_gpu_pct는 엔진 프로세스 자신의 자원 사용률이다 (Manager의 시스템 전역 모니터와 별개 — Manager가 "엔진 때문인가"를 판단하는 입력).

  • ProcSelfMeter (CPU, MSG-067): /proc/self/stat의 utime + stime jiffies를 heartbeat 간 delta로 측정. 분모는 CLK_TCK × num_cpus × wall_elapsed_secs (CLK_TCK는 sysconf(_SC_CLK_TCK), 보통 100Hz). comm 필드가 공백·괄호를 포함할 수 있어 rfind(')') 이후부터 파싱하며(괄호 이후 0-based 11/12번째 필드), 어떤 실패든 0.0 fallback + [0.0, 1.0] clamp (INV-091/092 — heartbeat 송출을 절대 차단하지 않음). 첫 샘플은 warm-up으로 0.0.
  • GpuSelfMeter (GPU, MSG-068 Phase 2): sample(wall_elapsed) -> f64 trait. OpenCL queue profiling(CL_QUEUE_PROFILING_ENABLE)으로 커널 busy ns를 누적해 wall로 나눈다. 기본은 NoOpGpuMeter(항상 0.0)이고 --heartbeat-gpu-profile opt-in.
실측 교훈 (CLAUDE.md + cli.rs doc 주석): Adreno에서 queue profiling을 켜면 ~54 ms/token 오버헤드가 붙는다. production TBT 측정 시 --heartbeat-gpu-profile은 반드시 꺼라. production 처리량의 정답 소스는 heartbeat의 actual_throughput(EMA) 또는 Decode: X ms/tok 로그다. 또한 현재 build_command_executor()는 backend Arc를 받지 못해 flag를 켜도 gpu_meter가 None으로 남는다(주석: "P5+에서 overload 도입 예정") — self_gpu_pct는 실질 항상 0.0이다.

별개로 sys_monitor.rs의 SystemMonitor trait(mem_stats())이 있다 — LinuxSystemMonitor는 /proc/meminfo의 MemTotal/MemAvailable/MemFree를 바이트로 파싱하고, NoOpMonitor는 항상 usize::MAX를 반환한다. 후자는 CUDA discrete GPU에서 cuMemAllocManaged가 가상 주소공간 예약으로 MemAvailable을 인위적으로 낮춰 false alarm을 내는 환경용이다 — 이때 KV eviction은 시스템 RAM 압력이 아니라 정책 budget으로만 구동된다.

근거: engine/src/resilience/proc_self_meter.rs::ProcSelfMeter::sample (L47~67), ::parse_utime_stime (L97~106), engine/src/resilience/gpu_self_meter.rs::GpuSelfMeter (L32~40), engine/src/session/cli.rs::heartbeat_gpu_profile (L381~395 doc), engine/src/session/resilience_init.rs (L39~46), engine/src/resilience/sys_monitor.rs::NoOpMonitor (L80~99)
