기준 커밋: 46ee4c2d · 작성일: 2026-06-10 · llm.rs Engine Guide

11.1 워크플로우 한눈에 보기

이 챕터는 코드를 고친 뒤 PR을 올리기까지의 표준 경로를 다룬다: 호스트 빌드/테스트 → feature flag 이해 → Android 크로스 빌드 → 테스트 계층 통과 → 성능 측정. 특히 마지막 절(성능 측정 함정)은 이 코드베이스에서 가장 자주 틀리는 부분이므로 성능 PR 리뷰 전 반드시 읽어야 한다.

flowchart LR
    A["코드 수정"] --> B["호스트 빌드 + unit test\ncargo build / cargo test"]
    B --> C["fmt + clippy\n-D warnings"]
    C --> D["backend 정확성\ntest_backend (CPU vs OpenCL)"]
    D --> E["디바이스 e2e\nrun_device.py -d galaxy_s25 argus_cli"]
    E --> F["성능 측정\nwall-clock TTFT / Decode ms-tok"]
    F --> G["커밋 + PR"]
빌드·린트·유닛 테스트는 /sanity-check 스킬, 디바이스 배포·테스트는 /deploy-test 스킬로 자동화되어 있다 (CLAUDE.md 빠른 참조). 이 챕터는 그 스킬들이 내부에서 실행하는 것과 동일한 명령을 손으로 실행할 수 있도록 메커니즘을 설명한다.

11.2 호스트 빌드와 기본 명령

11.2.1 Workspace 구성과 release 프로필

저장소는 단일 Cargo workspace다. 멤버는 engine(크레이트명 llm_rs2), shared(llm_shared), manager(llm_manager), 그리고 plugin 표면인 crates/technique-api, crates/qnn_oppkg, glob 멤버 crates/techniques/*다. techniques 디렉토리는 glob이라 폴더만 추가하면 workspace 멤버로 자동 편입된다 — stage 축 plugin crate를 추가할 때 루트 Cargo.toml 수정이 필요 없다(10장 참조).

[workspace]
members = ["engine", "shared", "manager",
           "crates/qnn_oppkg", "crates/technique-api", "crates/techniques/*"]
resolver = "3"

[profile.release]
lto = "fat"
codegen-units = 1
panic = "abort"
opt-level = 3

release 프로필이 lto = "fat" + codegen-units = 1이므로 release 빌드는 느리다(전체 crate 그래프를 단일 codegen 단위로 LTO). 성능 수치는 release 빌드에서만 의미가 있고, 개발 중 빠른 반복은 cargo check/cargo test(debug)로 한다. panic = "abort"이므로 release 바이너리에서 panic은 unwinding 없이 즉시 프로세스를 종료한다 — catch_unwind 기반 복구는 release에서 동작하지 않는다.

근거: Cargo.toml (workspace root, L1–L18)

11.2.2 기본 명령 세트

CLAUDE.md 빠른 참조의 표준 명령:

cargo build --release -p llm_rs2
cargo test --workspace
cargo fmt --all && cargo clippy --workspace -- -D warnings

clippy는 -D warnings로 경고를 에러 승격한다 — 경고 하나라도 남으면 sanity-check 실패다.

11.2.3 타겟별 rustflags (.cargo/config.toml)

타겟rustflags비고
x86_64-unknown-linux-gnu+avx2,+fma호스트 PC
aarch64-linux-android+neon,+dotprod,+fhm,+fp16 + link-arg=-Wl,--allow-shlib-undefined + -L libs/aarch64Galaxy S25 등 Cortex-A/X
aarch64-unknown-linux-gnu+neon,+fp16 (dotprod/fhm 없음)Jetson Xavier(Carmel ARMv8.2) 호환 범용

같은 aarch64라도 Android 타겟만 dotprod/fhm을 켠다. dotprod 의존 NEON 경로를 추가할 때 Jetson 타겟에서는 해당 intrinsic이 빌드 플래그에 없으므로, SIMD 코드는 target_feature cfg를 타겟별로 확인해야 한다.

근거: .cargo/config.toml (4개 target 섹션)

11.3 Feature flags

engine/Cargo.toml의 feature 정의(발췌):

[features]

feature기본역할
openclONocl crate 활성. GPU 없는 호스트에서도 컴파일은 되며, 런타임에 OpenCL 디바이스가 없으면 GPU 경로가 실행되지 않을 뿐이다.
profileONcargo feature로서의 profile — qcf_timer!/op_start!/op_record! 등 instrument 매크로의 cfg gate. CLI 플래그 --profile과는 별개다(11.6절).
cuda / cuda-embeddedOFFcudarc 기반 CUDA backend. opencl과 동시 활성 시 compile_error!로 빌드 차단.
caoteOFFtechnique plugin crate 링크. 미활성 시 --eviction-policy caote는 unknown으로 거부.
qnnOFFbuild.rs가 bindgen으로 QNN SDK 헤더(third_party/qnn_sdk_2.33/include/QNN, QNN_SDK_ROOT env override 가능) 바인딩을 생성. production 코드는 이 feature 불필요 — microbench bin 전용.
vulkanOFFash 활성. microbench bin 전용 PoC.
htp_fastrpcOFFHTP FastRPC PoC backend. libloading은 feature와 무관한 non-optional 코어 의존(.so plugin dlopen 겸용).
resilienceOFFzbus(D-Bus) 활성. Jetson 디바이스 빌드에서 켠다. 8장 참조.

backend feature 상호배타는 컴파일 타임에 강제된다:

// engine/src/lib.rs L1-L4
#[cfg(all(feature = "opencl", feature = "cuda"))]
compile_error!("Features 'opencl' and 'cuda' are mutually exclusive. Enable only one.");
#[cfg(all(feature = "opencl", feature = "cuda-embedded"))]
compile_error!("Features 'opencl' and 'cuda-embedded' are mutually exclusive. ...");

근거: engine/Cargo.toml::[features] (L91–L105), engine/src/lib.rs (L1–L4), engine/build.rs::qnn_bindgen

함정: 호스트 빌드와 디바이스 빌드의 feature 집합이 다르다. 호스트 기본 빌드는 default = ["opencl", "profile"]이지만, devices.toml의 디바이스 빌드는 default_features = false 후 명시 목록만 켠다 — Galaxy S25는 ["opencl", "vulkan", "qnn", "htp_fastrpc"], Jetson은 ["cuda-embedded", "resilience"]. 즉 디바이스 빌드에는 cargo feature profile이 빠진다: instrument 매크로(qcf_timer! 등)는 디바이스에서 unit ()로 컴파일 제거되고, 호스트에서는 기본 포함이다. "호스트에서 보이던 QCF 타이밍이 디바이스에서 안 나온다"는 버그가 아니라 이 cfg 차이다. 반대로 호스트 기본 측정에는 instrument 오버헤드(작지만 0은 아님)가 항상 섞인다.

근거: devices.toml::[devices.galaxy_s25.build] (L83–L84), [devices.jetson.build] (L115–L116), engine/src/instrument.rs::qcf_timer! (cfg gate, L14–L24)

synth-q4-format technique crate는 optional이 아닌 일반 의존성이다 — --kv-format synth_q4가 production 바이너리에서 동작하도록 force-link되어 있다(Cargo.toml 주석: ADR-0008 D6). plugin crate라고 모두 opt-in feature 뒤에 있는 것은 아니다. 또 example-keep-recent는 dev-dependency로만 링크되어 cross-crate linkme 등록 테스트에 쓰인다.

근거: engine/Cargo.toml dependencies (L14–L18), dev-dependencies (L84)

11.4 Android 크로스 빌드 — run_device.py

크로스 빌드의 정식 경로는 scripts/run_device.py다. cargo를 직접 호출하지 않는 이유: NDK 툴체인 env(CC/AR/LINKER)와 디바이스별 feature 집합을 hosts.toml + devices.toml에서 자동 합성하기 때문이다.

# 최초 1회: 이 머신의 NDK 위치를 hosts.toml로 등록
python scripts/device_registry.py bootstrap-host

# 빌드 + 배포 + 실행 (3단계 일괄)
python scripts/run_device.py -d galaxy_s25 argus_cli --prompt "Hello" -n 128
python scripts/run_device.py --list-devices
python scripts/run_device.py -d galaxy_s25 --skip-build argus_cli -b opencl
# 빌드+배포만 (실행 생략, 추가 bin 동시 빌드)
python scripts/run_device.py -d galaxy_s25 --skip-exec argus_cli --extra-bin llm_manager

11.4.1 NDK env 합성 메커니즘 (builder.py)

_compose_toolchain_env()가 hosts.toml의 toolchain 엔트리(ndk_home + host_tag + api_level)에서 다음 env를 만든다. 타겟 triple aarch64-linux-android, API 21 기준:

CC_aarch64_linux_android  = {ndk_home}/toolchains/llvm/prebuilt/{host_tag}/bin/aarch64-linux-android21-clang
CXX_aarch64_linux_android = ...21-clang++
AR_aarch64_linux_android  = .../llvm-ar
CARGO_TARGET_AARCH64_LINUX_ANDROID_LINKER = (clang과 동일)

clang 바이너리가 없으면 ToolchainNotFoundError로 즉시 실패한다 — env가 잘못 잡힌 채 빌드가 절반쯤 진행되는 일이 없다. env 합성은 os.environ 사본 위에서 이루어지며 셸 환경을 오염시키지 않는다. 호스트 선택 우선순위는 LLM_RS2_HOST env override → env_marker+uname → uname → default_host → 에러 순이다(hosts.toml.example 헤더에 명시). hosts.toml은 gitignore되어 있고 각자 bootstrap-host로 생성한다.

Jetson은 NDK 대신 zig_target = "aarch64-unknown-linux-gnu.2.31"이 설정되어 cargo zigbuild로 분기한다 — zig가 glibc 2.31 타겟 링킹을 처리하므로 toolchain env 합성을 건너뛴다.

근거: scripts/device_registry/builder.py::_compose_toolchain_env (L20–L87), ::build_binary (L90–L195, zigbuild 분기 L113–L117·L163–L175), hosts.toml.example

11.4.2 배포·실행 단계

run_device.py는 build → deploy → execute 3단계를 수행하며 각각 --skip-build/--skip-deploy/--skip-exec로 생략 가능하다. 실행 단계의 자동 처리 두 가지:

  • 바이너리가 _INFERENCE_BINARIES = ("argus_cli", "legacy_generate", "generate")에 속하고 --model-path가 인자에 없으면 devices.toml의 model_dir를 자동 주입한다 (예: S25 → /data/local/tmp/models/qwen2.5-1.5b).
  • paths.lib_dir가 설정된 디바이스는 LD_LIBRARY_PATH를 그 값으로 설정해 실행한다 (S25는 /data/local/tmp — libcdsprpc.so 등 rpcmem 의존 .so 탐색용).

근거: scripts/run_device.py::main (L52–L174, model-path 주입 L146–L152, LD_LIBRARY_PATH L156–L158)

실측 교훈 (CLAUDE.md, 코드로 확인 불가한 운영 지식): Adreno 디바이스 권장 backend는 opencl --opencl-rpcmem이며, --tokenizer-path 명시가 필수다 — 자동 fallback이 sibling 모델의 tokenizer를 잡으면 vocab-size mismatch로 차단된다. Galaxy S25 벤치는 6 스레드만 사용한다(8T는 역효과). 테스트 기본 모델 포맷은 GGUF다.

11.5.1 계층 1: cargo test

cargo test --workspace가 모듈 내 unit test와 engine/tests/의 integration test를 모두 돈다. integration test에는 flash attention 수치 검증(flash_attn_decode.rs, flash_attn_prefill_dk128.rs 등), AUF/GGUF byte-equivalence(test_auf_gguf_byte_equivalence.rs), plugin dlopen 게이트(gate_c_backend_cap_dlopen.rs) 등이 있다. 특이 구조 하나: spec 불변식 테스트는 engine/tests/spec.rs가 #[path = "spec/..."]로 서브 파일들을 끌어모으는 단일 하네스다 — INV 번호(test_inv_003.rs 등)와 ENG-ALG 번호별 테스트, 그리고 trybuild 기반 compile_fail/compile_pass 디렉토리(세션 typestate가 컴파일 타임에 잘못된 사용을 거부하는지 검증)가 들어 있다.

근거: engine/tests/spec.rs (path-include 하네스), engine/tests/spec/compile_fail/, engine/Cargo.toml dev-dependencies trybuild (L81)

11.5.2 계층 2: test_backend — CPU vs GPU 수치 비교

engine/src/bin/test_backend.rs는 backend 구현들을 같은 입력으로 돌려 결과를 직접 비교하는 정확성 스위트다(CLAUDE.md 기준 "Tier 2"). 구조:

  • --backends auto,scalar (기본): 비교할 backend 목록을 콤마로 지정. 선택지는 cpu(NEON 등 최적화 CpuBackend), scalar(CpuBackendCommon — 기준 정답), avx2(x86_64 + 런타임 감지 시), opencl.
  • 검증 op 집합은 OpType enum: MatMul / MatMulTransposed / MatMulSlice / Softmax / RMSNorm / RoPE / KiviAttention. dtype·shape 조합별로 돌려 TestResult { status, error, duration, ... }를 수집한다.
  • 판정 기준: 일반 op는 diff > 1e-2 → FAIL. KIVI attention은 양자화 비트별 완화 임계(Q2는 더 큰 quantization error를 허용)를 적용한 L2 error로 판정한다.

OpenCL backend를 고른 경우 KIVI native attention 검증을 위해 동일 ocl 인스턴스에서 파생한 KiviAttentionBackend capability handle을 함께 보관한다 — backend 정확성 테스트가 capability 표면까지 커버한다는 뜻이다.

GPU 커널(.cl)이나 NEON 경로를 건드린 PR은 호스트에서 cargo run --release --bin test_backend, 그리고 디바이스에서 python scripts/run_device.py -d galaxy_s25 test_backend로 양쪽 모두 PASS를 확인해야 한다 — 호스트 OpenCL(있다면)과 Adreno 드라이버의 동작이 다른 사례가 반복적으로 있었다.

근거: engine/src/bin/test_backend.rs::OpType (L30–L39), 판정 (L344: diff > 1e-2, L736–L746: KIVI bits별 threshold), KIVI capability handle (L69–L99)

11.5.3 계층 3: 디바이스 e2e

실모델 추론은 argus_cli(단일 프롬프트 happy path 전용)로 돌린다. argus_cli는 미지원 조합을 침묵 fallback 없이 bail!로 즉시 거부한다 — chat/ppl/eval/weight-swap/KIVI/tensor-partition/profile은 전부 reject 대상이다. 거부 메시지에 향후 이관처(argus-chat/argus-bench/argus-eval)가 명시된다. e2e 판정에 쓰는 출력은 표준출력의 TTFT, Decode: X ms/tok, Avg TBT 라인(11.6절)과 stderr의 [DecodeLoop] kv storage = typed|OPAQUE 같은 경로 증거 라인이다.

근거: engine/src/bin/argus_cli.rs::reject_unsupported_modes_v0 (L49–L98), engine/src/session/assembly/build_standard_loop.rs (L104–L114 kv storage 로그)

eviction e2e를 검증할 때는 긴 프롬프트(약 300 토큰 이상)가 필요하다 — 짧은 프롬프트는 min-cache floor에 막혀 eviction이 no-op이 된다(자세한 동작은 7장).

11.6 성능 측정 — 함정과 원칙

11.6.1 원칙: production 수치는 wall-clock

TBT(token-by-token latency)의 정본은 decode 루프를 감싼 std::time::Instant wall-clock이다. run_standard_happy_path()가 prefill과 decode를 따로 감아 다음을 출력한다:

let t_decode = std::time::Instant::now();
let result = decode_loop.run(args.num_tokens - 1, first_token)?;
let decode_total_ms = t_decode.elapsed().as_secs_f64() * 1000.0;
...
println!("TTFT: {:.2} ms", prefill_ms);
println!("Decode: {:.2} ms/tok ({:.1} tok/s) [{} tokens]", ...);
println!("Avg TBT: {:.2} ms ({:.1} tokens/sec)", ...);

성능 비교(특히 llama.cpp 등 타 엔진과의 비교)는 항상 이 라인을 쓴다. 장기 실행 중 throughput은 resilience executor가 heartbeat에 싣는 actual_throughput(EMA)으로도 관찰할 수 있다.

근거: engine/src/session/standard_happy.rs::run_standard_happy_path (L82–L131), engine/src/resilience/executor.rs (L601 actual_throughput: self.throughput_ema)

11.6.2 왜 per-op 프로파일링은 절대값을 부풀리는가

GPU backend에서 op 단위 시간을 host 쪽 시계로 재려면 op 경계마다 backend.synchronize()(OpenCL에선 clFinish)로 큐를 비워야 한다. 그렇지 않으면 enqueue만 측정되고 실제 커널 실행은 비동기로 흘러가 버리기 때문이다. 문제는 이 sync 자체가 비싸다는 것: production decode는 토큰당 수백 개 커널을 sync 없이 흘려보내는데, op마다 큐를 드레인하면 파이프라이닝이 깨진다. 코드의 자기 서술이 가장 정확하다:

// engine/src/backend/opencl.rs L646-L649 (new_with_profile_events doc)
/// This is the replacement for the legacy `--profile` mode, which
/// inserted two `clFinish()` calls per op and inflated decode ms/tok
/// by ~54 ms on Adreno. Event-based measurement has near-zero CPU
/// overhead because profiling info is collected by the GPU itself.
// engine/src/session/cli.rs L359-L366 (--profile flag doc)
/// Legacy mode: inserts two `clFinish()` calls per op on GPU, which
/// inflates decode ms/tok by ~54 ms on Adreno. Useful for **relative**

즉 --profile류 측정의 per-op breakdown은 "어느 op이 상대적으로 큰가"에만 유효하고, 그 합이나 ms/tok 절대값은 sync 오버헤드(~54 ms/tok, Adreno 실측)로 부풀려져 있다.

11.6.3 --profile이 plan 경로를 끄는 구조

두 번째 왜곡 요인은 plan 경로 이탈이다. OpenCL decode의 정상 경로는 FullKernelPlan — 정적 커널 인자(weight, workspace, 차원)를 plan 빌드 시점에 미리 바인딩해 두고 토큰마다 동적 스칼라(start_pos, cache_seq_len, write_pos)만 갱신하는, llama.cpp식 tight enqueue 루프다. per-op 계측은 이 pre-bound 루프와 양립하지 않으므로, --profile이 켜지면 happy path 가드에서 탈락하고 plan을 못 탄다:

// engine/src/session/assembly/build_standard_loop.rs L60-L71
pub fn is_standard_happy_path(args: &Args) -> bool {
    args.qcf_dump.is_none()
        && args.skip_ratio.unwrap_or(0.0) == 0.0
        && !args.d2o_layer_alloc()
        && !args.profile          // ← profile 활성 시 happy path(=plan 경로) 탈락
        && !args.profile_events
        && args.eviction_policy() == "none"
        && args.tensor_partition == 0.0
        ...

plan 경로 자체의 게이트는 ModelForward::try_build_plan()에 있다: plan_enabled(= !args.no_gpu_plan) → sticky_disabled(한 번 build 실패 시 lock-out) → backend.name() == "OpenCL" → Gemma3 제외, 4중 분기를 통과해야 plan이 빌드된다. plan에서 떨어지면 매 토큰 forward_into() fallback으로 돌므로 per-dispatch CPU 오버헤드가 더해진다. 따라서 profile 켠 측정은 (a) op마다 sync 비용, (b) plan 미사용 비용이 이중으로 합산된 수치다.

근거: engine/src/session/assembly/build_standard_loop.rs::is_standard_happy_path (L60–L71), engine/src/session/forward/model_forward.rs::try_build_plan (L164–L218), engine/src/backend/opencl/plan.rs 모듈 doc (L1–L8)

11.6.4 현재 코드에서 --profile의 실제 상태

현재 트리에는 --profile을 실행하는 바이너리가 없다. 플래그 정의(cli.rs)와 --profile/--profile-events 상호배타 검증(init.rs)은 남아 있지만, 이 경로를 실제로 구동하던 monolith legacy_generate는 커밋 d5ed71d2("5-F F1 — legacy_generate bin + engine/legacy/generate.rs 폐기")에서 제거되었다. 현행 바이너리는 전부 거부한다: argus_cli는 "--profile / --profile-events not yet supported (planned for v1)"로 bail, argus_bench도 동일하게 차단. 따라서 "--profile이 plan을 끄고 op마다 sync 2회를 넣는다"는 운영 지침(CLAUDE.md)은 폐기된 legacy 경로의 동작 설명이며, 메커니즘(왜 부풀려지는가)은 여전히 유효하지만 지금 그 플래그로 측정 자체를 할 수 없다. 현행 계측 수단은 아래 11.6.5의 도구들이다.
