기준 커밋: 46ee4c2d · 작성일: 2026-06-10 · llm.rs Engine Guide

1. 한눈에 보기

KV cache는 decode 중 가장 빠르게 자라는 메모리 소비자다(토큰마다 layer × head × dim 만큼 K/V가 추가). 이 챕터는 (1) 토큰 데이터가 실제로 어떤 바이트 배치로 저장되는가(KVCache + KVLayout), (2) 그 데이터를 누가·언제·어떻게 줄이는가(2-tier 관리: CachePressureHandler 파이프라인 ⊥ EvictionPolicy), (3) 정밀도를 낮추는 두 경로(Q4_0 typed KV, KIVI)를 다룬다. 모든 관리 모듈은 engine/src/kv/ 한 디렉토리에 응집되어 있다.

2. KVCache 자료구조

2.1 필드와 불변식

pub struct KVCache {
    pub k_buffer: Tensor,          // K 저장 버퍼 (dtype = F32/F16/Q4_0/U8(opaque))
    pub v_buffer: Tensor,          // V 저장 버퍼 (K와 동일 geometry)
    pub current_pos: usize,        // 유효 토큰 수 (논리 길이)
    pub high_water_pos: usize,     // current_pos 의 역대 최대값
    pub max_seq_len: usize,        // 논리 상한 — 초과 시 "KV Cache overflow" 에러
    capacity: usize,               // 물리 버퍼 용량 (토큰 단위) — grow/shrink 로 변동
    kv_heads: usize,               // KV head 수 (Llama 3.2 1B = 8)
    head_dim: usize,               // head 차원 (Llama 3.2 1B = 64)
    pub(crate) layout: KVLayout,   // SeqMajor | HeadMajor
    memory: Option<Arc<dyn Memory>>, // Some = dynamic(grow 가능), None = 고정 할당
}

불변식: current_pos ≤ high_water_pos ≤ capacity ≤ max_seq_len. high_water_pos는 "실제로 쓴 적 있는 최대 위치"로, release_unused_pages()가 madvise 범위를 한 번도 안 쓴 tail까지 넓히지 않도록 제한하는 용도다(§3.3). memory가 None(new() 생성)이면 grow()는 에러, shrink_to_fit()은 0을 반환하는 no-op이다.

근거: engine/src/kv/kv_cache.rs::KVCache (struct L44–61), ::new (L66), ::new_dynamic (L87), ::grow (L222–225 에러 분기)

2.2 KVLayout 과 offset 수식

KVLayout은 L2 공유 모듈 kv_cache_ops.rs에 정의된 2-variant enum이다. 4D 논리 shape 기준:

  • SeqMajor: [batch, seq_pos, kv_heads, head_dim] — 한 position의 모든 head가 연속.
  • HeadMajor: [batch, kv_heads, seq_pos, head_dim] — 한 head의 모든 position이 연속.

element 단위 offset/stride 는 다음 세 메서드가 단일 정의처다:

// engine/src/kv/kv_cache.rs
pub fn offset(&self, pos: usize, head: usize) -> usize {
    match self.layout {
        KVLayout::SeqMajor  => pos * self.kv_heads * self.head_dim + head * self.head_dim,
        KVLayout::HeadMajor => head * self.capacity * self.head_dim + pos * self.head_dim,
    }
}
pub fn pos_stride(&self) -> usize  // SeqMajor: kv_heads*head_dim, HeadMajor: head_dim
pub fn head_stride(&self) -> usize // SeqMajor: head_dim, HeadMajor: capacity*head_dim

// Q4_0 은 element 가 아니라 block index:
pub fn q4_block_offset(&self, pos, head, blocks_per_pos) -> usize {
    // SeqMajor:  (pos * kv_heads + head) * blocks_per_pos
    // HeadMajor: (head * capacity + pos) * blocks_per_pos   // blocks_per_pos = head_dim/QK4_0
}

근거: engine/src/kv_cache_ops.rs::KVLayout (L18), engine/src/kv/kv_cache.rs::offset (L183), ::pos_stride (L192), ::q4_block_offset (L204), ::head_stride (L213)

함정: HeadMajor offset에 capacity가 들어간다. HeadMajor의 head stride는 max_seq_len이 아니라 현재 물리 capacity 기준이다. 따라서 grow()/shrink_to_fit()으로 capacity가 바뀌면 모든 head의 base 주소가 이동하고, 재할당 시 head별로 따로 복사해야 한다(§3). capacity를 캐싱해 둔 외부 코드(GPU plan의 stride 인자 등)는 재할당 후 반드시 갱신되어야 한다.
실측 교훈 (CLAUDE.md): production은 HeadMajor 고정. flash attention / attention_scores 자체는 stride 인자를 받아 layout-agnostic이지만, GPU Plan의 kernel_kv_scatter_f32_to_f16(HeadMajor offset 하드코딩)과 qcf/d2o/eviction/shift_positions_for_head가 HeadMajor를 전제한다. SeqMajor 선택 시 Plan 경로가 silent garbage를 낸다(2026-04-14 실측). 코드상으로도 KVCache::new*의 기본값은 BC용 SeqMajor지만, production 할당 경로는 전부 .with_layout(KVLayout::HeadMajor)를 명시한다.

근거: engine/src/session/bin_setup.rs::alloc_standard_kv_caches (L281–324, HeadMajor + new_dynamic), ::alloc_opaque_kv_caches (L331–380), engine/src/session/forward/model_forward.rs (L685)

2.3 write 경로: update() — 입력은 항상 seq-major

update(new_k, new_v)의 입력 tensor는 layout과 무관하게 항상 seq-major [batch, seq_len, kv_heads, head_dim]이다(matmul 출력 그대로). 동작 순서:

  1. 용량 확인: current_pos + seq_len > capacity면 grow(), > max_seq_len이면 "KV Cache overflow" 에러.
  2. opaque 분기: is_opaque()면 F32 입력을 encode_via_descriptor로 양자화하며 HeadMajor byte offset에 직접 scatter(update_opaque, HeadMajor 전용).
  3. SeqMajor: 연속 1회 copy_slice (Q4_0이면 block 단위 count).
  4. HeadMajor: seq-major 입력을 head별 목적지로 scatter — dst = h * capacity * head_dim + (current_pos + s) * head_dim. F16/F32이고 양쪽 버퍼가 GPU 버퍼가 아니며 포인터가 non-null이면 직접 ptr::copy_nonoverlapping fast path(헤드당 ~128B 복사에서 Backend::copy_slice 오버헤드 회피), 아니면 backend.copy_slice 폴백.
  5. current_pos += seq_len; high_water_pos = max(...).
GPU 버퍼 직접 memcpy 금지의 이유 — CUDA managed memory(cuMemAllocManaged)는 non-null host 포인터를 반환하지만 실제로는 GPU 상주 페이지를 가리킨다. host memcpy는 driver migration을 우회해 stale 데이터를 읽고/쓰며 KV cache를 오염시킨다(PPL 폭발). 그래서 can_direct_copy 조건에 !is_gpu_buffer() 검사가 양쪽 모두에 들어 있다.

근거: engine/src/kv/kv_cache.rs::update (L540–689, 주석 L604–622), ::update_opaque (L698–730)

참고로 production write는 KVCache::update를 직접 부르기보다 StandardFormat::write_inner를 경유한다. 거기서 GPU + F16/F32 + HeadMajor decode/prefill이면 fused scatter 커널 (kv_scatter_f32_to_f16 / kv_scatter_f32_to_f32_batch) 1-dispatch fast path를 타고, 비-F32(F16/Q4_0) CPU 경로는 backend.cast로 scratch에 변환 후 update를 부른다(§6.1).

3. capacity 수명주기: grow / shrink / madvise

3.1 grow — doubling, HeadMajor는 per-head 복사

grow(min_capacity)는 new_cap = (capacity * 2).max(min_capacity).min(max_seq_len)으로 새 버퍼를 memory.alloc_kv()로 할당하고 기존 데이터를 복사한다. SeqMajor는 연속이라 copy_slice 1회면 되지만, HeadMajor는 capacity가 head stride 자체이므로 head마다 (h*old_stride → h*new_stride, current_pos*head_dim) 복사를 반복한다. Q4_0이면 모든 offset/count를 QK4_0(=32)으로 나눠 block 단위로 변환하고, opaque면 opaque_bytes_per_head() 기반 byte 단위로 복사하며 새 버퍼를 반드시 OpaqueBuffer로 재포장해 sidecar descriptor를 보존한다(미포장 시 descriptor 소실 → garbage). 공개 진입점은 ensure_capacity(min_tokens)다.

근거: engine/src/kv/kv_cache.rs::grow (L222–364), ::ensure_capacity (L1245–1258)

3.2 shrink_to_fit — 1.5배 headroom, 64-정렬

// 목표 capacity: current_pos 의 1.5배, 64-정렬, 최소 64
let new_cap = ((self.current_pos * 3 / 2) + 63) & !63;
let new_cap = new_cap.max(64);
if new_cap >= self.capacity { return Ok(0); } // 줄어들지 않으면 no-op

grow의 역연산으로 작은 버퍼를 재할당해 복사한다(물리 메모리 해제 보장). 반환값은 (old_buf_size - new_buf_size) * 2 (K+V). next_power_of_two 대신 1.5배를 쓰는 이유는 주석에 명시돼 있다 — 예: current_pos=2049가 4096으로 반올림되면 영원히 shrink가 안 되는 edge를 피한다. opaque는 대칭 구현 shrink_to_fit_opaque(HeadMajor 전용, SeqMajor는 보수적 no-op)로 분기한다.

근거: engine/src/kv/kv_cache.rs::shrink_to_fit (L369–480), ::shrink_to_fit_opaque (L484–533)

3.3 release_unused_pages — shrink 우선, madvise 폴백

eviction 후 CacheManager가 cache마다 호출하는 메모리 반환 진입점. 두 단계다:

  1. 75% 미만 활용이면 shrink: memory.is_some() && current_pos * 4 < capacity * 3이면 shrink_to_fit()으로 재할당. 임계 75%는 H2O keep_ratio=0.5가 capacity/2까지 줄였을 때 shrink가 발동하도록 잡은 값이다(주석).
  2. madvise(MADV_DONTNEED): shrink 미발동 시, host-managed 버퍼(F16/F32)에 한해 [current_pos .. high_water_pos) 구간의 물리 페이지를 OS에 반환한다. HeadMajor는 head마다 따로 advise. 시작 주소는 page 올림, 끝은 내림(부분 사용 페이지 보호). GPU-managed(UnifiedBuffer 등 driver-pinned) 버퍼와 Q4_0/opaque dtype은 skip(0 반환). 완료 후 high_water_pos = current_pos로 리셋.

근거: engine/src/kv/kv_cache.rs::release_unused_pages (L1036–1102), ::madvise_dontneed (L1267–1291), 호출처 engine/src/kv/cache_manager.rs::execute_dispatch (L298–302)

4. element vs block vs byte — 단위 변환 함정

Backend::copy_slice / buffer_shift의 offset/count 인자는 "element 개수"가 아니라 버퍼 dtype의 type_size 단위 개수다. dtype별로 의미가 다르다:

KV dtype1 단위의 크기f32 element → 단위 변환비고
F324 B (element)그대로—
F162 B (element)그대로—
Q4_0BlockQ4_0 = 18 B (32 elem/block)n / QK4_0 (÷32)d: f16 + qs: [u8;16]
opaque (U8)1 B (byte)bytes_for_elems() descriptor 회계bytes_per_head = (head_dim/block_elems) * block_bytes

근거: engine/src/quant.rs::BlockQ4_0 (L14–17, size assert L28 == 18), QK4_0 (L9 = 32), engine/src/kv/kv_cache.rs::opaque_bytes_per_head (L157–164)

실제로 터졌던 버그: Q4_0 shrink_to_fit OOB read → SIGSEGV. grow()는 처음부터 element→block 변환을 했지만 shrink_to_fit()은 누락돼 있었다. element 수를 그대로 count로 넘기면 18B 블록 기준으로 ~32배 over-read가 일어난다. 경로는 eviction → release_unused_pages(75% 미만) → shrink_to_fit. 수정 후 코드에는 경고 주석이 박혀 있다: 
// **Q4_0: copy_slice 의 count/offset 은 element 가 아니라 block 단위다**
// (type_size = sizeof(BlockQ4_0) = 18B). element 로 넘기면 ~QK4_0 배
// over-read → OOB(SIGSEGV). grow() 와 동일하게 변환한다.
KV 버퍼를 만지는 PR을 리뷰할 때는 모든 copy_slice/buffer_shift 호출의 단위를 dtype 분기별로 확인하라. 기존 f16/f32 전용 테스트는 이 버그를 잡지 못했다 (non-dynamic cache는 shrink에 진입하지 않음).

근거: engine/src/kv/kv_cache.rs::shrink_to_fit 주석 (L418–420) 및 Q4_0 block 변환 (L437–445)

5. 2-tier 관리 구조

5.1 상위 tier: CachePressureHandler 파이프라인

상위 trait은 "eviction을 넘어선 일반화된 캐시 관리 동작" 1개를 표현한다. 계약은 단순하다 — handle(ctx)에서 ctx.pressure_level과 캐시 상태를 보고 스스로 행동 여부를 결정하고, 안 하면 ActionResult::NoOp을 반환한다: 

pub trait CachePressureHandler: Send + Sync {
    fn handle(&self, ctx: &mut HandlerContext) -> Result<ActionResult>;
    fn name(&self) -> &str;
}

pub struct HandlerContext<'a> {
    pub caches: &'a mut [KVCache],            // layer 당 1개
    pub importance: Option<&'a [f32]>,        // per-token 누적 attention score
    pub head_importance: Option<&'a [f32]>,   // [n_kv_heads * max_seq_len] row-major
    pub n_kv_heads: usize,                    // 0 = GQA 모드 비활성
    pub pressure_level: PressureLevel,
    pub mem_available: usize,
    pub target_ratio: Option<f32>,            // 외부 신호의 ratio override
    pub qcf_sink: Option<&'a mut Vec<QcfMetric>>,
    pub layer_ratios: Option<&'a [(f32, f32)]>, // D2O per-layer (hh, recent) 배분
}

PressureLevel은 llm_shared::Level의 type alias로 Normal < Warning < Critical < Emergency 순서의 Ord를 derive한다. CachePressurePipeline은 PressureStageConfig { min_level, handler } 목록을 생성 시 min_level 오름차순으로 정렬해 두고, 실행 시 ctx.pressure_level >= stage.min_level인 stage를 순차 실행한다 — 뒤 핸들러는 앞 핸들러가 바꿔 놓은 캐시 상태를 본다(테스트 test_context_updated_after_eviction이 이 의미를 고정: 40 → 20 → 10).

실행 결과 enum은 현재 4-variant다: 

pub enum ActionResult {
    NoOp,
    Evicted { tokens_removed: usize, new_pos: usize },
    Swapped { tokens_swapped: usize },
    WeightSwapped { layers_changed: usize, freed_bytes: usize, duration_ms: f64 },
}

5.2 하위 tier: EvictionPolicy — 영구 삭제 전용 legacy

pub trait EvictionPolicy: Send + Sync {
    fn should_evict(&self, cache: &KVCache, mem_available: usize) -> bool;
    fn evict(&self, cache: &mut KVCache, target_len: usize) -> Result<()>;
    fn name(&self) -> &str;
    // score-aware override 지점 (default 는 위로 위임):
    fn evict_with_scores(&self, cache, target_len, importance: &[f32]) -> Result<()>;
    fn evict_with_head_scores(&self, cache, target_len, flat, head_imp, n_kv_heads) -> Result<()>;
    // Phase α-K: in-place evict 와 동일 의미의 keep-list 산출 (default None = 미지원)
    fn plan_keep(&self, current_pos, target_len, importance) -> Option<(Vec<usize>, Vec<Merge>)>;
}

하위 trait은 토큰 영구 삭제만 표현한다. EvictionHandler 어댑터가 Box<dyn EvictionPolicy>를 감싸 상위 파이프라인에 연결한다 — target_len = current_pos × effective_ratio (외부 신호의 ctx.target_ratio가 있으면 그것 우선)을 계산하고 ScoreContext(None/Flat/PerHead)를 조립한 뒤, 실제 실행은 CacheManager::run_policy_eviction 단일 코어에 위임한다(§5.3). plan_keep은 in-place 변형 대신 보존 목록을 산출하는 plan-returning 이행 경로로, per-head(H2O+)나 가중 merge(D2O)는 단일 layer-wide keep-list로 표현이 불가해 default None이다.

근거: engine/src/kv/eviction.rs::EvictionPolicy (L13–77), engine/src/kv/eviction_handler.rs::EvictionHandler::handle (L43–81)

6.1 SlidingWindowPolicy — 최근 N 토큰

생성자가 protected_prefix = max(4)를 강제한다(attention sink 보호 — 인자로 0~3을 줘도 4). evict 시 keep = target_len.clamp(min_keep, max_keep), max_keep = window + prefix, min_keep = min(prefix + 16, max_keep). prefix 뒤의 오래된 prune_count개를 건너뛰고 나머지를 shift_positions(prefix + prune_count, prefix, remaining) 1회로 당긴다. prefix가 항상 ≥4이므로 prune_prefix 분기는 사실상 dead code다(plan_keep 주석에 명시).

근거: engine/src/kv/eviction/sliding_window.rs::SlidingWindowPolicy::new (L28–35), ::evict (L43–88)

6.2 StreamingLLMPolicy — sink + window 고정 2-영역

[sink(S)] [recent window(W)] 두 영역만 유지. target_len이 0이거나 S+W 이상이면 정확히 S+W로 압축하고, 더 작으면 window를 target_len - S(최소 1)로 줄인다. 중간 gap 전체를 shift_positions(recent_start, sink_size, effective_window) 1회로 제거. 파라미터가 런타임 결정이라 registry 등록 대신 force_evict_by_policy_ref로 호출된다.

근거: engine/src/kv/eviction/streaming_llm.rs::evict (L52–78), engine/src/kv/cache_manager.rs::force_evict_by_policy_ref (L491–514 주석)

6.3 H2OPolicy — 3-파티션 + 누적 score 랭킹

keep = target_len.max(prefix + 2)에서 시작해 예산을 나눈다: 

available     = keep - prefix
hh_budget     = available * keep_ratio      // 기본 0.5 = 논문의 50:50
recent_budget = available - hh_budget
recent_start  = max(current - recent_budget, prefix)

evictable 구간 [prefix .. recent_start)의 (pos, score)를 score 내림차순 정렬 → 상위 hh_budget개를 HH로 선택 → position 오름차순 재정렬(순서 보존) → keep_all = HH ++ [recent_start..current)를 compact_keep_positions(keep_all, prefix)로 압축, current_pos = prefix + keep_all.len(). score가 없으면 prefix+recent만 남기는 sliding 폴백. should_evict는 항상 false — H2O는 resilience 신호로만 발동한다(score 누적은 매 토큰, 결정은 외부).

근거: engine/src/kv/eviction/h2o.rs::evict_with_scores (L85–176), ::should_evict (L42–48)

6.4 H2OPlusPolicy — GQA-aware per-head HH 선택

예산 산식은 H2O와 동일하되, head_importance[kv_h * max_seq + pos]로 KV head마다 독립적으로 HH를 고른다. 모든 head가 같은 개수(hh_budget)를 유지하되 다른 토큰을 남긴다 — 단일 current_pos 불변식을 지키기 위한 설계다. head별로 compact_keep_positions_for_head(kv_h, keep_all, prefix)를 호출하므로 HeadMajor가 필수다 (SeqMajor면 shift_positions_for_head의 assert로 panic). 압축 후 current_pos = prefix + hh_budget + actual_recent.

주의: per-head eviction 이후에는 "position p"가 head마다 다른 원본 토큰을 가리킨다. RoPE position과 물리 위치가 head별로 어긋나는 것을 허용하는 근사이며, 이후의 per-token 연산(예: flat score 재사용)은 의미가 약해진다. 리뷰 시 H2O+ 경로 뒤에 토큰 단위 가정을 하는 코드가 추가되지 않는지 확인하라.

근거: engine/src/kv/eviction/h2o_plus.rs::evict_with_head_scores (L107–167)

6.5 D2OHandler — eviction + 보상 merge

D2O는 EvictionPolicy가 아니라 CachePressureHandler를 직접 구현한다 — "가중 병합"은 영구 삭제 trait 표면에 담기지 않기 때문이다. 기본값: keep_ratio 0.75(논문 N:M=3:1), protected_prefix 4, target_ratio 0.5, ema_beta 0.7, merge_e 0.1. 상태(ema_threshold, merge/delete 누계)는 Mutex<D2OState>로 호출 간 누적된다. layer 루프 단계:

  1. 계획 (mutation 없음): compute_d2o_plan — H2O식 3-파티션으로 retain/evict 분할.
  2. nearest 매칭: 각 evicted 토큰을 layer-wide K cosine(전 head concat K, head별이 아님)으로 가장 가까운 retained 토큰에 단일 argmax 매칭 (K는 dtype별 dequant reader 경유).
  3. EMA 임계 τ (Eq.10): 첫 호출은 max-sim 평균으로 초기화, 이후 τ = β·max(sim) + (1−β)·τ.
  4. 필터: sim ≥ τ인 토큰만 merge 대상, 나머지는 그냥 삭제.
  5. scatter-reduce merge (Eq.11): retained c_j와 그 그룹 {e_i}에 대해 D = Σ exp(u_ij) + e, w_c = e/D, w_ei = exp(u_ij)/D — 가중치 합 1 보장. K/V 모두, 모든 head에 같은 layer-wide 가중치 적용. dtype별 구현: F32/F16 직접, Q4_0은 블록 dequant → 가중합 → requantize.
  6. compact: compact_keep_positions(retain_all, 0) 후 current_pos = retain_all.len().

가드/폴백: importance가 없으면 NoOp. KV 버퍼가 GPU-only(as_ptr() == null, discrete GPU)면 CPU에서 K를 읽을 수 없어 merge를 끄고 H2O식 eviction-only로 강하한다. ctx.layer_ratios가 있으면 layer별 (hh_r + rec_r) 비율로 target을 차등 적용하고 (Phase B layer allocation, d2o_layer_alloc.rs), 이때 마지막 layer는 항상 보호된다. 같은 파일의 D2OStage는 동일 compute_d2o_plan을 공유해 plan-returning KVCacheStage(technique-api) 표면으로 노출한 변형이다 — 버퍼 변형은 엔진 executor가 실행.

근거: engine/src/kv/d2o_handler.rs::D2OConfig (L26–59), ::compute_d2o_plan (L204–305), ::handle (L308–405, GPU 가드 L342), ::scatter_reduce_merge_layer_wide (L721–763), ::compute_eq11_weights (L700–710), ::D2OStage (L423–451)

7. KV quantization 상세

7.1 Q4_0 typed KV — KVCache 그대로, dtype만 교체

Q4_0 KV는 별도 타입이 아니라 KVCache의 k_buffer.dtype() == DType::Q4_0인 경우다. 저장 단위는 BlockQ4_0(scale d: f16 + 4-bit×32 packed qs: [u8;16] = 18 B, 0.5625 B/elem). 대칭 양자화: d = max|x| / 7, 값을 [−8, 7]로 클램프해 nibble 2개씩 packing.

쓰기 경로의 양자화 책임은 KVCache가 아니라 호출자에 있다 — KVCache::update는 입력이 이미 cache dtype이라고 전제하고 block 단위 복사만 한다. production에서는 StandardFormat::write_inner의 비-F32 cast 경로가 F32 입력을 backend.cast(new_k, k_cast)로 Q4_0 scratch(memory.alloc lazy 할당, shape 변경 시 재할당)에 변환한 뒤 update를 부른다. Q4_0은 GPU fused scatter fast-path가 없어 GPU에서도 이 cast 경로다.

읽기(attention) 경로: CPU backend는 attention_gen이 dtype-aware라 Q4_0을 자체 처리하지만, GPU에는 Q4_0 dequant-attention 커널이 없어 BlockQ4_0 raw 바이트를 float로 오독한다 — 그래서 StandardFormat::attention_into가 kv_dtype == Q4_0 && backend.is_gpu()면 CPU 폴백 경로로 우회한다.

근거: engine/src/quant.rs::BlockQ4_0::quantize (L35–49), engine/src/kv/standard_format.rs::write_inner 비-F32 cast 분기 (L259–324), Q4_0+GPU attention 우회 주석 (L459–464)

7.2 KIVI — KiviCache: 양자화 본체 + F32 residual

KIVI(ICML 2024)는 "최근 R 토큰은 F32로 두고, 가득 차면 배치로 양자화해 압축 저장소로 flush"하는 설계다. KVCache와 무관한 별도 타입 KiviCache이며, 양자화를 외부 cast가 아니라 자기 내부에서 수행한다(kv_dtype()이 F32를 반환해 layer가 F32를 그대로 넘기게 함). 핵심 상태: 

pub struct KiviCache {

7.3 flush_residual — K/V 양자화 축이 다르다

residual이 차면 flush_residual()이 n_groups = res_pos / 32개 그룹을 압축한다. KIVI의 핵심 비대칭:

  • Key = per-channel: head별·채널별로, 토큰 축을 가로질러 32개 값을 한 block으로 (vals[t] = res_k[head_base + (tok_start+t)*head_dim + ch]). K의 채널별 outlier 분포에 맞춘 설계.
  • Value = per-token: head별·토큰별로, 그 토큰의 head_dim을 32개씩 잘라 block으로.

flush 직전(F32 원본이 남아 있을 때) NMSE/OPR proxy를 계산해 flush_proxies에 쌓고 (QCF_kv 패밀리, 04장의 score 경로와 연결), awqe_enabled이고 직전 decode의 post-softmax score 스냅샷이 있으면 AWQE/AW-VOPR도 추가한다. flush 후 남은 토큰(res_pos % 32)은 residual 앞으로 copy_within으로 당긴다. bits == 16이면 flush 자체가 no-op(전 시퀀스 F32 residual-only 모드).

근거: engine/src/kv/kivi_cache.rs::flush_residual (L901–1026: K per-channel L980–993, V per-token L995–1007)

7.4 읽기 경로와 layout 주의

get_view()는 압축분을 증분 dequant(q2_deq_tokens 마커로 새 flush만 처리)해 attn_k/v_buf에 조립하고, SharedBufferView로 유효 영역만 노출하는 zero-copy F32 view를 반환한다. GPU 모드(new_gpu, OpenCL 한정)는 residual/attn 버퍼를 GPU에 상주시키고 hot path를 커널로 처리하되 양자화(flush)는 cold path라 CPU에 남긴다. layout() 반환값이 모드에 따라 다른 점에 주의: bits=16 GPU 모드만 HeadMajor (residual [kv_heads, res_cap, head_dim]가 그대로 attention 대상이라 stride 계산에 capacity() = res_cap 사용), 나머지(CPU 전부, bits 2/4/8 GPU의 assembled view)는 SeqMajor다.

KIVIFormat(format wrapper)의 attention_into dispatch 순서: (1) seq_len > 1(prefill)이면 KIVI native multi-token 커널이 없어 dequant view + StandardFormat::prefill_attention 재사용. (2) decode에서 backend.is_gpu() && as_kivi_attention() && has_kivi_attn_kernel(bits) && is_nosub_device() && 토큰 > 0이면 dequant 없이 KiviRawBuffers(qk/qv blocks + F32 residual)를 직접 읽는 fused 커널 attention_gen_kivi dispatch. (3) 그 외에는 F32 view → backend.attention_gen 폴백. is_nosub_device는 NVIDIA류를 뜻한다 — Adreno(subgroup 지원)는 F32 dequant 경로가 더 빨라 native를 쓰지 않는다(코드 주석의 게이팅 보존).

근거: engine/src/kv/kivi_cache.rs::get_view (L2355–2395), ::layout (L2278–2287), engine/src/kv/kivi_format.rs::KIVIFormat::attention_into (L86–203), ::attention_native (L212–291), engine/src/kv_cache_ops.rs::KiviRawBuffers (L27–44)

7.5 bit transition 과 QuantizeHandler

transition_bits(new_bits)는 기존 압축 블록을 F32로 dequant 후 새 bit로 재양자화한다 (2/4/8 ↔ 16 모두 지원, 오차는 dequant→requant 사이클마다 누적). 압력 레벨과 bit의 매핑은 QuantizeHandler::target_bits_for_pressure가 정의한다: Warning→8, Critical→4, Emergency→2, Normal→None. 단 QuantizeHandler의 handle() 자체는 항상 NoOp인 stub이다 — HandlerContext가 &mut [KVCache]만 들고 있어 KiviCache에 접근할 수 없기 때문. 실제 transition은 파이프라인 밖에서 매핑 함수를 직접 호출해 수행하도록 설계되어 있다(주석).

근거: engine/src/kv/kivi_cache.rs::transition_bits (L1034–), engine/src/kv/quantize_handler.rs::target_bits_for_pressure (L30–37), ::handle (L47–59)

7.6 SwapHandler — 참고: 정밀도가 아닌 위치를 바꾸는 핸들러

같은 디렉토리의 SwapHandler는 quantization이 아니라 디스크 offload다. 두 모드: (1) lossy — 가장 오래된 total × offload_ratio 토큰을 그냥 prune, (2) disk write-back — prune 전에 prefix 구간을 swap_dir/cache_L{layer}_{start}-{end}_{k,v}.bin으로 덤프하고 SwapRecord 장부에 기록, recall_caches로 복원 가능. 디스크 모드는 F32/F16만 지원하고 Q4_0/KIVI는 경고 후 lossy로 강하한다. CacheManager::enable_swap으로 만들어지며 압력 파이프라인에는 등록되지 않는다 — Manager의 KvOffload/RestoreDefaults 지시에만 반응한다.

근거: engine/src/kv/swap_handler.rs (모듈 doc L1–16, SwapRecord L30, offload_caches L89), engine/src/kv/cache_manager.rs::enable_swap (L111–113)

9. 코드만으로 확정하지 못한 것 / 의심 지점

  • QuantizeHandler는 영구 stub인지(파이프라인 등록 자체가 무의미), KIVI transition을 실제로 호출하는 외부 루프 위치가 어디인지 — session 전체를 추적하지 않아 본 챕터에서는 매핑 함수만 기술했다.
  • StandardFormat/KIVIFormat 주석에 "unwired (production 호출처 0)"가 남아 있으나 일부 메서드(take_inner의 UER seam 등)는 chat 멀티턴 경로를 언급한다 — fmt 경로의 현재 wiring 범위는 04장의 forward 경로 분석과 교차 확인 필요.
  • determine_pressure_level의 threshold_bytes가 CLI에서 어떤 기본값으로 주입되는지 미확인.
  • H2O+에서 per-head 압축 후 flat importance 배열은 더 이상 물리 위치와 정합하지 않는데, 후속 eviction 호출에서 score 배열을 어떻게 재정렬/리셋하는지는 score accumulator 쪽(범위 밖) 확인 필요.
