10. 확장 How-to

기준 커밋: 46ee4c2d · 작성일: 2026-06-10 · llm.rs Engine Guide

0. 이 챕터의 사용법

이 챕터는 llm.rs 엔진을 확장하는 4가지 대표 시나리오를 step-by-step으로 다룬다. 각 시나리오마다 기존 구현체 1개를 코드에서 끝까지 따라가며(trait 계약 → 생성 지점 → 등록 지점 → 소비 지점) 등록 지점을 전수 추적하고, 마지막에 "건드릴 파일 체크리스트" 표로 정리한다. 도메인 개념(KV 캐시, 백엔드)의 상세는 각 챕터에 위임하고, 여기서는 확장에 필요한 등록 배선만 본다.

4개 축의 개방 정도가 다르다는 점이 이 코드베이스의 핵심 특징이다. stage 축(시나리오 b)만 진짜 open(레지스트리 기반, 엔진 코어 수정 0)이고, 나머지 3개는 enum/match 기반 closed dispatch라서 엔진 소스를 직접 수정해야 한다. backend 축도 capability 한정으로 dlopen 레지스트리가 추가되는 중이다(§3.5).

1. 시나리오 (a) — 새 CachePressureHandler 추가

1.1 trait 계약

CachePressureHandler는 KV 캐시 관리 2-tier 추상화의 상위 trait다. 계약은 메서드 2개가 전부이며, 핵심 제약은 &self다 — 파이프라인이 핸들러를 Box<dyn CachePressureHandler>로 소유하고 불변 참조로 호출하므로, 내부 상태가 필요한 핸들러는 interior mutability를 써야 한다 (실제로 D2OHandler가 Mutex<D2OState>로 EMA 상태를 든다). 

pub trait CachePressureHandler: Send + Sync {
    fn handle(&self, ctx: &mut HandlerContext) -> Result<ActionResult>;
    fn name(&self) -> &str;   // 로깅/디버깅용
}

1.2 파이프라인 게이트 — 언제 호출되는가

핸들러는 PressureStageConfig { min_level, handler }로 파이프라인에 묶인다. CachePressurePipeline::new()는 stage들을 min_level 오름차순으로 정렬하고, execute()는 ctx.pressure_level >= stage.min_level인 모든 stage를 순서대로 실행한다. 같은 레벨 핸들러 여러 개도 모두 실행되며, 뒤 핸들러는 앞 핸들러가 변형한 캐시 상태를 본다(순차 합성).

PressureLevel의 산출은 CacheManager::determine_pressure_level()이 담당하며 경계값이 코드에 고정돼 있다: 

mem_available >= threshold        → Normal     (파이프라인 미실행)
mem_available >= threshold / 2    → Warning
mem_available >= threshold / 4    → Critical
그 미만                            → Emergency
// threshold = --memory-threshold-mb * 1024 * 1024

단, force_evict*() 계열(budget eviction, resilience directive)은 메모리 검사를 우회하고 항상 Emergency로 실행하므로 모든 stage의 min_level 게이트가 열린다. 새 핸들러를 설계할 때 "Warning에서만 동작"을 의도했다면 force 경로에서도 호출된다는 점을 handle() 내부에서 방어해야 한다.

근거: engine/src/pressure.rs::CachePressurePipeline::{new,execute} (L174, L184), engine/src/pressure/cache_manager.rs::{determine_pressure_level, execute_dispatch} (L153, L201)

2. 시나리오 (b) — 새 EvictionPolicy(= KVCacheStage) 추가

2.1 두 세계: legacy EvictionPolicy와 plan-returning KVCacheStage

eviction 정책 표면은 마이그레이션 중이라 두 trait이 공존한다. legacy는 EvictionPolicy — evict(&self, cache: &mut KVCache, target_len)가 버퍼를 in-place로 직접 변형한다. 정석(World B)은 technique_api::KVCacheStage — plan(&self, ctx: &dyn StageCtx) -> Option<KVCachePlan>이 "무엇을 남길지"만 계산해 반환하고, 버퍼 변형은 엔진 executor가 독점한다(plan-returning). 빌트인 sliding/streaming/h2o는 이미 production에서 World B 경로(레지스트리 → plan → compact)로 돈다. 

// crates/technique-api/src/lib.rs
pub trait KVCacheStage: Send + Sync {
    fn name(&self) -> &str;                                  // --eviction-policy 이름과 매칭, 유일해야 함
    fn plan(&self, ctx: &dyn StageCtx) -> Option<KVCachePlan>; // None = no-op
}
pub struct KVCachePlan {
    pub keep: KeepSpec,            // LayerWide(Vec<usize>, prefix 포함 ascending) | PerHead(Vec<Vec<usize>>)
    pub merges: Vec<WeightedMerge>, // 가중 병합 지시 (Σ weight ≈ 1), 없으면 빈 Vec
}
#[repr(C)]
pub struct StageParams {           // 엔진이 CLI args를 평탄화해 전달
    pub eviction_window: usize, pub protected_prefix: usize,
    pub keep_ratio: f32, pub sink_size: usize, pub streaming_window: usize,
}

근거: crates/technique-api/src/lib.rs::{KVCacheStage, KVCachePlan, KeepSpec, StageParams} (L220, L206, L196, L237)

2.2 기존 구현체 추적 — SlidingWindowPolicy의 알고리즘과 등가 plan

SlidingWindowPolicy::new(window_size, protected_prefix)는 생성자에서 protected_prefix = protected_prefix.max(4)로 clamp한다(attention sink 보장). evict()의 keep 길이 산식: 

max_keep = window_size + protected_prefix
min_keep = min(protected_prefix + 16, max_keep)
keep     = target_len.clamp(min_keep, max_keep)
// current_pos <= keep 이면 no-op
// prefix > 0: cache.shift_positions(prefix + prune_count, prefix, remaining); current_pos -= prune_count

같은 정책의 plan_keep()은 동일 산식으로 keep-list [0..prefix) ∪ [prefix+prune_count..current)를 ascending으로 산출만 하고 버퍼는 건드리지 않는다. 두 경로의 byte-equality는 compact_parity 테스트(4정책 × F32/F16/Q4_0)가 게이트한다.

근거: engine/src/pressure/eviction/sliding_window.rs::{new, evict, plan_keep} (L28, L43, L98), engine/src/pressure/eviction/compact_parity.rs

2.4 score 기반 정책이면: score_based 목록

importance가 필요한 정책(H2O류, CAOTE)은 decode 루프가 force_evict_with_scores()로 score를 흘려보내야 KVStageCtx의 importance()가 Some이 된다. 이 여부는 정책 이름의 하드코딩된 matches! 목록으로 결정된다: 

// session/chat/session.rs L671
let score_based = matches!(args.eviction_policy.as_str(), "h2o" | "h2o_plus" | "d2o" | "caote");

새 score-aware 정책을 레지스트리에만 등록하고 이 목록을 빠뜨리면, 정책은 정상 선택되지만 importance가 영영 None으로 들어와 조용히 degenerate한다(CAOTE는 weight=0). 전형적인 silent failure 지점.

근거: engine/src/session/chat/session.rs (L667–674)

2.5 per-head / 가중 merge 정책의 한계

plan 표면의 현재 경계: KeepSpec::PerHead는 엔진 executor가 bail!("per-head executor not implemented")로 미구현이고, 그래서 per-head 정책 H2OPlusPolicy는 레지스트리를 타지 않고 조립 지점에서 레거시 직생성으로 남아 있다 (plan_keep() default가 None을 반환해 "단일 layer-wide keep으로 표현 불가"를 표시). 가중 merge가 필요한 d2o는 D2OStage가 "d2o"로 등록은 돼 있으나 production은 여전히 if-branch가 가로채 D2OHandler(layer-alloc 지원)로 처리한다. 새 정책이 LayerWide keep + merge로 표현되면 레지스트리 경로로 충분하고, per-head가 필요하면 executor부터 만들어야 한다.
