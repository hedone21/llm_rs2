1. Backend — 연산 dispatch의 단일 표면

1.1 시그니처 발췌

Backend는 Send + Sync trait이며 약 70개 메서드를 가진다. 그중 required는 소수이고 나머지는 전부 default 구현이 있는 "opt-in 사다리"다. 새 backend를 추가할 때 반드시 구현해야 하는 것은 아래뿐이다. 

pub trait Backend: Send + Sync {
    fn as_any(&self) -> &dyn std::any::Any;
    fn name(&self) -> &str;
    fn device(&self) -> &str;

    // required 연산 코어
    fn matmul(&self, a: &Tensor, b: &Tensor, out: &mut Tensor) -> Result<()>;
    fn matmul_transposed(&self, a: &Tensor, b: &Tensor, out: &mut Tensor) -> Result<()>;
    fn matmul_slice(&self, a, b, rows, cols, out) -> Result<()>;
    fn add_assign(&self, a: &mut Tensor, b: &Tensor) -> Result<()>;
    fn scale(&self, x: &mut Tensor, v: f32) -> Result<()>;
    fn silu_mul(&self, a: &mut Tensor, b: &Tensor) -> Result<()>;
    fn rms_norm(&self, x: &mut Tensor, w: &Tensor, eps: f32, add_unit: bool) -> Result<()>;
    fn softmax(&self, x: &mut Tensor) -> Result<()>;
    fn rope_inplace(&self, x: &mut Tensor, start_pos: usize, theta: f32) -> Result<()>;
    fn copy_from(&self, t: &Tensor) -> Result<Tensor>;
    fn cast(&self, src: &Tensor, dst: &mut Tensor) -> Result<()>;

    // required지만 한 줄짜리 — default 불가 (아래 1.2 참조)
    fn cpu_companion(&self) -> &dyn Backend;

    // 이하 전부 default 제공: attention_gen, fused_norm_merge,
    // kv_scatter_*, copy_slice, buffer_shift, enqueue_*_async,
    // yield_after_layer, gpu_score_acc, as_kivi_attention, get_extension, ...
}

근거: engine/src/backend.rs::Backend (trait, L128–1322)

fused op는 "default = 비융합 합성, override = 단일 커널" 패턴이다. 예를 들어 matmul_ffn_gate_up_silu의 default는 gate matmul → up matmul → silu_mul 3-op fallback이고, OpenCL backend가 이를 단일 커널로 override한다. fused_norm_merge(tensor-partition 레이어 경계의 3-입력 merge + RMSNorm 융합)도 동일 패턴인데, 여기엔 명시적 aliasing 계약이 있다: residual_out은 prior_residual과 alias 가능(in-place), out은 어떤 입력과도 alias 금지, 세 partial 입력은 서로 distinct 버퍼여야 한다.

근거: engine/src/backend.rs::matmul_ffn_gate_up_silu (L176), ::fused_norm_merge (L255, aliasing 계약 doc L247–249)

1.2 계약 / 불변식

  • offset은 element 단위, Q4_0의 1 element = 1 block = 18바이트. copy_slice / buffer_shift의 src_offset/dst_offset/count는 바이트가 아니라 element 단위이며, 내부에서 src.dtype()으로 type_size를 정한다 (F32=4, F16=2, U8=1, Q4_0=size_of::<BlockQ4_0>()=18, 그 외 dtype은 fallback 1). 호출자는 src/dst dtype 일치를 스스로 보장해야 한다 — trait 차원의 검증은 없다.
  • supports_* 플래그를 확인하지 않고 batch scatter를 부르면 segfault. kv_scatter_f32_to_f16_batch의 default 구현은 as_ptr()/as_mut_ptr() 호스트 포인터 슬라이스를 쓰므로 device-only 버퍼(zero-copy 매핑 없는 OpenCL)에서 segfault한다. 호출자는 반드시 supports_kv_scatter_batch() (default false)를 먼저 확인하고, false면 per-position cast + update 경로로 폴백해야 한다. supports_kv_scatter_f32_batch/supports_async_transfer/supports_layer_graph도 같은 caller-guard 패턴.
  • GpuEvent의 dummy 의미론. async 메서드(enqueue_read_buffer_async 등)의 default는 동기 실행 후 GpuEvent::dummy()(inner 핸들 모두 None)를 반환한다. is_dummy()가 true인 이벤트에 대한 wait는 no-op. 단 wait_event_blocking의 default는 전체 synchronize() 배리어다 — 정확하지만 compute와의 overlap을 잃으므로, 실제 이벤트를 반환한 backend는 per-event wait로 override해야 한다.
  • cpu_companion은 default가 없다(의도적). default로 self를 반환하려면 &Self → &dyn Backend coercion에 Self: Sized bound가 필요한데, 그러면 &dyn Backend를 통한 dispatch가 막힌다. 그래서 모든 구현체가 한 줄 override를 직접 쓴다 — CPU backend는 self, GPU backend는 construction 시 주입받은 owned companion.
  • get_extension은 cold path 전용. forward/decode loop 안 호출 금지(ggml의 get_proc_address 차용, string-keyed Any downcast). key는 자유 문자열이 아니라 모듈 const만 허용: EXT_OPENCL_QUEUE, EXT_OPENCL_SECONDARY, EXT_RPCMEM_ALLOCATOR. 미등록 key는 silent None.
  • yield_after_layer는 매 layer 호출되는 hot hook. default body가 LLMRS_DECODE_YIELD_EVERY/LLMRS_DECODE_YIELD_US env로 게이트되며, env 미설정 시 yield_every()==0 short-circuit으로 사실상 zero-cost. 동작 시엔 synchronize()로 in-flight 커널을 drain한 뒤 sleep한다 (에러는 swallow — yield는 best effort).
  • weight swap 후 SOA registry 순서 계약. invalidate_noshuffle_soa_registry()는 non-empty swap batch당 정확히 1회, ratio_generation bump 이전에 호출되어야 하고(ENG-ALG-221/INV-130), 그 직후 ensure_noshuffle_soa_registered/restore_pre_converted_soa_registration으로 새 cl_mem 주소가 재등록되어야 다음 plan rebuild가 lookup miss(silent garbage) 없이 진행된다(INV-131).

근거: engine/src/backend.rs::copy_slice (L1096), ::kv_scatter_f32_to_f16_batch (L487, guard doc L468–480), ::GpuEvent (L56), ::wait_event_blocking (L669), ::cpu_companion (L1197, rationale L1192–1196), ::get_extension (L1319), ::yield_after_layer (L1227), ::invalidate_noshuffle_soa_registry (L777)

함정: copy_slice의 element↔block 회계는 trait가 검증해 주지 않는다. Q4_0 텐서에 element 개수를 그대로 넘기면 32배(QK4_0) 가까운 OOB가 난다 — KV 캐시 쪽 호출자(grow/shrink_to_fit 등)가 element→block 변환을 누락한 사례가 실제로 SIGSEGV로 이어진 적이 있다. Q4_0 텐서를 만지는 PR에서는 offset 단위를 반드시 확인하라. 또한 default 메서드 다수가 raw 포인터 기반 CPU 폴백이므로, GPU 전용 버퍼 위에서 "컴파일은 되는데 죽는" 형태로 실패한다.

1.4 왜 이 모양인가

단일 god-trait + default 사다리는 "새 backend가 required 최소만 구현하면 정확하게 동작하고, 성능은 override로 점진 획득"이라는 온보딩 곡선을 만든다. default 구현 자체가 CPU reference라서 test_backend 바이너리의 CPU vs GPU 비교 검증과도 맞물린다. 예컨대 default attention_gen은 F32/F16 KV를 모두 처리하고, layout을 shape 휴리스틱 (k_shape[1] == num_heads_kv && k_shape[1] != k_shape[2] → HeadMajor)으로 감지해 HeadMajor offset (kv_h * capacity + t) * head_dim을 계산한다 — GPU 커널이 따라야 할 의미론의 정본이다.

한편 trait 후반부의 gpu_score_acc/as_kivi_attention/profile_events_enabled 류 hook은 과거 hot path에 흩어져 있던 as_any().downcast_ref::<OpenCLBackend>() 체인을 trait 메서드 1곳으로 치환한 흔적이다. 이 hook들은 다시 CapabilityRegistry(§5.1)로 이동 중이라 — "god-trait에 메서드 추가" 대신 "capability trait 신설 + registry 등록"이 현재 권장 확장 방식이다. cuda/cuda-embedded를 compile_error!로 상호배타시킨 것은 backend::cuda alias가 한 빌드에서 단일 의미를 갖게 하기 위함이다.

근거: engine/src/backend.rs::attention_gen (default impl L285–406, layout 감지 L337–340), capability hook 주석 (L1149–1184, §13.8-L L1264–1307)

실측 교훈 (CLAUDE.md): 성능 비교는 항상 wall-clock으로 — --profile은 매 op마다 backend.synchronize()를 2회 호출해 ~54 ms/token의 sync 오버헤드를 더하므로 절대값이 부풀려진다. backend별 커널 세부(.cl 커널 정책, Adreno register spill 등)는 09. 백엔드 구현에 위임한다.

2. TensorHandle / StageCtx — plugin의 읽기 표면

2.1 시그니처 발췌

// crates/technique-api/src/lib.rs
#[repr(u32)]
pub enum TensorKind { Key, Value, AttnWeights, Scores }

#[repr(C)]
pub struct TensorShape { pub rows: usize, pub cols: usize, pub per_head: bool }

pub trait TensorHandle {
    fn shape(&self) -> TensorShape;
    fn dtype(&self) -> TensorDtype;            // F32 | F16 | Q4_0 (진단용)
    fn read_row(&self, row: usize, kv_head: usize, out: &mut [f32]);
}

pub trait StageCtx {
    fn current_pos(&self) -> usize;            // 유효 토큰 수
    fn target_len(&self) -> usize;             // 엔진이 해소한 절대 budget
    fn layer_idx(&self) -> usize;
    fn importance(&self) -> Option<&[f32]>;    // flat per-token (zero-copy 예외)
    fn n_kv_heads(&self) -> usize;
    fn head_dim(&self) -> usize;
    fn tensor(&self, kind: TensorKind) -> Option<&dyn TensorHandle>;  // ★ 단일 메커니즘

    // 이하 default sugar — 전부 tensor() 위임, 엔진은 override 불필요:
    // dequant_k / dequant_v / head_score / has_head_scores / attn_weight / has_attn_weights
}

근거: crates/technique-api/src/lib.rs::TensorKind (L33), ::TensorHandle (L71), ::StageCtx (L93–177)

2.2 계약 / 불변식

  • 읽기 산출은 항상 f32 out-param. read_row는 슬라이스를 반환하지 않고 out: &mut [f32]를 채운다. out.len() == shape().cols가 호출자 계약 (Key/Value는 cols=head_dim, AttnWeights/Scores는 cols=1). dtype 분기(F32/F16/Q4_0 dequant)는 핸들 구현이 내부 흡수한다.
  • dyn-safe 필수. plan(&self, ctx: &dyn StageCtx)가 trait object로 받으므로 제네릭 파라미터· 연관 타입·impl Trait 인자 금지. 이것이 out-param 규약의 직접 원인이다.
  • tensor(kind)가 단일 텐서 접근 메커니즘. dequant_k/head_score/attn_weight 등은 전부 tensor() 위의 default sugar — 엔진 구현체는 tensor() 하나만 구현하면 된다. 새 입력(Query 등)은 TensorKind variant 추가 1줄 + 엔진 impl 1곳으로 끝난다(OCP). 유일한 예외가 flat importance()인데, scalar를 per-element read_row로 돌리면 H2O 랭킹 경로가 순손해라 zero-copy 슬라이스를 직접 노출한다.
  • kind 미가용 = None = graceful degrade. score-free 정책은 tensor(Scores)==None을 보고, sugar는 no-op(out 불변) 또는 0.0을 돌려준다. AttnWeights는 "직전 decode step의 last-layer per-head attention weight" 근사값이므로 has_attn_weights() 게이트 확인이 계약이다.
  • 호출 간 상태는 ctx로 thread하지 않는다. d2o EMA 같은 누적 상태는 plugin struct가 &self + interior mutability(Mutex)로 보유한다(ADR-0004 D4).
  • FFI-forward. TensorShape는 #[repr(C)], enum들은 #[repr(u32)] — 동일 타입이 .so C-ABI 경계(StageCtxAbi fn-ptr 테이블)를 reshape 없이 건넌다.

근거: crates/technique-api/src/lib.rs::StageCtx doc (L83–92), ::dequant_k (L129), ::importance 예외 doc (L91–92), ::StageCtxAbi (L300)

2.4 왜 이 모양인가

technique-api는 KVCache/Backend 등 엔진 타입을 참조하지 않는다 — 기법 crate가 엔진 내부에 결합하면 "엔진 코어 수정 0으로 기법 추가"라는 목표 자체가 깨지기 때문이다. 그래서 캐시 읽기를 이 crate가 정의하는 추상 (StageCtx)으로 노출하고, 엔진이 그것을 &KVCache 위로 구현한다. 정적 링크 단계에선 borrow로, .so 단계에선 같은 추상이 fn-ptr 테이블로 교체되어 plugin 저자는 정적/동적 무관하게 동일한 impl KVCacheStage 코드를 쓴다("write-once, link-either-way"). tensor(kind) 단일화는 value-aware 기법(CAOTE의 v_i)을 위해 V 노출이 필요해지자 accessor를 늘리는 대신 kind 열거로 통합한 결과다 — read dispatch 비용은 additive accessor와 동등(PoC ±0~1%)함을 확인하고 채택했다.

3.2 계약 / 불변식

  • 결정은 plugin, 변형은 엔진(D1). stage는 버퍼를 직접 만지지 않는다. plan을 엔진 executor execute_kv_plan이 실행한다: merges가 있으면 apply_weighted_merges(pre-compact 좌표계, F32/F16/Q4_0 모두 scatter_reduce와 bit-identical) → compact_keep_positions(keep, 0) → set_current_pos(keep.len()).
  • keep은 prefix 포함 ascending. 그래야 compact 재배치가 in-place evict의 버퍼/current_pos와 bit-identical하다(compact_parity 테스트가 4정책×3dtype에서 증명).
  • new_pos는 plan에 싣지 않는다. 엔진이 keep.len()으로 도출한다. PerHead는 전 head 길이 동일 가정 — 단, PerHead executor는 현재 미구현으로 도달 시 bail!한다(h2o_plus head_score source 미완, 단계 ⑤ deferred).
  • merge 가중치 정규화: Σ from.weight + into_weight ≈ 1 (magnitude 보존). merge-free 정책은 빈 Vec.
  • budget 환산은 엔진 책임. ratio→len 환산(EvictionHandler)이 끝난 절대값을 ctx.target_len()으로 읽는다. "언제 발동하는가"(WHEN)도 엔진 소유 — StageBackedPolicy::should_evict는 항상 true를 돌려주고 실제 게이트는 run_policy_eviction의 target_len/MIN_EVICT 가드가 담당한다(ADR-0004 D6).

3.4 왜 이 모양인가 — 어댑터가 양방향으로 2개인 이유

위 다이어그램에서 눈에 띄는 것은 EvictionPolicyAsStage(EvictionPolicy → KVCacheStage)와 StageBackedPolicy(KVCacheStage → EvictionPolicy)가 동시에 존재한다는 점이다. 이것은 branch-by-abstraction 마이그레이션의 스냅샷이다: production eviction 경로(run_policy_eviction → evict*)의 구조는 그대로 두면서, 내부 실행을 World A(in-place evict)에서 World B(plan→compact)로 바꿨다. session 조립부(session/chat/session.rs:653, assembly/build_bench_loop.rs:117)가 make_stage(name, params)로 stage를 만들고 StageBackedPolicy로 감싸 기존 EvictionPolicy 자리에 꽂는다. plan을 반환하게 한 이유는 (1) in-place &mut KVCache로는 merge(D2O)·per-head를 표현할 수 없고, (2) .so C-ABI에서 "plugin이 엔진 버퍼를 직접 변형"은 안전하게 평탄화할 수 없기 때문이다 — PlanAbi/PlanArena는 plan(데이터)만 건너고, "각자 자기 것 free"(host는 복사 직후 plan_free(owner) 호출)로 cross-allocator UB를 차단한다.

5.2 계약 / 불변식

  • evict(cache, target_len) = 캐시를 target_len 토큰으로 축소. in-place 변형이며 호출 후 current_pos가 줄어 있어야 한다. RoPE 포지션은 계속 단조 증가(물리 위치만 감소) — 자세한 의미론은 07. KV 캐시.
  • plan_keep의 keep은 prefix 포함 ascending이어야 compact(keep, 0)가 in-place evict와 bit-identical하다. per-head H2O+처럼 단일 layer-wide keep으로 표현 불가한 정책은 default None으로 미지원을 표시한다.
  • 발화 게이트는 trait 밖에 있다. production 경로에서 실질 가드는 CacheManager::run_policy_eviction의 tokens_to_remove < MIN_EVICT_TOKENS(=64) skip이다 — 수십 토큰 제거를 위해 full sort+memcpy를 도는 "sticky eviction"을 막는다. 이 가드에 걸리면 NoOp이고 current_pos는 불변이다.
  • score 공급 여부에 따른 dispatch: EvictionHandler가 ctx.importance/head_importance 조합으로 ScoreContext::{None, Flat, PerHead}를 만들어 코어에 넘기고, 코어가 evict / evict_with_scores / evict_with_head_scores 중 하나를 고른다. H2O는 score 없이 불리면 sliding 유사 동작으로 폴백한다(테스트로 고정된 거동).

로딩 계약 (PR 리뷰 포인트):

  • .so당 dlopen 1회, 3축 엔트리는 부분집합으로 export 가능(번들). 어느 축에도 기여 0인 .so만 bail. wrong-type은 reject가 아니다.
  • 봉투 ABI 버전 게이트: register_*_v2()가 by-value 봉투(StageExportAbi 등)를 반환하고 host가 abi_version mismatch면 .so를 거부한다.
  • 2-pass 원자 등록: pass 1에서 이름 추출 + 빌트인 충돌(빌트인 우선, 동적 거부) + 봉투 내부 중복 검사, pass 2에서 write-lock 1회로 동적 중복 검사 + 일괄 push — .so 단위 all-or-nothing.
  • backend-cap의 category 다리(D7): 엔트리는 {name, category, vtable} 태그드 포인터이고, host가 category로 vtable을 카테고리별 테이블로 캐스팅한다. 현재 알려진 카테고리는 BACKEND_CAP_CATEGORY_ATTENTION = 1 하나 — 미지의 category는 로드 거부(새 카테고리 = host match arm 추가 = 재컴파일).
  • 해석 순서는 항상 정적 우선 → 동적 fallback. 호출부(make_stage, resolve_kivi_capability)는 source를 모른다. Arc<Library>를 registry가 영구 보관해 vtable/handle dangling을 막는다(leak-and-keep).
  • vtable fn-ptr은 panic 금지, i32 반환(0=OK, 음수=plugin의 깨끗한 논리 오류 → host는 로그 후 no-op 처리). GPU 자원은 borrow-for-call — plugin이 cl_mem을 retain하면 계약 위반.

6.4 plugin 저자 표면 — dual-wiring 매크로

// crates/techniques/<name>/src/lib.rs — 엔진 코어 수정 0
technique_api::register_kv_stage!("my_policy", |p: StageParams| Box::new(MyStage::new(p)));
technique_api::register_kv_format!("nf4",  || Box::new(Nf4));
technique_api::register_kivi_attention_plugin!("my_kivi", |args| Box::new(MyKivi::build(args)));
technique_api::export_plugin!();   // .so 당 1회 — 3축 엔트리 심볼 emit

각 매크로는 정적 경로(linkme 슬라이스 기여)와 동적 경로(plugin-cdylib feature 게이트 하의 C thunk + vtable 기여)를 한 번에 배선한다. 동적 경로를 feature로 게이트하는 이유는 정적 force-link 빌드에서 여러 plugin의 #[no_mangle] 엔트리 심볼이 충돌하는 것을 원천 차단하기 위해서다. 기여 static은 익명 const _: () = {} 스코프에 격리되어 한 crate에서 다회 호출(다수 stage/format)이 가능하다. 구체 절차는 10. 확장 How-to.

근거: crates/technique-api/src/lib.rs::register_kv_stage! (L657–727), ::register_kivi_attention_plugin! (L1381–1463), crates/techniques/ (caote, synth-q4-format, example-* 9개)

6.5 왜 이 모양인가

세 층(엔진 CapabilityRegistry / linkme 정적 / dlopen 동적)이 겹쳐 보이지만 각자 다른 질문에 답한다: CapabilityRegistry는 "이 프로세스의 이 backend가 이 능력을 가졌나"(인스턴스 매핑), linkme 슬라이스는 "이 바이너리에 어떤 기법이 링크되어 있나"(컴파일 타임 인벤토리), dlopen registry는 "런타임에 어떤 .so가 추가됐나"(재컴파일 없는 배포)다. KiviAttentionBackend가 셋을 관통하는 단일 trait(D8)인 것이 핵심 설계다: 엔진 정적 OpenCL 구현 · host dlopen 어댑터(DynKiviAttentionBackend) · plugin .so가 전부 같은 1벌을 구현하고, 시그니처를 &Tensor가 아닌 ABI struct(KiviAttnArgs의 raw cl_mem)로 잡아 plugin이 엔진 타입을 비참조한 채 컴파일된다 — &Tensor → cl_mem 다리는 소비자(kivi_format/kivi_cache)가 한 번만 수행해 static·dynamic이 공유한다.

근거: engine/src/capability/dynamic_backend_registry.rs module doc (L1–14), engine/src/capability/kivi_attention.rs (re-export shim — 동일 TypeId 보존), crates/technique-api/src/lib.rs D8 주석 (L1215–1220)
