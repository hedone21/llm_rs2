01. 시스템 개요

기준 커밋: 46ee4c2d · 작성일: 2026-06-10 · llm.rs Engine Guide

1. 무엇을 위한 코드베이스인가

llm.rs(저장소명 llm_rs2)는 Rust로 작성된 온디바이스 LLM 추론 프레임워크다. 1차 타겟은 ARM64 Android/엣지 디바이스이고, 연산 백엔드는 CPU(NEON, x86에서는 AVX2), OpenCL GPU(Adreno), CUDA(Jetson 임베디드 / PC)를 지원한다. OpenCL과 CUDA는 컴파일 타임에 상호 배타로 강제된다 — 둘을 동시에 켜면 compile_error!로 빌드가 실패한다. 

// engine/src/lib.rs:1
#[cfg(all(feature = "opencl", feature = "cuda"))]
compile_error!("Features 'opencl' and 'cuda' are mutually exclusive. Enable only one.");
#[cfg(all(feature = "opencl", feature = "cuda-embedded"))]
compile_error!("Features 'opencl' and 'cuda-embedded' are mutually exclusive. Enable only one.");

근거: engine/src/lib.rs::compile_error (L1~L4), engine/Cargo.toml [features] (opencl/cuda/cuda-embedded/htp_fastrpc)

이 프로젝트가 일반 추론 엔진과 구별되는 지점은 두 가지다.

  • Resilience(복원력) 내장 — 별도 프로세스인 llm_manager가 시스템 메모리·온도·전력·연산 부하를 감시하고, IPC로 엔진에 directive(throttle, KV eviction, weight swap, suspend 등)를 내려보낸다. 엔진의 추론 루프는 매 토큰마다 이 directive를 poll하여 런타임 중에 동작을 바꾼다. argus_cli에서 resilience는 default-on이며 --no-resilience로만 끈다 (args.enable_resilience = !args.no_resilience).
  • 확장 축의 분리 — KV/weight의 저장 표현(Format), 상주 데이터 관리 동작(Stage), 실행 위치(Hardware)를 직교 축으로 다루고, crates/technique-api를 통해 외부 crate가 엔진 코어 수정 없이 새 기법을 등록한다 (10장).

근거: engine/src/bin/argus_cli.rs::main (L27), crates/technique-api/src/lib.rs::KVCacheStage (trait, L220)

모델 입력은 GGUF / AUF(Argus Unified Format, GGUF 파생 self-contained 자산) / HuggingFace Safetensors 세 포맷을 받는다. 로더는 detect_primary_format()으로 경로에서 포맷을 판별해 TransformerModel::load_from_config(GGUF/AUF) 또는 load_with_dtype(Safetensors)로 분기한다. weight는 F16/Q4_0(블록당 18B, 32 elem) 등으로 양자화 지원된다.

근거: engine/src/session/init.rs::SessionInitCtx::build (L506~L544, detect_primary_format 분기), engine/src/quant (BlockQ4_0, QK4_0)

릴리스 프로필 — 워크스페이스 루트에서 lto = "fat", codegen-units = 1, panic = "abort", opt-level = 3으로 고정된다. fat-LTO 때문에 미사용 심볼이 잘려나갈 수 있어, 빌드 직후 내장 KV format 4종의 링크 여부를 ensure_builtin_kv_formats_registered()로 self-test하는 fail-fast 코드가 추론 셋업에 들어 있다.

근거: Cargo.toml [profile.release], engine/src/session/bin_setup.rs::build_inference_ctx (L40)

실측 교훈 (코드로 확인 불가한 운영 지식, CLAUDE.md 인용) — (1) Adreno 디바이스 권장 backend는 opencl --opencl-rpcmem이며, CL_MEM_ALLOC_HOST_PTR 기반 UnifiedBuffer는 Adreno에서 zero-copy 효과가 없다(진짜 zero-copy는 rpcmem DMA-BUF interop). (2) 성능 측정은 --profile 없이 한다 — per-op clFinish가 ~54 ms/tok의 sync 오버헤드를 더한다. 이 수치는 코드 주석에도 남아 있다(init.rs L158: "adds ~54 ms/tok"). (3) Galaxy S25 벤치는 6 스레드가 최적, 8 스레드는 역효과.

4. 시스템 컴포넌트 다이어그램

런타임에는 두 개의 Rust 프로세스(Engine, Manager)와 주변 도구가 IPC·파일로 엮인다. Engine 내부는 session(오케스트레이션) → models(forward) → backend(연산 dispatch) → memory(allocator)의 수직 스택과, 그 옆에 붙는 pressure(KV 캐시 + 관리 파이프라인), resilience(IPC 클라이언트)로 구성된다. 모듈별 책임과 계층 규칙은 03장, 추상화 카탈로그는 05장에서 다룬다.

5.1 셋업에서 결정되는 것들

SessionInitCtx::build가 backend 선택을 수행한다. --backend cpu면 CpuBackend가 primary가 되고 OpenCL이 가능하면 secondary로 자동 초기화된다(SwitchHw 대비, 실패는 soft-fail). --backend opencl이면 GPU가 primary이자 SwitchHw 왕복용 secondary를 겸하고, 이때만 CapabilityRegistry에 KiviAttentionBackend capability가 등록된다(CPU/CUDA는 빈 registry). backend·memory 4개의 Arc는 Hardware resolver 하나로 흡수되어 이후 코드가 hardware.resolve(DeviceTarget::Cpu)처럼 (backend, memory) 페어를 푼다.

근거: engine/src/session/init.rs::SessionInitCtx::build (L196~L424 backend match, L337~L338 caps.register, L825~L831 Hardware::new), engine/src/session/standard_happy.rs (L55~L59 hardware.resolve)

zero-copy 강제 활성 분기 — OpenCL primary일 때 --no-zero-copy를 줘도 --enable-resilience(즉 default-on), tensor partition, prefill CPU chunk 중 하나라도 활성이면 CPU-accessible 버퍼가 필요하므로 zero-copy가 강제로 다시 켜지고 경고만 출력된다. "분명히 껐는데 UnifiedBuffer가 잡힌다"는 리뷰 코멘트가 나오면 이 분기를 보라.

근거: engine/src/session/init.rs (L287~L301 effective_zero_copy 계산)

tokenizer는 --tokenizer-path 우선, GGUF면 {stem}.tokenizer.json → quant suffix 제거한 stem → sibling tokenizer.json 순서로 탐색한다. 로드 후 vocab 호환성 검사가 강제된다: tokenizer vocab이 model vocab을 8개(oob_tolerance) 초과해 넘으면 OOB embedding lookup 위험으로 bail!, model vocab이 tokenizer vocab을 max(model_vocab/20, 256) 초과해 넘어도 잘못된 tokenizer로 보고 bail!한다. sibling 모델의 tokenizer를 잘못 잡는 사고를 코드 레벨에서 차단하는 장치다.

근거: engine/src/session/bin_setup.rs::resolve_tokenizer_path (L203~L237), ::check_vocab_compatibility (L240~L277, oob_tolerance=8 / pad_tolerance 수식)

5.2 KV cache 할당 — typed vs opaque 분기

KV 캐시는 layer당 K/V 텐서 한 쌍씩, num_hidden_layers개가 할당된다. shape은 항상 [1, kv_heads, capacity, head_dim]의 HeadMajor 고정이고 (.with_layout(KVLayout::HeadMajor)), grow-on-demand dynamic 캐시 (KVCache::new_dynamic)로 만들어진다. 초기 capacity 수식: 

// bin_setup.rs L87~L96
initial_kv_capacity = (tokens.len() + num_tokens)
    .next_power_of_two()
    .max(128)
    .min(max_seq_len)   // --initial-kv-capacity > 0 이면 그 값.min(max_seq_len)

// 버퍼 크기 (layer당, K와 V 각각)
n_values = initial_kv_capacity * kv_heads * head_dim
Q4_0:  (n_values / QK4_0) * size_of::<BlockQ4_0>()   // 32 elem당 18B 블록
그 외: n_values * kv_type.size()                       // f32=4B, f16=2B

저장 표현 선택은 2단 dispatch다. --kv-format <이름>이 있으면 format 레지스트리 우선: 내장 typed 이름(f32/f16/q4_0/q8_0)이면 해당 DType으로, 아니면 make_format()(정적 linkme → 동적 .so 순)으로 해소한 뒤 descriptor가 내장 DType과 bit-equivalent면 typed fast path, novel descriptor면 opaque 경로(OpaqueBuffer = U8 inner + sidecar descriptor, 바이트 회계는 desc.bytes_for_elems())로 간다. --kv-format 미지정이면 --kv-type {f32|f16|q4} 하위호환 경로다. 상세는 06장·07장.

근거: engine/src/session/bin_setup.rs (L80~L175 dispatch, L281~L324 alloc_standard_kv_caches, L331~L380 alloc_opaque_kv_caches)

5.3 DecodeLoop — 토큰당 8단계 고정 골격

DecodeLoop는 추론 루프의 단일 소유자다. 6종의 trait object를 owned로 들고 (Forward, EvictionStage, SwapStage, CommandSource, TokenSampler, DecodeObserver 목록 — 외에 EngineReport, TokenTickSink), builder는 typestate(NoForward → HasForward)로 Forward 미주입 상태의 .build()를 컴파일 타임에 막는다. 매 step의 순서는 코드 주석 그대로 (a) command poll → (a.5) resilience eviction → (a.6) offload/recall → (b) eviction stage → (c) swap before → (d) forward(seq_len=1) → (e) swap after → (f) sample → (g) observers → (h) TBT pacing이다.

주의할 동작 둘: ① resilience evict directive는 sticky 1회 발동이다 — evict_applied 플래그로 같은 directive의 매 poll 재적용(캐시 과다 축소)을 막고, RestoreDefaults로 plan.evict가 None이 되면 reset된다. ② eviction으로 KV가 줄면 pos를 새 위치로 되감고 forward.on_kv_prune(new_pos)를 호출한다 — "eviction은 caller(루프) 책임"이라는 이 코드베이스의 설계가 루프 골격에 그대로 박혀 있다.

근거: engine/src/session/decode_loop.rs::DecodeLoop (struct, L30~L53), ::run (L124~L346), NoForward/HasForward (L24~L27), evict_applied (L46~L48, L168~L194)

첫 토큰 샘플링은 루프 밖에서 일어난다: prefill()이 마지막 logits를 반환하면 caller가 sampling::sample(&mut logits, &tokens, ...)로 first_token을 뽑고 run(budget = num_tokens - 1, first_token)에 넘긴다. prompt 전체가 repetition penalty history로 sampler에 통보된다(observe_token 루프). 끝나면 TTFT(=prefill 시간), Decode: X ms/tok, Avg TBT가 stdout에 찍힌다 — 성능 리포트에서 봐야 할 정규 지표 라인이다.

근거: engine/src/session/standard_happy.rs (L82~L131), engine/src/session/decode_loop.rs::prefill (L79~L105)

8. 이 챕터에서 기억할 것

  • 프로세스는 둘: Engine(argus_cli/argus_bench)과 Manager(llm_manager). 결합은 llm_shared wire 타입 + IPC뿐.
  • 추론의 단일 소유자는 DecodeLoop — 토큰당 (a)~(h) 8단계 고정 골격, eviction/swap/command는 전부 forward 사이에 끼어든다.
  • KV 캐시는 layer당 K/V 한 쌍, HeadMajor [1, kv_heads, capacity, head_dim], grow-on-demand. 표현은 --kv-format(registry) → --kv-type(하위호환) 2단 dispatch.
  • 확장은 technique-api 단방향 의존 + linkme/dlopen 등록 — 엔진 코어 수정 없이 stage/format을 추가한다.
  • directive는 push가 아니라 토큰 단위 pull(poll)이다 — 적용 지연의 단위가 "토큰"임을 리뷰 때 기억할 것.
