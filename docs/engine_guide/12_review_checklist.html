12. 컨벤션 & 리뷰 체크리스트

기준 커밋: 46ee4c2d (+ working tree) · 작성일: 2026-06-10 · llm.rs Engine Guide

12.0 이 장의 목적 — 컴파일러가 못 잡는 버그를 리뷰어가 잡는다

이 코드베이스에서 실제로 터진 버그들의 공통점은 타입 시스템이 보호하지 못하는 암묵 계약 위반이라는 것이다. copy_slice의 count 파라미터는 그냥 usize이고, element 단위인지 block 단위인지는 dtype에 따라 달라진다. KV 캐시의 메모리 배치(HeadMajor)는 enum 필드일 뿐이고, 이를 전제하는 코드는 16개 이상의 파일에 흩어져 있다. RoPE 회전에 쓰는 논리 위치와 캐시 버퍼의 물리 위치는 둘 다 usize pos로 표기된다. 이 셋 중 어느 하나라도 혼동하면 컴파일은 통과하고, 결과는 SIGSEGV(운이 좋은 경우) 또는 silent garbage(운이 나쁜 경우)다.

이 장은 (1) 프로젝트 코딩 컨벤션, (2) 과거 버그에서 도출한 4대 불변식, (3) 그것을 PR에서 확인하는 체크리스트 표로 구성된다. 구현 절차(빌드·테스트·배포)는 11장에 있다.

12.1 컨벤션

12.1.1 모듈 파일 스타일 — no-mod.rs

디렉토리 모듈의 루트는 foo/mod.rs가 아니라 형제 파일 foo.rs다. 즉 foo.rs(모듈 선언·re-export)와 foo/(서브모듈 파일들)가 나란히 존재한다. engine/src/는 이 스타일로 전면 정리가 끝난 상태다: 

engine/src/
├── backend.rs      ← 모듈 루트 (Backend trait + copy_slice/buffer_shift 기본 구현)
├── backend/        ← cpu/, opencl.rs, opencl/, cuda_pc.rs, htp_fastrpc.rs ...
├── pressure.rs     ← 모듈 루트 (CachePressureHandler, ActionResult, PressureLevel)
├── pressure/       ← kv_cache.rs, cache_manager.rs, eviction/, kivi_cache.rs ...
├── session.rs / session/
├── format.rs / format/
└── ... (전부 동일 패턴)

현재 engine/src/와 crates/techniques/에는 mod.rs가 0개다. 유일한 잔존은 crates/qnn_oppkg/src/{ops,graph}/mod.rs 2개 — paper microbench용 보존 crate라 sweep 대상에서 제외된 것이다. 리뷰 포인트: 신규·이동 모듈이 mod.rs를 새로 만들면 reject.

근거: find engine/src crates -name mod.rs 결과 = crates/qnn_oppkg/src/ops/mod.rs, crates/qnn_oppkg/src/graph/mod.rs 2건뿐. engine/src/ 디렉토리 목록에서 backend.rs+backend/ 등 형제 패턴 확인.

12.2 불변식 #1 — 단위 회계: element / block / byte

KV 캐시는 한 토큰을 양자화 형태로 저장할 수 있다(도메인 환기: Q4_0은 float 32개를 18바이트 블록 하나로 압축하는 포맷이다). 문제는 버퍼 복사 API의 offset/count 단위가 dtype에 따라 의미가 바뀐다는 것이다. Backend::copy_slice/buffer_shift의 기본 구현은 내부에서 type_size를 곱해 바이트 주소를 계산한다: 

// engine/src/backend.rs — Backend trait 기본 구현
fn copy_slice(&self, src: &Tensor, dst: &mut Tensor,
              src_offset: usize, dst_offset: usize, count: usize) -> Result<()> {
    let type_size = match src.dtype() {
        DType::F32 => 4,
        DType::F16 => 2,
        DType::U8  => 1,
        DType::Q4_0 => std::mem::size_of::<crate::quant::BlockQ4_0>(), // = 18
        _ => 1,
    };
    // byte_offset = offset * type_size, byte_count = count * type_size
    ...
}

즉 count/offset의 단위는 type_size가 가리키는 "1개"다. Q4_0에서 1개 = BlockQ4_0(= f16 d + [u8; 16] qs = 18B) = float 32개분(QK4_0 = 32)이다. 따라서 같은 토큰 수를 복사하더라도 dtype별로 넘겨야 하는 count가 다르다:

dtype회계 단위pos p개 × head H × dim D 복사 시 counttype_size
F32 / F16elementp * H * D4 / 2 B
Q4_0 (typed)blockp * H * D / QK4_0 (QK4_0=32)18 B (BlockQ4_0)
opaque (.so 동적 format)bytedescriptor-keyed: bytes_for_elems(n)1 B (U8)

근거: engine/src/backend.rs::copy_slice (L1096, type_size match), engine/src/quant.rs::QK4_0 (L9, =32) / BlockQ4_0 (L14, f16 + [u8;16] = 18B), engine/src/pressure/kv_cache.rs::grow (L240, opaque arm은 opaque_desc().bytes_for_elems L235).

12.2.1 실제 사고: 00918e9e — shrink_to_fit의 block 변환 누락 SIGSEGV

2026-06-09에 수정된 실제 버그다. KVCache는 grow-on-demand 구조라 용량을 키우는 grow()와 eviction 후 줄이는 shrink_to_fit()이 대칭으로 존재하고, 둘 다 "새 버퍼 할당 → copy_slice로 기존 데이터 이주" 패턴이다. grow()는 Q4_0일 때 element→block 변환(/ QK4_0)을 SeqMajor·HeadMajor 양쪽에 했지만, shrink_to_fit()은 element count를 그대로 넘겼다. 결과: 실제 버퍼의 약 32배(QK4_0배)를 읽는 OOB read → SIGSEGV.

왜 f16/f32와 opaque는 무사하고 Q4_0만 터졌나. f16/f32는 element 단위가 곧 정답이고, opaque format은 처음부터 U8 byte 회계(opaque_bytes_per_head())를 쓴다. 즉 이 함정은 typed Q4_0 한 좌표에서만 발현한다. dtype 하나만 테스트하고 "복사 로직은 dtype 무관하니까 됐다"고 판단하면 정확히 이 구멍에 빠진다.
왜 기존 테스트가 못 잡았나 — dynamic 여부가 두 번째 축이다. 기존 회귀 테스트 test_release_unused_pages_noop_on_q4_dtype은 KVCache::new()(non-dynamic, memory=None)로 만든 캐시였다. shrink_to_fit()은 memory가 None이면 즉시 Ok(0) 반환하므로 버그 경로에 아예 진입하지 못했다. 수정 커밋이 추가한 회귀 테스트 2종은 new_dynamic() + Galloc으로 shrink를 실제 발동시키고, block 첫 바이트에 marker를 심어 no-OOB + shrink 후 데이터 보존까지 검증한다. 교훈: KV 캐시 버퍼 조작 테스트는 dtype(f16/q4_0/opaque) × dynamic 여부 × layout(SeqMajor/HeadMajor) 매트릭스에서 버그가 발현하는 좌표를 명시적으로 찍어야 한다.

12.3 불변식 #2 — HeadMajor 레이아웃 전제

KVLayout은 두 variant가 있고 offset 수식이 다르다 (구조 상세는 7장): 

// engine/src/pressure/kv_cache.rs::offset (L183)
match self.layout {
    KVLayout::SeqMajor  => pos * self.kv_heads * self.head_dim + head * self.head_dim,
    KVLayout::HeadMajor => head * self.capacity * self.head_dim + pos * self.head_dim,
}

production은 HeadMajor 고정이다. 그런데 함정이 둘 있다. 첫째, KVCache::new()/new_dynamic()의 디폴트는 SeqMajor다(L79/L105) — 생성 후 .with_layout(KVLayout::HeadMajor)를 빌더로 호출해야 한다. 표준 할당 경로 session/bin_setup.rs::alloc_standard_kv_caches는 이를 수행하지만, 새 할당 경로를 추가하는 PR이 이 호출을 빠뜨리면 SeqMajor 캐시가 HeadMajor 전제 코드로 흘러든다. 둘째, layout을 잘못 섞어도 에러가 나지 않는 경로가 있다 — OpenCL Plan 경로는 layout을 enum으로 받지 않고 stride 값에서 역추론한다:

12.4 불변식 #3 — RoPE 포지션은 단조 증가, 물리 위치는 감소 가능

도메인 환기: RoPE는 K/Q 벡터에 위치 의존 회전을 적용하는 위치 인코딩이고, eviction은 KV 캐시에서 토큰을 영구 삭제하는 동작이다. 이 코드베이스에서 "위치"는 두 종류이며 둘 다 usize라 타입으로 구분되지 않는다:

  • 논리 위치 (RoPE 위치) — 토큰이 시퀀스에서 차지하는 절대 위치. forward의 start_pos로 전달되어 backend.rope_inplace(&mut q_rope, start_pos, rope_theta)의 회전각을 결정한다. eviction과 무관하게 단조 증가해야 한다.
  • 물리 위치 (current_pos) — KV 버퍼에서 다음 쓰기가 일어나는 슬롯. eviction의 prune_prefix()/shift_positions() compaction으로 감소할 수 있다.

두 위치가 분리되는 핵심 메커니즘: KVCache::update()는 항상 current_pos에 쓰고 start_pos를 모른다. start_pos는 오직 RoPE 회전에만 쓰인다. 그리고 eviction의 shift_positions()는 바이트를 옮길 뿐 재회전하지 않으므로, 캐시에 남은 K 벡터는 원래 위치의 RoPE 회전을 그대로 보존한다. 이 두 사실에서 불변식이 따라 나온다 — eviction 후 다음 토큰의 RoPE 위치를 compacted current_pos로 되돌리면, 캐시된 토큰들이 새 토큰보다 "미래"에 있는 것처럼 보이는 회전 불연속이 생긴다. eval_loop.rs에 이 불변식이 주석으로 명문화되어 있다: 

// engine/src/session/eval/eval_loop.rs L295-299 (원문)

// The next decode token's RoPE must continue from prompt_len (original position)

    Note over P: 다음 토큰 쓰기 슬롯 = 300 (감소 정상)
리뷰어 주의 — production 경로의 회색지대. eval_loop.rs는 위 불변식을 지키지만, session/decode_loop.rs는 eviction 후 self.pos = new_pos(compacted 물리 위치)로 동기화하고(L176, L210, L245), step()은 그 ctx.pos를 forward_into의 start_pos — 즉 RoPE 위치 — 로 그대로 전달한다(model_forward.rs L453). chat/session.rs의 on_turn_end(L297)도 동일하다. 이는 eval_loop 주석이 금지하는 바로 그 패턴으로 보인다 — 의도된 단순화인지 잠재 품질 버그인지 코드만으로는 확정할 수 없다(이 장 말미 open question). eviction·pos 동기화를 건드리는 PR에서는 "이 pos가 RoPE용인지 캐시 슬롯용인지"를 반드시 질문하라.

부수 불변식: eviction이 KV position을 shift하면 보유 중인 GPU kernel plan이 stale offset을 갖는다. ModelForward::on_kv_prune()은 이 때문에 gpu_plan = None으로 invalidate하여 다음 step에서 lazy rebuild시킨다. 새 eviction/offload 경로를 추가하면서 on_kv_prune() 통지를 빠뜨리면 다음 step부터 silent garbage다.

12.5 불변식 #4 — tokenizer는 명시하고, vocab 정합성은 검증된다

모델 파일과 tokenizer가 따로 노는 GGUF 운용에서, tokenizer 자동 탐색이 sibling 모델의 tokenizer를 잘못 잡으면 모델은 OOB 없이 잘못된 vocab으로 decode를 진행해 garbage를 출력하면서 silent하게 통과한다. 실제로 fast path 검증 중 production과 fast path가 동일한 garbage를 내며 byte-equal 비교를 통과한 사고가 있었고(commit 45cb4892), 그 가드로 check_vocab_compatibility가 추가됐다. 현재 구현(session/bin_setup.rs):

  • resolution 순서 (resolve_tokenizer_path, L203): ① --tokenizer-path 명시값 → ② GGUF면 sibling 탐색: {stem}.tokenizer.json → quant suffix(-f16/-q4_0 등) 제거한 {stem'}.tokenizer.json → 디렉토리의 tokenizer.json → ③ safetensors면 {dir}/tokenizer.json.
  • vocab 검증 (check_vocab_compatibility, L240): tok_vocab > model_vocab + 8이면 OOB embedding lookup 위험으로 즉시 error; 8 이내 초과는 warning(multimodal special token 허용). 반대로 model_vocab > tok_vocab + max(model_vocab/20, 256)이면 wrong-tokenizer로 error — Qwen2.5-1.5B의 trained=151665 / padded=151936 같은 정상 padding은 통과시키는 임계값이다.
실측 교훈 (CLAUDE.md): Adreno 디바이스 테스트에서는 --tokenizer-path 명시가 필수 운용 수칙이다. fallback은 마지막 안전망이지 의존 대상이 아니다 — fallback이 잡은 tokenizer가 같은 family면 vocab 검증도 통과할 수 있다. 디바이스 테스트 스크립트·문서를 추가하는 PR에서 tokenizer 경로가 명시돼 있는지 확인하라.

근거: engine/src/session/bin_setup.rs::resolve_tokenizer_path (L203), ::check_vocab_compatibility (L240, oob_tolerance=8 L247, pad_tolerance=max(vocab/20,256) L265), git show 45cb4892 (사고 경위 — D-D.6 fast path 검증 중 동일 garbage byte-equal 통과).

12.6 측정·커널 관련 실측 교훈 (요약)

다음은 코드만으로 확인 불가한 디바이스 실측 교훈(CLAUDE.md)으로, 성능 PR 리뷰 시 근거 데이터의 유효성을 판정하는 기준이다. 상세는 9장(백엔드)·11장(워크플로우)에 위임한다.

  • 성능 수치는 --profile 없이 측정된 것만 절대값으로 인정. --profile은 plan 경로를 비활성화하고 매 op마다 synchronize() 2회를 끼워 토큰당 약 54 ms의 sync 오버헤드를 더한다. per-op breakdown은 상대 비교에만 유효.
  • .cl 커널 수정 PR의 Adreno 제약: DK=128 flash attention은 per-thread 32 float4 초과 시 register spill; sub_group_reduce_*는 SLM tree-reduce보다 33~55% 느림(Adreno 830 실측); 엔진 간 비교는 항상 wall-clock(CL_QUEUE_PROFILING_ENABLE은 driver-specific 패널티).
  • zero-copy 주장 검증: Adreno에서 CL_MEM_ALLOC_HOST_PTR(UnifiedBuffer)는 zero-copy 효과 없음 — 진짜 원천은 rpcmem DMA-BUF + CL_MEM_USE_HOST_PTR alias. ARM UMA 캐시 비일관성으로 read_buffer/write_buffer 경유 필수(as_ptr 직접 접근은 stale cache). 상세는 6장.
