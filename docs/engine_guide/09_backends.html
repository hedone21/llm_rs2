1.1 cpu_companion — GPU backend의 호스트 동반자

cpu_companion()은 trait에서 default 구현이 없는 유일한 capability 메서드다. &Self → &dyn Backend coercion이 Self: Sized bound 없이는 불가능하고, 그 bound를 붙이면 &dyn Backend 디스패치가 막히기 때문에 각 구현체가 한 줄 override를 직접 쓴다. CPU backend는 self를, GPU backend는 생성 시 주입받은 cpu_singleton()(process-wide OnceLock 단일 인스턴스)을 반환한다. 이 동반자는 OpenCL plan.rs의 tensor partition fast path가 NEON fused 커널에 접근하는 통로다 (§3.3).

근거: engine/src/backend.rs::Backend::cpu_companion (L1197 + 주석 L1186~1196), engine/src/backend/cpu.rs::cpu_singleton (L30), engine/src/backend/opencl.rs L550~552·L1590

2.1 get_extension — cold-path string-keyed downcast

표준 vtable 밖의 backend-specific 진입점은 ggml의 get_proc_address를 차용한 get_extension(name)으로 통일했다. 자유 문자열은 silent-None을 부르므로 모듈 const만 키로 쓴다: EXT_OPENCL_QUEUE("opencl_queue"), EXT_OPENCL_SECONDARY, EXT_RPCMEM_ALLOCATOR("rpcmem_allocator"). forward/decode loop 안 호출 금지 — Any downcast + lookup branch가 매 layer 누적된다. 반면 hot-path용 hook(as_kivi_attention, gpu_score_acc)은 전용 trait 메서드로 분리되어 있고, 소비자는 construction 시점에 handle을 1회 pull해 보관한다(CapabilityRegistry, §5.1).

근거: engine/src/backend.rs EXT_* const (L100~116) + get_extension (L1319), engine/src/capability.rs::CapabilityRegistry (L40)

3. CPU backend — NEON 디스패치 구조

3.1 타입 alias와 3단 위임

backend/cpu.rs는 컴파일 타겟별로 CpuBackend를 결정한다: aarch64 → CpuBackendNeon, x86_64 → CpuBackendAVX2, 그 외 → CpuBackendCommon. NEON 구현은 자신이 최적화한 op만 직접 처리하고 나머지(matmul, softmax, rope_inplace, cast 등)는 CpuBackendCommon::new()로 위임한다. 즉 Common이 모든 op의 scalar 정답 구현이고, NEON/AVX2는 그 위의 선택적 가속층이다.

// neon.rs — dtype 기반 1차 디스패치
fn matmul_transposed(&self, a: &Tensor, b: &Tensor, out: &mut Tensor) -> Result<()> {
    match b.dtype() {
        DType::F32  => self.matmul_transposed_f32(a, b, out),
        DType::F16  => self.matmul_transposed_f16(a, b, out),
        DType::Q4_0 => self.matmul_transposed_q4_0(a, b, out),
        _ => CpuBackendCommon::new().matmul_transposed(a, b, out),
    }
}

근거: engine/src/backend/cpu/neon.rs::CpuBackendNeon::matmul_transposed (L67~74), engine/src/backend/cpu.rs (L14~21 alias)

3.2 Q4_0 경로 — 런타임 ISA 감지와 Q8 워크스페이스

Q4_0 matmul은 활성화를 먼저 Q8_0으로 양자화한 뒤 정수 dot을 돈다. 핵심 분기:

  • ISA 선택은 런타임: std::arch::is_aarch64_feature_detected!("dotprod")로 vec_dot_q4_0_q8_0_sdot(#[target_feature(enable = "neon,dotprod")]) vs 비-dotprod fallback vec_dot_q4_0_q8_0을 고른다. i8mm 변형(vec_dot_q4_0_q8_0_i8mm)과 4-row 묶음 변형(_sdot_4rows)도 있다.
  • Q8 양자화 버퍼는 thread_local: Q8_WORKSPACE: RefCell<Vec<BlockQ8_0>> — 매 matmul마다의 Vec 할당(~112 allocs/token)을 제거. 메인 스레드가 채운 뒤 병렬 구간에서는 raw 포인터(SendSyncPtr)로 읽기만 한다.
  • 병렬화 토글: USE_RAYON: AtomicBool — F16 matmul을 SpinPool(자체 thread_pool, park/unpark ~1-3µs) 대신 Rayon으로 돌리는 A/B 벤치 스위치(--use-rayon).

근거: engine/src/backend/cpu/neon.rs L17 (USE_RAYON), L23~25 (Q8_WORKSPACE), L2305 (dotprod 감지), L2774~2775 (vec_dot_q4_0_q8_0_sdot), L3062 (i8mm)

3.3 CpuKernelSet — 함수 포인터 테이블로 노출되는 fused 커널

decode(M=1)에서 같은 활성화 x로 gate/up/down 같은 여러 weight를 곱할 때, freestanding unsafe fn 두 개가 변환·디스패치 비용을 1회로 합친다:

  • fused_matmul_f16(a: *const f32, k, matmuls: &[(*const u16, *mut f32, usize)]) — A를 F32→F16으로 1회 변환 후 최대 3개 matmul을 단일 SpinPool 디스패치로 처리. 청크 크기는 total_rows / (n_threads * 8)을 NR=4 배수로 올림.
  • fused_matmul_q4_0(...) — 활성화 Q8_0 양자화를 1회만 수행(위 thread_local 워크스페이스 재사용).

이 둘은 cpu_kernels.rs의 static CPU_KERNEL_SET: CpuKernelSet에 함수 포인터로 묶이고, CpuBackendNeon::cpu_kernels()만 Some(&CPU_KERNEL_SET)을 반환한다(GPU backend·AVX2·Common은 None). 소비자는 OpenCL plan.rs의 GPU↔CPU tensor partition fast path로, backend.cpu_companion().cpu_kernels() 체인으로 도달한다 — L2 backend가 L3를 downcast하던 패턴의 대체다. 비-aarch64에서는 CpuKernelSet이 필드 없는 placeholder로만 존재한다.

근거: engine/src/cpu_kernels.rs::CPU_KERNEL_SET (L45), engine/src/backend/cpu/neon.rs::fused_matmul_f16 (L3523) / fused_matmul_q4_0 (L3592), engine/src/backend/opencl/plan.rs L698~724 (소비자)

3.4 attention_gen — F16 KV flash decoding

NEON attention_gen은 KV dtype으로 분기한다: F16이면 전용 attention_gen_f16_neon(F16·F32 직접 dot + 벡터화 softmax), F32/Q4_0이면 Common의 naive 구현. F16 경로 안에서 다시 cache_seq_len >= FLASH_DECODE_THRESHOLD(1024)이면 chunk 단위 online-softmax flash decoding으로 간다. FLASH_DECODE_CHUNK_SIZE = 1024는 head_dim=128 기준 K+V 한 chunk가 512 KiB(Snapdragon 8 Elite L2 2 MiB의 ~25%)가 되도록 잡은 값이다.

근거: engine/src/backend/cpu/neon.rs::attention_gen (L186~224), FLASH_DECODE_CHUNK_SIZE/FLASH_DECODE_THRESHOLD (L248·L257), attention_gen_f16_neon_flash (L796)

4. OpenCL backend

4.1 커널 소스 임베드 + 런타임 컴파일 + 2단 fallback

engine/kernels/에는 약 80개의 .cl 파일이 있다. 로딩 방식의 정확한 사실: 소스 텍스트는 include_str!("../../kernels/*.cl")로 바이너리에 컴파일 타임 임베드되고, OpenCL 드라이버 컴파일(Program::builder().src(...).build(&context))이 런타임(OpenCLBackend::new 시점)에 일어난다. "런타임 로드"는 파일시스템에서 읽는다는 뜻이 아니라 드라이버 JIT 빌드를 가리킨다. 각 program은 3단 폴백을 탄다:

  1. 원본 (subgroup 사용 커널) →
  2. kernels/fallback/*_nosub.cl (subgroup 미지원 디바이스용) →
  3. 빈 dummy 커널 ("__kernel void kernel_mul_mat_f32_f32() {}") — 초기화는 살리되 해당 경로 dispatch는 사실상 죽는다.

컴파일 옵션은 build_cl_opts(device): 디바이스 CL C 버전이 2.0 이상이면 -cl-std=CL2.0 + 항상 -cl-mad-enable -cl-unsafe-math-optimizations -cl-finite-math-only -cl-fast-relaxed-math. 플랫폼/디바이스 선택은 OCL_PLATFORM/OCL_DEVICE_TYPE env로 제어한다. 커널 소스 자체도 cl_intel_subgroups/cl_qcom_reqd_sub_group_size extension 매크로로 Intel(SIMD 16) vs Adreno(wave 64/128) 분기를 내장한다.

근거: engine/src/backend/opencl.rs::new_with_profile_events (L650~), include_str! 체인 (L785~), build_cl_opts (L204), CL_FAST_MATH_FLAGS (L198), engine/kernels/mul_mv_q4_0_f32.cl 상단 extension 매크로

빌드된 커널 객체는 KernelCache struct에 CoreKernel로 모인다(50여 필드). 선택적 커널은 Option<CoreKernel> — flash attention은 DK/DV가 컴파일 타임 define이라 head_dim별 변형이 따로 존재한다: kernel_flash_attn_f32_f16_dk64/_dk128(prefill), _q1_dk64/_q1_dk128(decode), _f32_dk256(Gemma3). 접근은 UnsafeCell + "single-threaded 추론" 가정의 unsafe { &*self.kernels.get() } 패턴이다.

근거: engine/src/backend/opencl.rs::KernelCache (L229~), OpenCLBackend struct (L362~)

4.2 matmul 디스패치 — dtype × M × SOA registry

trait 진입점 matmul_transposed는 weight dtype으로 matmul_q4_0 / matmul_f16 / matmul(F32)에 분기하고, 각 함수 안에서 토큰 수 M과 가용 커널에 따라 다시 갈라진다:

flowchart TD
    A["matmul_transposed(a, b, out)"] --> B{"b.dtype()"}
    B -->|Q4_0| Q["matmul_q4_0"]
    B -->|F16| F["matmul_f16"]
    B -->|F32| G["matmul (GEMV f32)"]
    Q --> Q1{"m == 1 그리고 SOA registry hit + q_img 존재?"}
    Q1 -->|yes| QN["matmul_q4_0_noshuffle: image1d GEMV (Adreno TP cache)"]
    Q1 -->|no| Q2{"m > 1, n%4==0, k%4==0, SOA hit?"}
    Q2 -->|yes| QG["matmul_q4_0_gemm_adreno: mul_mat_Ab_Bi_8x4"]
    Q2 -->|no| Q3{"m >= 32 + GEMM 커널 존재?"}
    Q3 -->|yes| QM["matmul_gemm_q4_0: tiled GEMM BM=64"]
    Q3 -->|no| QV["generic GEMV: kernel_mul_mat_q4_0_f32"]
    F --> F1{"m > 8 + GEMM 커널?"}
    F1 -->|yes| FM["matmul_gemm_f16"]
    F1 -->|no| F2{"m == 1 + 1row 커널?"}
    F2 -->|yes| FR["1row GEMV (N_DST=1, 64 threads/WG)"]
    F2 -->|no| F3{"n > 4096 + L4 커널?"}
    F3 -->|yes| FL["L4 GEMV (N_DST=4)"]
    F3 -->|no| FV["기본 4-wave GEMV"]

Q4_0의 핵심 자료구조는 noshuffle SOA registry(noshuffle_soa_registry: UnsafeCell<HashMap<usize, NoshuffleSoaEntry>>)다. 키는 원본 weight의 cl_mem 포인터 주소(usize). 로드 타임에 prepare_noshuffle_buffers() 또는 AUF pre-converted 경로(alloc_pre_converted_soa_tensor)가 Q4_0 weight를 nibble-unshuffle + 2D transpose된 SOA(q_buf + d_buf + optional image1d_buffer_t q_img)로 변환·등록하면, matmul_q4_0이 lookup hit 시 자동으로 noshuffle GEMV/GEMM으로 디스패치한다. 이 registry가 cl_mem 주소를 키로 쓰기 때문에 weight swap 후에는 stale 키가 남는다 — SwapExecutor는 batch당 1회 invalidate_noshuffle_soa_registry() 호출 후 재등록(ensure_noshuffle_soa_registered / restore_pre_converted_soa_registration)이 계약이다. 누락 시 plan rebuild가 lookup miss로 AOS GEMV에 떨어지거나 silent garbage가 난다.

근거: engine/src/backend/opencl.rs::matmul_q4_0 (L2262~2365 분기), noshuffle_soa_registry (L453~457), matmul_f16 (L2124~2197), engine/src/backend.rs::invalidate_noshuffle_soa_registry doc (L765~)

4.3 attention_gen — flash decode fast path와 legacy 경로

OpenCL attention_gen의 분기 조건: GPU score accumulator가 활성이거나 scores_out이 None이면 flash_attention_decode_gpu(Q1 커널, single-pass online softmax, float4 K/V read)를 시도하고, 성공하면 종료. CPU readback용 scores_out이 필요한데 GPU acc가 비활성인 경우에만 legacy kernel_attn_gen_half(F16 KV) / kernel_attn_gen(F32) + CPU readback으로 떨어진다. KV layout은 shape에서 추론한다 — k_shape[1] == num_heads_kv && k_shape[1] != k_shape[2]이면 HeadMajor로 보고 kv_pos_stride = head_dim, kv_head_stride = capacity * head_dim을 커널 인자로 넘긴다(07장의 stride-agnostic 커널 계약).

근거: engine/src/backend/opencl.rs::attention_gen (L6000~6101), flash_attention_decode_gpu (L3468)

4.4 enqueue_kernel_labeled — 모든 dispatch의 단일 깔때기

모든 커널 enqueue는 enqueue_kernel_labeled(kernel, default_label, work_dim, gws, lws)를 지난다. profile_events_enabled가 꺼져 있으면 event 없이 그냥 enqueue(오버헤드 0), 켜져 있으면 ocl::core::Event를 라벨과 함께 수집하고 decode loop가 주기적으로 flush_and_aggregate_profile()에서 CL_PROFILING_COMMAND_END - START ns를 라벨별 누적한다. 라벨은 set_op_label hint가 있으면 그것을(같은 GEMV 커널이라도 matmul_qkv/matmul_ffn/lm_head 구분), 없으면 default를 쓴다. 이는 매 op마다 clFinish() 2회를 박아 Adreno decode를 ~54 ms/tok 부풀리던 legacy --profile의 대체다.

근거: engine/src/backend/opencl.rs::enqueue_kernel_labeled (L1725), flush_and_aggregate_profile (L1777), new_with_profile_events doc (L637~649)

4.5 rpcmem 통합 — Adreno 진짜 zero-copy

--opencl-rpcmem이 켜지면 OpenCLBackend::new_with_options(profile_events, opencl_rpcmem=true)가 eager하게 RpcmemAllocator::new()를 시도한다: libcdsprpc.so(기본 경로 /vendor/lib64/libcdsprpc.so)를 dlopen하고 심볼 3개(rpcmem_alloc/rpcmem_free/rpcmem_to_fd)만 lookup한다. 실패하면 stderr 경고 1회 후 조용히 강등되어 표준 UnifiedBuffer 경로로 동작한다(호스트 빌드 보호). heap id는 RPCMEM_HEAP_ID_SYSTEM = 25.

할당 경로의 핵심 분기는 OpenCLMemory(backend/opencl/memory.rs)에 있다:

  • alloc_kv() — rpcmem_allocator가 Some이면 rpcmem heap 할당 + CL_MEM_USE_HOST_PTR alias cl_mem 생성(RpcmemKvBuffer). alias 생성 실패 시 rpcmem을 롤백 free하고 버퍼 단위로 UnifiedBuffer에 폴백한다(INV-RPCMEM-003).
  • alloc() (activation) — rpcmem을 절대 쓰지 않는다(INV-RPCMEM-007). zero-copy 모드면 UnifiedBuffer(CL_MEM_ALLOC_HOST_PTR), 아니면 device-only OpenCLBuffer.
  • weight swap 경로 — Backend::alloc_alias_weight_buffer(LISWAP-6)가 RpcmemSecondaryStore의 per-layer rpcmem region 위에 CL_MEM_READ_ONLY | CL_MEM_USE_HOST_PTR alias(RpcmemAliasBuffer)를 만들어 swap 시 clEnqueueWriteBuffer H2D 복사를 제거한다. lifetime은 guard 2종(Weak<dyn MmapKeepAlive> + Arc<dyn RpcmemRegionGuard>)이 버퍼 안에 보관되어 보장된다.

같은 Arc<RpcmemAllocator> 인스턴스는 get_extension(EXT_RPCMEM_ALLOCATOR)로도 노출되어 secondary loader가 공유한다.

근거: engine/src/backend/opencl.rs::new_with_options (L606~626), engine/src/memory/rpcmem/allocator.rs (L43 RPCMEM_HEAP_ID_SYSTEM, L102 기본 so 경로), engine/src/backend/opencl/memory.rs::alloc_kv (L166~218) / alloc_kv_rpcmem (L65~104), engine/src/memory/rpcmem/opencl_alias.rs::RpcmemAliasBuffer

실측 교훈 (CLAUDE.md, 코드로 확인 불가한 디바이스 지식): CL_MEM_ALLOC_HOST_PTR(UnifiedBuffer / --zero-copy)는 Adreno에서 zero-copy 효과가 없다 — 드라이버 internal pool alloc으로 처리된다. 진짜 zero-copy 원천은 rpcmem DMA-BUF heap + CL_MEM_USE_HOST_PTR alias다. 또한 ARM UMA의 캐시 비일관성 때문에 매핑 포인터 직접 접근(stale cache)이 아니라 read_buffer/write_buffer 경유가 필수다. S25 실측에서 decode TBT는 rpcmem on/off 모두 ≈32 ms/tok로 동급이었다(Qwen2.5-1.5B Q4_0, 2026-05-26).

5. capability plugin dlopen 경로 — Backend 축(3번째 axis)

plugin 시스템은 stage·format·backend-cap 3축이 같은 골격(linkme 정적 + dlopen 동적, 축별 분리 registry)을 공유한다. 이 절은 backend-cap 축, 즉 특정 backend에만 있는 선택적 능력(현재 카테고리 1개: KIVI fused attention)을 외부 .so가 제공하는 경로를 다룬다. stage/format 축은 10장 참조.

5.1 canonical trait은 technique-api가 소유한다

KiviAttentionBackend trait의 정의 본체는 엔진이 아니라 crates/technique-api/src/lib.rs에 있다(D8 single-trait 설계). 시그니처가 &Tensor가 아닌 #[repr(C)] ABI struct(KiviAttnArgs/KiviGatherArgs — cl_mem을 *mut c_void로, 반환은 i32 0=OK·음수=err)라서 plugin이 엔진 타입을 전혀 참조하지 않고 컴파일된다. 엔진의 정적 OpenCL impl(impl KiviAttentionBackend for OpenCLBackend, opencl.rs L7215~), host dlopen 어댑터, plugin .so 세 곳이 모두 이 한 벌을 구현한다. 엔진 쪽 engine/src/capability/kivi_attention.rs는 re-export shim이다(동일 TypeId 보존 → CapabilityRegistry register/get 일관).

// technique-api — ABI-shaped canonical trait
pub trait KiviAttentionBackend: Send + Sync {
    fn has_kivi_attn_kernel(&self, bits: u8) -> bool;   // 2/4/8
    fn is_nosub_device(&self) -> bool;
    fn attention_gen_kivi(&self, args: &KiviAttnArgs) -> i32;  // 매 토큰 hot-path
    fn kivi_gather_update(&self, args: &KiviGatherArgs) -> i32;
}

엔진 내 소비자는 kv/kivi_format.rs가 backend.as_kivi_attention()으로, kivi_cache.rs가 construction 시점에 pull한 handle로 호출한다 — 매 토큰 lookup 없음.

근거: crates/technique-api/src/lib.rs::KiviAttentionBackend (L1284), KiviAttnArgs (L1245), engine/src/capability/kivi_attention.rs (re-export shim), engine/src/kv/kivi_format.rs L143·L274, engine/src/backend/opencl.rs L7215~7261

  • 진입점: session/plugin_dispatch.rs::register_dynamic_plugins. .so당 dlopen 1회 후 try_register_stage + try_register_format + try_register_backend_cap 세 축을 모두 시도한다(번들 .so 허용 — 단일축은 부분집합으로 정당). 세 축 합계 0이면 export_plugin! 누락으로 보고 bail.
  • 봉투 ABI: register_backend_caps_v2() -> BackendCapExportAbi { abi_version: u32, count: usize, vtables: *const BackendCapVTableAbi }. BACKEND_CAP_ABI_VERSION = 1 불일치면 .so 전체 거부. 각 엔트리는 { name: *const c_char, category: u32, vtable: *const c_void } 태그드 포인터.
  • category 다리(D7): host는 category로 vtable을 카테고리별 테이블로 캐스팅한다. 현재 BACKEND_CAP_CATEGORY_ATTENTION = 1 → *const KiviAttnVTable arm 1개뿐이고, 미지의 category는 등록 자체를 거부한다(새 카테고리 = host에 match arm 추가 = 재컴파일).
  • 등록 원자성: pass 1에서 이름 UTF-8·null vtable·빌트인(find_kivi_attention) 충돌·봉투 내부 중복을 모두 검사하고, pass 2에서 write-lock 1회로 동적 중복 검사 + 일괄 push — .so 하나의 등록이 부분 실패로 반쯤 남지 않는다.
  • 해석 우선순위: resolve_kivi_capability(name, make_args)는 정적 linkme 슬라이스(KIVI_ATTENTION_REGS) 우선 → 동적 registry fallback. 양쪽 miss는 None(graceful unknown). 동적 hit 시 vtable.make(KiviMakeArgs)로 plugin이 host의 cl_context/cl_device_id/build_opts를 빌려 커널을 1회 빌드하고, DynKiviAttentionBackend 어댑터가 opaque handle + vtable + Arc<Library>(.so 생존 보장)를 들고 trait를 구현한다. Drop에서 vtable.drop(handle) 정확히 1회.

5.3 example-backend-cap — synthetic 예제와 게이트 테스트

crates/techniques/example-backend-cap/(55 LOC)은 GPU 수학을 하지 않는 ABI round-trip 증명용 plugin이다. attention_gen_kivi는 mem 포인터 null 검사 후, scalar 필드들로 결정적 sentinel num_heads_q*1000 + head_dim + bits*0.5 + scale을 계산해 scores_out[0]에 기록한다 — host 게이트가 KiviAttnArgs의 필드 정렬·값이 ABI 경계를 정확히 넘었는지 확인하는 장치다. 등록은 매크로 두 줄:

technique_api::register_kivi_attention_plugin!("synth_kivi_attn",
    |_a: &KiviMakeArgs| -> Box<dyn KiviAttentionBackend> { Box::new(SynthKivi) });
technique_api::export_plugin!();   // .so 당 1회 — 3축 엔트리 emit

register_kivi_attention_plugin!은 정적 경로(linkme KIVI_ATTENTION_REGS 기여 — rlib force-link 빌드용)와 동적 경로(C thunk로 감싼 KiviAttnVTable + 봉투 엔트리 — plugin-cdylib feature 게이트)를 동시에 emit한다. Cargo 설정이 핵심이다: crate-type = ["cdylib", "rlib"], 그리고 plugin-cdylib feature가 꺼진 정적 빌드에서는 #[no_mangle] C 엔트리를 emit하지 않아 다수 plugin force-link 시 심볼 충돌을 차단한다.

통합 테스트 engine/tests/gate_c_backend_cap_dlopen.rs가 전 경로를 검증한다: cargo로 .so를 빌드 → register_dynamic_plugins → 이름 가시화 → resolve_kivi_capability → bool 쿼리 round-trip → sentinel 일치(32*1000 + 64 + 1.0 + 0.125) 단언. 실제 KIVI 커널 실행·&Tensor↔cl_mem 다리 무회귀는 S25 device 재증명 영역으로 게이트 밖이라고 명시되어 있다.

6. test_backend — CPU vs GPU 정확성 검증

engine/src/bin/test_backend.rs(1,082 LOC)는 backend 구현 간 수치 일치를 확인하는 Tier 2 검증 바이너리다. --backends auto,scalar(기본)로 backend 목록을 만든다: auto/cpu → CpuBackend(타겟별 NEON/AVX2), scalar → CpuBackendCommon, avx2, opencl/gpu → OpenCLBackend::new()(실패 시 경고 후 skip — GPU 없는 호스트에서도 동작).

6.1 op 테스트 루프

shape 매트릭스는 (m,k,n) 7종 — (1,64,32)부터 (64,256,128)까지 + Llama 7B 현실 shape (1,4096,4096). 각 backend × shape에 대해 MatMulTransposed(F32, k%32==0이면 Q4_0도), MatMulSlice, RMSNorm, Softmax, RoPE를 돌린다. 절차:

  1. 결정적 데이터 생성 — A는 (i%100)*0.01 - 0.5, B는 (i%123)*0.01 - 0.5. Q4_0은 테스트 자체가 블록 양자화(quantize_q4_0: max/7.0 스케일, nibble packing)를 수행하고 블록을 보관.
  2. OpenCL이면 backend.copy_from()으로 GPU 업로드.
  3. op을 10 iterations 실행, 매회 synchronize() — 시간도 함께 측정(정확성+간이 벤치 겸용).
  4. OpenCL이면 OpenCLBuffer downcast + buffer.read().enq()로 결과 readback.
  5. 검증: 대표 원소 1개(r_m = m/2, r_n = n/2)의 참조값을 CPU에서 재계산(Q4_0은 보관한 블록을 dequantize 후 dot)하고 abs diff > 1e-2면 FAIL.

근거: engine/src/bin/test_backend.rs::main (L60~243), perform_matmul_test (L372~624), run_matmul_test threshold (L344)

리뷰 포인트: matmul류 검증은 출력 전체가 아니라 중앙 원소 1개의 spot-check다. RMSNorm/Softmax/RoPE는 reference 계산 match arm이 없어 ref_sum=0.0과 출력 원소를 비교하게 되는데, 이 경우 PASS/FAIL은 사실상 "실행이 죽지 않았는가 + 출력이 0 근처인가"의 의미라 정확성 보증이 약하다. 커널 수정 PR에서 test_backend PASS만으로 정확성을 주장하면 의심하라 — e2e token-id 일치(11장 deploy-test)가 더 강한 게이트다.

6.2 KIVI attention oracle 테스트

OpenCL backend에 한해 Q2/Q4/Q8 각 비트로 run_kivi_attention_test를 돈다. 구성: kv_heads=8, head_dim=64, n_heads_q=32(GQA ratio 4), residual_size=32, 128 토큰(4 flush). CPU/GPU 동일 데이터의 KiviCache 두 벌을 만들고 — CPU 쪽은 get_view()로 dequant된 KV를 꺼내 naive single-token attention, GPU 쪽은 attention_gen_kivi() native 커널 — 출력의 relative L2 error(sqrt(sum(diff²))/sqrt(sum(ref²)))를 비교한다. threshold는 양자화 오차를 반영해 비트별로 다르다: Q2 0.15, Q4 0.08, Q8 0.05. GPU handle은 backend 생성 시 같은 인스턴스에서 파생한 Arc<dyn KiviAttentionBackend>를 전달한다(§5.1의 construction-time pull 패턴과 동일).

근거: engine/src/bin/test_backend.rs::run_kivi_attention_test (L709~773), perform_kivi_attention_test (L776~, L1051~1080 L2 계산), main L67~99 (handle 파생)

7. Adreno 커널 실측 교훈 3종

아래 3건은 CLAUDE.md에 기록된 디바이스 실측 교훈(코드만으로는 재확인 불가)이며, .cl 커널을 수정하는 모든 PR이 준수해야 하는 제약이다. 각 교훈이 코드에 어떻게 반영되어 있는지를 함께 적는다.

교훈 1 — DK=128 flash attention은 per-thread 32 float4 초과 시 register spill. 코드 대응: flash_attn_f32_f16.cl은 #if DK == 128일 때 FA_SUBGROUP_SPLIT=1, LANES_PER_QROW=2로 전환한다 — Q-row 하나를 lane 2개가 DK/DV 절반씩 나눠 맡아 per-thread q_priv[]/o_acc[] footprint를 2배 줄이고, half 간 partial sum은 SLM 1회로 합친다. DK=64(Llama 3.2 1B)는 회귀 방지를 위해 원래의 1-thread-per-Q-row 레이아웃을 유지한다. 리뷰 시 DK=128 경로에 per-thread private 배열을 늘리는 변경은 spill 재발 위험으로 본다.

근거: engine/kernels/flash_attn_f32_f16.cl L26~38 (FA_SUBGROUP_SPLIT 매크로) + L52~60 (lane layout 주석); 실측 한계치는 CLAUDE.md 실측 교훈

교훈 2 — sub_group_reduce_*는 SLM tree-reduce 대비 33~55% 느리다 (Adreno 830 실측). 코드 대응: 자주 도는 score 경로 커널(attention_scores.cl)은 max/sum 모두 "SLM tree-reduce" 주석과 함께 local memory 트리 환산을 쓴다. 반면 mul_mv_f16_f16.cl, mul_mv_id_*.cl 등 일부 GEMV 커널에는 sub_group_reduce_add가 남아 있다 — 이들은 llama.cpp 유래 커널로, Adreno hot path에서 신규로 sub_group_reduce_*를 도입하는 변경은 이 실측을 근거로 반려 대상이다.

8. 리뷰 체크리스트 요약

  • 새 default trait 메서드가 host 포인터를 만지면 → supports_* 가드 짝이 있는가, 호출지가 가드하는가 (§2).
  • get_extension·as_any().downcast_ref가 forward/decode loop 안에 들어오지 않았는가 (§2.1).
  • NEON 신규 커널: dotprod/i8mm 런타임 감지 분기와 비-dotprod fallback이 모두 있는가 (§3.2).
  • OpenCL 신규 커널: include_str! 등록 + nosub fallback 고려 + KernelCache 필드 + enqueue_kernel_labeled 경유인가 (§4.1, §4.4).
  • Q4_0 weight를 만지는 변경: noshuffle SOA registry의 cl_mem 키 invalidate/재등록 순서가 지켜지는가 (§4.2).
  • rpcmem 경로: activation alloc에 rpcmem이 새지 않는가(INV-RPCMEM-007), per-buffer fallback이 유지되는가 (§4.5).
  • plugin ABI 변경: BACKEND_CAP_ABI_VERSION bump + gate 테스트 갱신이 같이 가는가 (§5.2).
  • .cl 수정: Adreno 실측 교훈 3종 위반 여부 (§7).
