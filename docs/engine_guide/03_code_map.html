2. no-mod.rs 모듈 스타일

이 코드베이스의 디렉토리 모듈 루트는 foo/mod.rs가 아니라 형제 파일 foo.rs다. 예를 들어 backend 모듈은 engine/src/backend.rs(trait Backend 정의 + 서브모듈 선언)와 engine/src/backend/(cpu, opencl, cuda_pc, … 구현 디렉토리)의 쌍으로 구성된다. backend.rs 머리의 주석이 이 이동을 직접 기록하고 있다: "Moved from backend/mod.rs to the top-level engine/src/backend.rs per Step 4-A (Rust 2018+ pattern: trait definition lives next to its implementations, no mod.rs needed)". 

engine/src/
├── backend.rs        ← 모듈 루트: trait Backend + pub mod cpu; pub mod opencl; ...
├── backend/          ← 서브모듈 본체
│   ├── cpu.rs  +  cpu/        (같은 패턴이 재귀 적용)
│   ├── opencl.rs  +  opencl/
│   └── ...
├── kv.rs             ← CachePressureHandler trait + pipeline + 서브모듈 선언 (구 pressure.rs, γ-1 rename)
├── kv/               ← kv_cache.rs, eviction.rs, kivi_cache.rs, ... (구 pressure/)
└── ...

이 스타일의 실익은 두 가지다. (1) 모듈의 공개 표면(trait·공유 타입)이 디렉토리 밖 단일 파일에 모여 디렉토리를 열지 않고도 모듈 계약을 읽을 수 있다 — backend.rs에 trait Backend, kv.rs에 CachePressureHandler/CachePressurePipeline/ActionResult, memory.rs에 trait Memory가 산다. (2) 에디터 탭이 전부 mod.rs로 보이는 문제가 없다. 신규·이동 모듈은 반드시 이 스타일을 따라야 한다(프로젝트 전역 컨벤션, 12. 컨벤션 & 리뷰 체크리스트).

근거: engine/src/backend.rs L1–5 (Step 4-A 이동 주석), engine/src/memory.rs L1–4, engine/src/buffer.rs L1–3 — 세 파일 모두 동일한 "no mod.rs" 이동 기록

리뷰 함정: PR에 foo/mod.rs 신설이 보이면 컨벤션 위반이다. 기존 코드에 nested mod.rs가 일부 남아 있을 수 있으나(일괄 sweep은 별도 chore 예정 — CLAUDE.md 기록), 신규 코드가 그 패턴을 모방하면 안 된다.

4. 모듈 간 의존 방향 그래프

화살표는 의존 방향이다(A → B = A가 B를 import). 그래프의 핵심 구도: 옛 core/ 디렉토리가 해체되어 톱레벨 L2 모듈군(backend.rs·buffer.rs·memory.rs·tensor.rs·shape.rs·format.rs·kv_cache_ops.rs 등)이 되었고, 이 L2 공통 어휘가 backend 구현·models·layers·pressure 모든 도메인의 공통 의존이다. 각 파일의 주석이 이 승격 이력을 직접 기록한다 — backend.rs: "originally core/backend.rs", memory.rs: "originally core/memory.rs", inference.rs/kv.rs: "Step 4-C/4-D: core/에서 promote". 즉 과거 문서가 말하는 "core가 공통 의존"이라는 소유 구조는 지금 코드에서 플랫한 L2 톱레벨 층으로 실현되어 있다.

근거: engine/src/backend.rs L33 ("originally core/backend.rs"), engine/src/memory.rs L27, engine/src/inference.rs L1 ("Step 4-C: core/에서 promote"), engine/src/kv.rs L8 ("Step 4-D")

tensor ↔ backend 상호 참조는 의도된 구조다. Tensor가 Arc<dyn Backend> 필드를 들고 (tensor.rs L11), Backend의 연산 메서드가 &Tensor를 받는다(backend.rs L134). 같은 crate 안이라 순환 import가 허용되며, "텐서가 자기 실행 위치를 안다"는 설계의 비용이다. 마찬가지로 backend.rs ↔ capability.rs도 상호 참조다 — backend가 capability 서브모듈의 trait(KiviAttentionBackend 등)를 re-export하고, capability 구현이 backend 타입을 안다.

의존 방향에서 눈여겨볼 비자명한 간선 세 가지:

  • pressure → models: weight swap 계열(weight_swap_handler, weight/*)이 models::weights의 LayerWeights 등을 직접 다룬다. KV만이 아니라 weight도 pressure 도메인의 관리 대상이기 때문. 역방향(models → pressure 소유 자원)은 runtime_resources_access.rs trait로 역전시켜 끊었다.
  • resilience → pressure: CommandExecutor가 Manager의 Evict 명령을 CachePressurePipeline 실행으로 번역한다. enum-as-data 식별자(EvictMethod)도 pressure에서 re-export.
  • L1 backend → L2 kv_cache_ops / partition_workspace: OpenCL plan.rs가 KV 캐시 형태와 partition workspace를 알아야 해서, 해당 타입들을 L3에서 L2로 승격해 "backend가 도메인 모듈을 import하는" 위반을 해소했다 (각 파일 doc의 §13.8-G shared identifier promotion 기록).

근거: engine/src/kv/weight_swap_handler.rs·weight/*.rs의 crate::models import (grep 검증), engine/src/resilience/executor.rs의 crate::kv import, engine/src/partition_workspace.rs·kv_cache_ops.rs 모듈 doc

5. crates/ 워크스페이스 멤버

5.1 technique-api — plugin 등록 표면

확장 기법(stage 축)이 엔진 코어 수정 0으로 자기를 등록하기 위한 단방향 의존 표면이다. 의존 방향은 engine → technique-api ← technique crate — 이 crate는 엔진 타입(KVCache/Backend)을 참조하지 않고, 기법이 읽을 캐시 상태를 자체 정의 읽기 추상 StageCtx로 노출한다(엔진이 &KVCache 위로 구현). 핵심 계약: 

pub trait KVCacheStage: Send + Sync {
    fn name(&self) -> &str;                                  // CLI --eviction-policy 매칭
    fn plan(&self, ctx: &dyn StageCtx) -> Option<KVCachePlan>; // None = no-op
}
pub struct KVCachePlan {
    pub keep: KeepSpec,            // LayerWide(Vec<usize>) | PerHead(Vec<Vec<usize>>)
    pub merges: Vec<WeightedMerge>, // d2o 식 가중 병합 지시 (없으면 빈 Vec)
}
#[distributed_slice]
pub static KV_CACHE_STAGES: [KVCacheStageReg] = [..];        // linkme 링크 타임 수집

기법은 plan만 반환하고 버퍼를 직접 변형하지 않는다 — 실행(compact)은 엔진이 독점한다. 정적 등록 외에 GATE-C .so cdylib 경로도 같은 파일에 있다: trait object를 fn-ptr 테이블(StageCtxAbi) + opaque handle로 평탄화한 C-ABI(KV_STAGE_ABI_VERSION = 2)이고, 엔진 쪽 진입은 session/plugin_dispatch.rs::register_dynamic_plugins()(--load-plugin, .so당 dlopen 1회 후 stage ⊥ format 양 축 try_register)이다. backend 축 capability 등록 슬라이스 BACKEND_CAPABILITIES(trait BackendCapability)도 같은 crate에 평행하게 존재한다. plugin 작성 절차는 10. 확장 How-to.

5.3 qnn_oppkg · shared · manager

  • crates/qnn_oppkg — QNN GPU OpPackage. production 추론 경로에서는 제거되었고(engine/Cargo.toml 주석: "production engine/src/은 이 crate에 의존하지 않는다"), --features qnn의 microbench 바이너리 (논문 M3/M4/M5 evidence)에서만 빌드된다.
