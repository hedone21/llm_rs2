1. 한눈에 보는 계층 구조

추론 한 번은 다섯 계층을 수직으로 관통한다. 위에서부터 session 계층(루프 소유·정책 훅), 모델 계층(TransformerModel::forward_into — embedding→layer loop→lm_head 오케스트레이션), 레이어 계층(TransformerLayer — RMSNorm→QKV→RoPE→attention→FFN의 본체), format 계층(KVCacheFormat trait object — KV write와 attention을 저장 형태별로 캡슐화), 마지막으로 backend 계층(Backend trait — matmul/rope/attention 커널 dispatch)이다. 이 챕터는 위 세 계층 + format 호출 표면까지를 다룬다. format 구현체 내부는 07장, backend 커널은 09장 참조.

2. session 계층 — 누가 루프를 소유하는가

2.1 standard happy path와 DecodeLoop

단일 프롬프트 추론(argus_cli)의 표준 경로는 run_standard_happy_path()다. 이 함수는 build_standard_loop()로 DecodeLoop를 조립한 뒤 decode_loop.prefill(&tokens) → 마지막 logits에서 sampling::sample()로 첫 토큰 산출 → decode_loop.run(num_tokens - 1, first_token) 순으로 실행하고, TTFT / Decode: X ms/tok / Avg TBT를 출력한다. 단, 이 경로는 is_standard_happy_path() 가드를 통과한 경우에만 쓰인다 — qcf_dump·skip_ratio·d2o_layer_alloc·profile·profile_events· eviction_policy != "none"·tensor_partition != 0·swap 계열 플래그 중 하나라도 켜지면 legacy 경로로 빠진다.

근거: engine/src/session/standard_happy.rs::run_standard_happy_path, engine/src/session/assembly/build_standard_loop.rs::is_standard_happy_path (L60)

DecodeLoop는 decode 루프 그 자체를 소유하는 구조체로, 7개 trait object를 들고 있다: Forward(필수, typestate builder가 강제), EvictionStage, SwapStage, CommandSource, TokenSampler, Vec<DecodeObserver>, TokenTickSink(+EngineReport). 미지정 슬롯은 defaults.rs의 NoOp 구현으로 채워진다. run()의 step당 순서는 코드 주석 라벨 그대로:

단계호출역할
(a)cmd_source.poll(ctx, kv_snap)resilience ExecutionPlan 수신 — suspend면 break, throttle이면 sleep
(a.5)forward.try_evict(...)plan.evict가 Some이고 첫 발동이면 mid-decode KV prune (sticky 1회 게이트 evict_applied)
(a.6)forward.try_offload / try_recallresilience KvOffload/RestoreDefaults — prefix를 disk로 swap/복귀

2.2 ModelForward — Forward trait의 표준 구현

Forward trait의 계약은 prefill(tokens, start_pos) -> Vec<f32>(마지막 토큰 logits)와 step(ctx, token) -> Vec<f32> 두 메서드가 핵심이고, 나머지(on_kv_prune/reset_kv/try_evict 등)는 default no-op이다. 표준 구현 ModelForward는 backend·memory·model Arc·KV cache·decode workspace·재사용 logits 텐서를 모두 소유한다. KIVI용 KiviForward, 토큰 스트리밍 offload용 OffloadForward가 같은 trait의 다른 구현이다.

ModelForward의 핵심 동작 세 가지:

  • fmt wrap (1회, lazy) — ensure_fmt_wrapped()가 prefill 시작 시 Vec<KVCache>를 mem::take로 비우고 Vec<Arc<StandardFormat>>로 감싼다(layer index = enumerate 순서). 물리 캐시는 fmt 안에 단 한 벌만 존재하고, 이후 forward·eviction 모두 interior mutability(Mutex)로 &self를 통과한다.
  • prefill 청크 분할 — derive_chunk_size()가 GPU max_single_alloc으로부터 min(budget/(vocab*4), max_alloc/(hidden*4), 512, seq_len)을 산출, 긴 프롬프트를 청크 루프로 나눠 각 청크를 forward_into(logits_last_only=true)로 처리한다. CPU는 분할 없음.
  • plan-first decode — step()은 OpenCL일 때 사전 바인딩된 GPU 커널 plan을 먼저 시도하고(§6), 실패·invalidation 시 forward_into(trait object 경로)로 폴백한다. 마지막에 read_logits()가 backend.synchronize() 후 [1,1,vocab] logits를 host로 읽는다 — 이 synchronize가 토큰당 유일한 강제 sync 지점이다.
// engine/src/session/traits.rs
pub trait Forward {
    fn prefill(&mut self, tokens: &[u32], start_pos: usize) -> anyhow::Result<Vec<f32>>;
    fn step(&mut self, ctx: &StepCtx, token: u32) -> anyhow::Result<Vec<f32>>;
    // finalize / on_kv_prune / reset_kv / try_evict / try_offload / try_recall = default
}

3. 모델 레벨 — forward_into의 Prefill/Decode 분기

TransformerModel::forward_into(TransformerModelForwardArgs)가 prefill과 decode를 모두 처리하는 단일 진입점이다. 인자 구조체의 핵심 필드는 input_tokens(U32 [batch, seq_len]), start_pos, fmts: &[Arc<dyn KVCacheFormat>](레이어당 1개), logits_out, x_gen(decode 전용 hidden scratch), workspace: Option<&mut LayerWorkspace>, logits_last_only이며, eval 전용으로 score_accumulator/skip_config/importance_collector/cache_self_need_scores가 있다 (production ModelForward는 전부 None/false 주입).

분기 기준은 단 하나다: let is_decode = seq_len == 1;

공통 골격: ① gather_embed(embedding lookup, embed_scale 옵션) → ② layer loop — 레이어마다 self.layers[i].load_weights()로 per-token weight snapshot을 떠서(Arc<TransformerLayer>, weight swap과의 경합 차단) SWIFT skip 플래그·Gemma3 local/global RoPE theta를 계산해 fmt fork에 위임 → ③ final rms_norm → ④ lm_head matmul (lm_head_on_cpu면 CPU 경로). 발산 A: decode인데 workspace가 None이면 prefill 경로를 탄다. is_decode && workspace.is_none()이면 1-token degenerate forward_prefill_fmt로 fall-through한다 (L1473–1512). 주석에 따르면 warmup(warmup_tokens=1)·qcf decode 경유만 이 분기에 들어오고 production 호출처는 모두 workspace=Some이다. forward_gen_fmt와 forward_prefill_fmt는 attention 누산 순서가 달라서(inline single-query vs flash) 이 분기를 모르고 workspace를 빼먹으면 에러 없이 미세하게 다른 수치가 나온다. decode 경로를 건드리는 PR 리뷰 시 이 분기 커버 여부를 확인하라.

4.2 Prefill: forward_prefill_fmt (seq_len 초과 1)

prefill 짝 forward_prefill_fmt(ForwardPrefillFmtArgs)는 같은 op 골격이되 multi-token이라 다음이 다르다:

  • PrefillWorkspace 텐서를 매 레이어 진입 시 실제 seq_len으로 reshape (alloc은 max로 1회).
  • residual을 fused norm이 아니라 copy_slice(x → residual) + in-place rms_norm(x)로 처리 — decode와 달리 residual 보존을 명시적 copy로 한다 (residual·residual_ffn 2개).
  • RoPE는 [batch, seq_len, n_heads, head_dim] shape로 한 번에.
  • KV write는 fmt.write_kv_batch, attention은 같은 fmt.attention_into지만 seq_len이 1보다 커서 impl 내부에서 prefill arm(multi-token causal)으로 분기한다(§5.2).
  • FFN은 fused 커널 없이 matmul × 2 + silu_mul(또는 gelu_tanh_mul) + down matmul.
  • scores 인자는 항상 None — prefill은 attention score를 누적하지 않는다.

근거: engine/src/layers/transformer_layer/forward_prefill_fmt.rs::forward_prefill_fmt (L56–203)

4.3 KV write — format 계층의 fast-path 사다리

StandardFormat(F32/F16/Q4_0 KVCache wrapper)의 write_kv/write_kv_batch는 공통 write_inner(new_k, new_v, backend, is_decode)로 합류해 다음 순서로 fast-path를 고른다:

  1. GPU + F16 + HeadMajor + decode → kv_scatter_f32_to_f16 1 dispatch (fused cast+scatter).
  2. GPU + F32 + HeadMajor → kv_scatter_f32_to_f32_batch (decode는 count=1, prefill은 count=seq_len).
  3. GPU + F16 + HeadMajor + prefill → kv_scatter_f32_to_f16_batch (supports_kv_scatter_batch() 게이트 필수 — host-pointer fallback이 device-only 버퍼에서 segfault).
  4. opaque format(.so block-quant) → cast 없이 KVCache::update가 descriptor encode+scatter.
  5. 그 외 비-F32(Q4_0 전부, 게이트 미충족 F16) → lock 안의 lazy k_cast/v_cast scratch에 backend.cast 후 cache.update. Q4_0는 GPU fast-path가 없어서 항상 이 경로다.

모든 경로가 write 후 advance_pos(1 또는 seq_len)로 current_pos를 전진시킨다. HeadMajor offset 산식 — dst_off = h*capacity*head_dim + pos*head_dim — 은 scatter 커널과 KVCache::update가 bit-identical하게 공유한다.

근거: engine/src/pressure/standard_format.rs::write_inner (L150–329), ::write_kv / write_kv_batch (L345–354)

5. Attention과 GQA 구현 상세

5.1 GQA 그룹 매핑 — 코드 전 경로 공통 산식

GQA는 여러 Q 헤드가 하나의 KV 헤드를 공유하는 구조다(도메인 환기 끝). 이 코드베이스의 매핑은 모든 경로에서 동일한 한 줄이다: 

let n_rep = n_heads_q / n_heads_kv;   // 이름이 gqa_ratio 인 곳도 있음
let kv_h  = h / n_rep;                // Q 헤드 h가 읽을 KV 헤드

Llama 3.2 1B 기준(모델 메타데이터: num_attention_heads=32, num_key_value_heads=8, head_dim=64) n_rep=4, 즉 Q 헤드 0..3이 KV 헤드 0을, 4..7이 KV 헤드 1을 공유한다. 별도 K/V repeat(복제) 텐서는 만들지 않고, 오로지 인덱스 계산으로 같은 KV 행을 4번 읽는다. 등장 위치별 형태:

경로코드GQA 인덱스
CPU decode 기본 attentionBackend::attention_gen default implkv_h = h / gqa_ratio; K offset = HeadMajor (kv_h*capacity + t)*head_dim, SeqMajor (t*num_heads_kv + kv_h)*head_dim. layout은 shape에서 감지(k_shape[1] == num_heads_kv)
CPU prefill flashflash_attention_forward_stridedQ 헤드를 rayon으로 병렬화, head별 k_head_offset = kv_h * kv_head_stride로 slice 시작점을 옮긴 뒤 flash_attention_head 호출

5.2 flash_attention_forward_strided의 stride 계약

CPU prefill attention의 핵심 함수. 시그니처의 stride 인자가 layout을 전부 흡수하므로 함수 자체는 layout-agnostic이다. prefill_attention(StandardFormat의 prefill arm)이 넘기는 실제 인자: 

// engine/src/pressure/standard_format.rs::prefill_attention (L956-999)
let (k_pos_stride, kv_head_stride) = if is_head_major {
    (head_dim, kv_capacity * head_dim)      // HeadMajor: 같은 head 안에서 pos 연속
} else {
    (n_heads_kv * head_dim, head_dim)       // SeqMajor: 같은 pos 안에서 head 연속
};
flash_attention_forward_strided(
    q_slice, k_slice, v_slice, out_batch,
    n_heads_q, n_heads_kv,
    seq_len,            // q_len
    cache_seq_len,      // kv_len (캐시 전체 유효 토큰)
    head_dim,
    n_heads_q * head_dim,  // q_stride (토큰 간)
    k_pos_stride, k_pos_stride,   // k_stride, v_stride
    n_heads_q * head_dim,  // out_stride
    kv_head_stride,        // KV 헤드 간 stride — GQA 점프 단위
    q_start_pos,           // = cache_seq_len - seq_len (causal mask 기준점)
    32, 32,                // br, bc 타일 (NEON 경로는 자체 타일 선택)
    window,                // Gemma3 SWA — None이면 full causal
);

causal mask는 global_c <= global_r + q_start_pos, sliding window는 global_c + ws > global_q로 flash 커널 내부에서 처리한다(블록 단위 skip 최적화 포함). aarch64에서는 q_len >= PREFILL_FLASH_THRESHOLD + head_dim 8배수 + 정수 GQA ratio + HeadMajor(k_stride == head_dim) 조건이 모두 맞으면 NEON tile + online-softmax 구현 flash_prefill_forward_f32_neon으로 우회한다.

근거: engine/src/layers/attention.rs::flash_attention_head (L78–221, mask L142–146), NEON gate (L260–293)

5.3 attention_into의 dispatch 트리

fmt.attention_into(q, backend, out, AttnDims{n_heads_q, window}, scores)는 호출자(레이어)가 저장 형태를 전혀 모르는 채 부르는 단일 표면이다. AttnDims가 q 속성 2개만 담는 이유: n_heads_kv/head_dim/capacity/current_pos/scale은 전부 format이 자기 상태로 아는 값이기 때문이다. StandardFormat impl의 분기: 

flowchart TD
    A["attention_into(q, dims, scores)"] --> B{"cache.is_opaque()?"}
    B -->|yes| C["descriptor로 K/V 전체 f32 unpack 후 F32 경로 재사용"]
    B -->|no| D{"seq_len 초과 1? (prefill)"}
    D -->|yes| E["prefill_attention: GPU flash_attention_prefill 시도"]
    E -->|"dispatch 성공"| E1["GPU flash prefill 커널"]
    E -->|"미지원 (Q4_0, head_dim, CPU)"| E2["dtype별 dequant 후 flash_attention_forward_strided (CPU)"]
    D -->|"no (decode)"| F["effective_cache_len = min(cache_seq_len, window)"]
    F --> G{"kv_dtype == Q4_0 그리고 backend.is_gpu()?"}
    G -->|yes| H["attention_q4_gpu_fallback: GPU에서 Q/K/V readback, CPU dequant+attention, 결과 write_buffer"]
    G -->|no| I["backend.attention_gen — CPU dtype-aware 또는 GPU flash decode 커널"]

decode arm의 window 처리: effective_cache_len = min(cache_seq_len, window)로 K 길이를 잘라 backend에 넘기고, 레이어 쪽에서는 ws.score_offset = cache_seq_len - effective_cache_len을 기록해 score accumulator가 ws.scores[t]를 캐시 절대 위치로 되돌릴 수 있게 한다. prefill arm은 반대로 clamp를 우회하고 전체 K + window를 flash 내부 마스킹에 위임한다(주석 "정정 C"). GPU decode는 OpenCLBackend::attention_gen override가 flash decode 커널(flash_attention_decode_gpu)을 우선 시도하며, CPU로의 score readback(scores_out=Some)이 필요하고 GPU score accumulator가 비활성이면 legacy 커널로 폴백한다.

근거: engine/src/pressure/standard_format.rs::attention_into (L356–508), engine/src/format/kv_cache_format.rs::AttnDims (L28), engine/src/backend/opencl.rs::attention_gen (L6000–6060), engine/src/layers/transformer_layer/forward_gen_fmt.rs (score_offset L154)

Q4_0 KV + GPU는 매 토큰 readback 폴백이다. OpenCL backend에는 Q4_0 dequant-attention 커널이 없어 backend.attention_gen에 그대로 넘기면 BlockQ4_0 raw 바이트를 float로 오독해 garbage가 난다. 그래서 attention_q4_gpu_fallback이 K/V 캐시 전체를 GPU→CPU로 읽어 rayon 병렬 dequant+attention 후 결과만 GPU에 되쓴다. 정확하지만 캐시가 길수록 비싸다 — Q4_0 KV를 GPU에서 쓰는 변경을 리뷰할 땐 이 비용을 전제로 보라. NaN guard(스코어 NaN → -inf, sum 붕괴 시 uniform)도 이 경로에만 있다.

근거: engine/src/layers/transformer_layer/forward_gen.rs::attention_q4_gpu_fallback (L189–351)

5.4 score 누적 (need_scores)

H2O/D2O eviction이 쓰는 post-softmax score는 decode에서만 생산된다. 결정식은 need_scores = acc_need || cache_self_need_scores이며, GPU score accumulator가 활성이면 acc_need=false(on-device 누적으로 대체, layer loop 끝에 gpu_acc.end_step). CPU 경로에서는 attention_into(scores=Some(&mut ws.scores))가 head별 [0..cache_seq_len]을 채우고, 모델 레벨에서 accumulate_layer(layer-wide) 또는 accumulate_layer_gqa(per-KV-head)로 흡수한다. production happy path는 score_accumulator=None이라 전부 비활성 — eviction과의 연결은 07장.

근거: engine/src/models/transformer.rs::forward_into (need_scores L1516–1523, accumulate L1587–1615), engine/src/inference/attention_scores.rs::AttentionScoreAccumulator

6. plan 경로 — 사전 바인딩 GPU 커널 plan

OpenCL decode의 진짜 핫패스는 forward_into가 아니라 FullKernelPlan이다. 매 토큰 같은 커널을 같은 인자(weight·workspace cl_mem·차원)로 dispatch하는 decode 특성을 이용해, build 시점에 모든 정적 인자를 커널 객체에 pre-bind하고 실행 시에는 동적 스칼라만 갱신한다 (파일 헤더: "mirrors llama.cpp's tight enqueue loop"). 구성:

  • FullKernelPlan { layers: Vec<LayerKernelPlan>, final_norm, lm_head: Option<KernelStep>, kv_capacity, ratio_generation_at_build, ... }. lm_head=None이면(CPU tied embedding 등) 호출자가 CPU matmul을 대신 수행.
  • 각 KernelStep은 pre-bound 커널 + work size + dynamic_args 목록. 동적 인자 종류는 StartPos / CacheSeqLen / WritePos / KvCapacity(+KIVI 전용 ResPos/Q2Tokens/...).
  • op 순서는 OpTag가 보여준다: RmsNorm → MatmulQKV → Rope → KvScatter → Attention → MatmulWo → AddRmsNorm → MatmulGateUp → SiluMul → MatmulDown → (AddAssign) → FinalNorm → LmHead — §4.1 표와 1:1 대응.

execute()는 레이어마다 StandardFormat::plan_geometry()(단일 lock으로 current_pos/capacity 스냅샷)를 읽어 cache_seq_len = write_pos = current_pos, attn_seq_len = cache_seq_len + 1(방금 scatter한 토큰 포함)을 동적 인자로 주입하고, 레이어 끝에 plan_advance(1)로 pos를 전진시킨다. embedding lookup(gather_embed)과 logits readback만 plan 밖이다.

Invalidation 3종 — 셋 중 하나면 PlanInvalidated를 반환하고 caller가 폐기/재빌드한다:

  1. g.capacity != self.kv_capacity — KV 캐시 resize (grow/shrink).
  2. g.current_pos >= g.capacity — capacity overflow 직전.
  3. ratio_generation atomic 카운터가 build 시점 스냅샷과 다름 — weight swap이 일어나 pre-bound cl_mem이 stale (INV-129).

ModelForward::step() 쪽 수명 관리: plan은 lazy 1회 build(try_build_plan — OpenCL + --no-gpu-plan 비활성 + 비Gemma3 가드), build 실패 시 sticky_disabled=true로 매 step 재시도 spam을 차단, eviction이 KV pos를 shift하면 on_kv_prune()이 gpu_plan = None으로 무효화한다(stale offset → silent garbage 방지). execute 실패는 그 step을 forward_into 폴백으로 처리하고 다음 step에서 lazy rebuild한다. 

flowchart TD
    S["ModelForward::step(token)"] --> W["decode_input에 token write_buffer"]
    W --> P{"gpu_plan 있음?"}
    P -->|"None이고 sticky 아님"| B["try_build_plan: OpenCL? no_gpu_plan? Gemma3? 검사 후 model.build_plan"]
    B -->|"None"| ST["sticky_disabled = true"]
    B -->|"Some"| E
    P -->|Some| E["model.execute_plan → plan.execute(ocl, start_pos, fmt handles)"]
    E -->|"Ok(true)"| R["read_logits: synchronize + readback"]
    E -->|"Ok(false) / Err (invalidated)"| F["forward_into 폴백 (trait object 경로)"]
    ST --> F
    F --> R

근거: engine/src/backend/opencl/plan.rs::FullKernelPlan (L950), ::execute (L1278–1330, invalidation L1309–1312), ::OpTag (L36), engine/src/models/transformer.rs::execute_plan (L2190), engine/src/session/forward/model_forward.rs::{try_build_plan, step, on_kv_prune}

8. 함정 · 실측 교훈 · 리뷰 체크포인트

성능 측정은 --profile 없이 (실측 교훈, CLAUDE.md). --profile은 plan 경로를 비활성화하고 (is_standard_happy_path가 false → legacy 경로) op마다 synchronize() 2회를 끼워 ~54 ms/token의 sync 오버헤드를 더한다. per-op breakdown은 상대 비교에만 쓰고, production TBT는 Decode: X ms/tok 로그 라인(§2.1의 run_standard_happy_path 출력)을 쓴다. 이는 코드만으로 확인 불가한 디바이스 실측 지식이다.

KVLayout은 production HeadMajor 고정. attention 함수군은 stride 인자로 layout-agnostic하지만, plan 경로의 scatter 커널과 eviction/QCF 코드가 HeadMajor offset을 전제한다. SeqMajor를 고르면 컴파일·실행 모두 통과하면서 plan 경로가 silent garbage를 낸다(2026-04-14 실측 교훈, CLAUDE.md/용어집). 새 KV 코드가 offset을 직접 계산한다면 (h*capacity + pos)*head_dim 패턴인지 확인하라.

decode와 prefill의 attention은 수치적으로 다른 경로다. decode는 single-query 누산(attention_gen), prefill은 flash online-softmax다. FP 누산 순서가 달라 같은 토큰을 두 경로로 계산하면 bit-identical하지 않다 (forward_gen_fmt.rs 헤더가 명시). 회귀 테스트에서 prefill 출력과 decode 출력을 직접 비교하는 설계는 금물.

리뷰 체크리스트(이 챕터 범위): 컨벤션 전반은 12장.

  • 새 per-layer 연산은 forward_gen_fmt와 forward_prefill_fmt 양쪽에 넣었는가? plan 경로(build_full_plan의 KernelStep 시퀀스)도 갱신했는가? 셋 중 하나만 고치면 경로별로 다른 모델이 된다.
  • KV 캐시 geometry를 바꾸는 코드는 plan invalidation을 트리거하는가(capacity 변화 또는 on_kv_prune)?
  • fmt.attention_into 호출 전에 write_kv가 선행하는가? current_pos()는 write 이후 값을 읽어야 새 토큰이 attention에 보인다 (plan 경로의 attn_seq_len = cache_seq_len + 1과 동치).
  • GQA 인덱스를 새로 쓰는 코드는 h / n_rep인가 h % n_rep인가 — 전자가 정답이며, 기존 5개 경로(§5.1 표)와 일치해야 한다.
