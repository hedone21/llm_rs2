06. 메모리 & 데이터 포맷

기준 커밋: 46ee4c2d · 작성일: 2026-06-10 · llm.rs Engine Guide

0. 이 챕터에서 다루는 것

이 챕터는 데이터가 어디에 어떤 바이트로 놓이는가를 다룬다. 구체적으로 네 층이다: (1) 바이트 컨테이너 추상화인 Buffer trait과 그 구현 카탈로그(buffer/, memory/), (2) CPU↔GPU zero-copy 전략 두 갈래(CL_MEM_ALLOC_HOST_PTR 기반 UnifiedBuffer vs rpcmem DMA-BUF + CL_MEM_USE_HOST_PTR alias), (3) Q4_0/Q8_0 등 quantization 블록의 바이트 레이아웃과 quant/dequant 함수 경로, (4) 모델 가중치가 디스크에서 메모리로 들어오는 mmap 경로(GGUF / AUF / Safetensors). KV 캐시의 구조와 관리 정책은 07장, 연산 dispatch는 09장 담당이고, 여기서는 그들이 딛고 서는 저장 계층만 본다.

1. 바이트 표면: Buffer trait과 DType

1.1 Buffer trait 계약

engine/src/buffer.rs의 Buffer trait이 엔진 전체의 물리 메모리 표면이다. 핵심 계약은 "한 덩어리의 바이트를 CPU 포인터와 (선택적으로) GPU 핸들 양쪽으로 노출한다"는 것: 

pub trait Buffer: Send + Sync {
    fn as_any(&self) -> &dyn Any;          // downcast 표면 (OpaqueBuffer 식별 등)
    fn dtype(&self) -> DType;
    fn size(&self) -> usize;                // 총 바이트
    fn as_ptr(&self) -> *const u8;          // CPU 읽기 포인터 (없으면 null)
    fn as_mut_ptr(&self) -> *mut u8;        // &self → *mut: 내부 가변성 전제
    fn cl_mem(&self) -> Option<&Mem>;       // OpenCL 핸들 (CPU 전용이면 None)
    fn sync_device(&self) -> Result<()>;
    // zero-copy 상태 기계 (기본 구현 = no-op)
    fn map_for_cpu(&self) -> Result<()> { Ok(()) }
    fn unmap_for_gpu(&self) -> Result<()> { Ok(()) }
    fn is_mapped(&self) -> bool { true }
    fn is_host_managed(&self) -> bool { true }  // madvise 유효 여부
    fn is_gpu_buffer(&self) -> bool { false }
}

주의할 계약 세부 두 가지. 첫째, as_mut_ptr(&self)는 &self에서 *mut u8을 내준다 — 구현체들은 Arc로 공유되고 호출자가 동기화를 책임지는 구조이며, 읽기 전용 구현(MmapBuffer, RpcmemAliasBuffer)도 진단·주소 식별 목적으로 같은 포인터를 반환하되 "쓰기는 계약상 금지"라고 주석에 명시한다 (AufViewBuffer만 null_mut() 반환). 둘째, is_host_managed()는 madvise가 효과 있는 앱 관리 메모리인지를 나타낸다 — heap/mmap은 true, driver-pinned UnifiedBuffer와 device-only OpenCLBuffer는 false.

근거: engine/src/buffer.rs::Buffer (trait, L50–108), engine/src/memory/host/mmap.rs::MmapBuffer::as_mut_ptr (L102), engine/src/buffer/auf_view_buffer.rs::AufViewBuffer::as_mut_ptr (L79)

1.2 DType과 size() 함정

pub enum DType { Q4_0, Q4_1, Q8_0, F16, BF16, F32, U8 }

impl DType {
    pub fn size(&self) -> usize {
        match self {
            DType::Q4_0 | DType::Q4_1 | DType::Q8_0 => 1, // block quantized — 별도 처리
            DType::F16 | DType::BF16 => 2,
            DType::F32 => 4,
            DType::U8 => 1,
        }
    }
}
함정: quant dtype의 DType::size()는 1을 반환한다. Q4_0의 실제 비용은 element당 0.5625B(블록당 18B/32원소)인데 size()는 1이다. 즉 numel × dtype.size()식 바이트 회계는 quant dtype에서 항상 틀린다. 코드베이스의 올바른 패턴은 dtype별 match로 블록 수를 계산하는 것 — (n_values / QK4_0) * size_of::<BlockQ4_0>() (예: kv_cache.rs::shrink_to_fit L393, standard_format.rs L284). PR 리뷰에서 quant dtype에 dtype.size()를 곱하는 새 코드를 보면 의심하라.

근거: engine/src/buffer.rs::DType::size (L37–45), engine/src/pressure/kv_cache.rs::shrink_to_fit (L392–397)

1.3 Tensor: Buffer에 형상과 실행 위치를 태깅

Tensor는 Shape + Arc<dyn Buffer> + Arc<dyn Backend> 3요소 결합이다. 텐서가 자기 backend 핸들을 들고 다니므로 "이 데이터를 어디서 연산할지"를 storage에서 직접 얻는다. Tensor::dtype()·size()는 buffer로 위임이고, as_slice::<T>()는 size() / size_of::<T>()개의 T로 raw 재해석한다 — Q4_0 버퍼에 as_slice::<BlockQ4_0>()를 쓰는 것이 합법인 이유이자, 잘못된 T를 주면 조용히 garbage가 되는 이유다.

근거: engine/src/tensor.rs::Tensor (struct, L8–12; as_slice L56)

2.1 수명 보증(keep-alive) 패턴

view형 버퍼들은 일관되게 "raw 포인터 + backing을 살리는 Arc 필드" 패턴을 쓴다. MmapBuffer는 직접적인 Arc<memmap2::Mmap> 대신 marker trait MmapKeepAlive(메서드 0개)를 통해 타입을 지운다 — primary 로드에서는 raw mmap이, secondary swap에서는 wrapper SecondaryMmap이 keep-alive로 들어오는데, Arc<dyn Any>와 달리 opt-in한 타입만 keep-alive 자격을 갖게 제한한다(주석의 INV-143). 같은 취지로 memory/secondary.rs는 L2 buffer가 L3 concrete 타입(SecondaryMmap, RpcmemLayerRegion)을 import하지 않도록 SecondaryMmapBytes(바이트 접근 1개)와 RpcmemRegionGuard(순수 drop-ordering anchor) 두 trait을 둔다.

근거: engine/src/memory/host/mmap.rs::MmapKeepAlive (trait, L15), engine/src/memory/secondary.rs::SecondaryMmapBytes/RpcmemRegionGuard (L35, L46)

SliceBuffer는 cl_mem()이 항상 None이다 — raw cl_mem 핸들에는 offset을 적용할 수 없기 때문에 의도적으로 GPU 표면을 닫고 CPU 포인터 fallback을 유도한다. GPU 커널에 sub-range를 넘기려면 ClSubBuffer(clCreateSubBuffer)를 쓰거나 커널 인자로 byte offset을 넘겨야 한다. 이 비대칭이 "왜 같은 view인데 두 타입이 있나"의 답이다.

근거: engine/src/buffer/slice.rs::SliceBuffer::cl_mem (L77–83), engine/src/memory/opencl/sub.rs::ClSubBuffer::new (L45)

2.2 Memory trait: allocator 표면

pub trait Memory: Send + Sync {
    fn alloc(&self, size: usize, dtype: DType) -> Result<Arc<dyn Buffer>>;
    fn alloc_kv(&self, size: usize, dtype: DType) -> Result<Arc<dyn Buffer>> {
        self.alloc(size, dtype)   // 기본: alloc 위임
    }
    fn used_memory(&self) -> usize;
}

alloc(활성화 텐서 등 범용)과 alloc_kv(KV cache 전용)가 분리된 이유는 §3에서 보듯 OpenCL 구현이 KV에만 rpcmem·madvise 친화 정책을 적용하기 때문이다. CPU 쪽 구현 Galloc은 SharedBuffer::new 한 줄이며 used_memory()는 항상 0을 반환한다(추적 미구현 — 주석 명시).

근거: engine/src/memory.rs::Memory (trait, L34–46), engine/src/memory/galloc.rs::Galloc (L21–30)

3. Zero-copy 전략: 두 갈래와 실측 교훈

3.1 갈래 1 — UnifiedBuffer (CL_MEM_ALLOC_HOST_PTR)

UnifiedBuffer::new는 MEM_READ_WRITE | MEM_ALLOC_HOST_PTR로 driver가 host-visible하게 할당한 cl_mem을 만들고, unmapped(GPU 접근 가능) 상태로 시작한다. CPU가 만지려면 map()으로 clEnqueueMapBuffer를 타야 하며, 이때 얻은 MemMap을 std::mem::forget으로 의도적으로 누수시켜 매핑을 유지하고 raw 포인터만 Mutex<*mut u8>에 보관한다. unmap()은 명시적 clEnqueueUnmapMemObject가 아니라 queue.finish() 후 포인터를 null로 만들고 is_mapped=false로 내리는 방식이다. as_ptr()/as_mut_ptr()는 unmapped 상태에서 null을 반환한다 — UnifiedBuffer를 받은 CPU 코드가 null 포인터를 만나면 "map을 안 했다"는 신호다.

근거: engine/src/memory/opencl/unified.rs::UnifiedBuffer::new (L50–68), ::map (L71–104, mem::forget L94), ::as_ptr (L152–160)

UnifiedBuffer를 쓸지 device-only OpenCLBuffer를 쓸지는 CLI 플래그가 아니라 backend 초기화 시 자동 감지로 결정된다: CL_DEVICE_HOST_UNIFIED_MEMORY가 true이거나 디바이스 이름에 adreno/mali/powervr이 포함되면 use_zero_copy=true(UMA 모드), env FORCE_DEVICE_ONLY로 강제 비활성화할 수 있다. 따라서 Adreno에서는 활성화 텐서·weight 업로드·KV가 기본적으로 전부 UnifiedBuffer로 할당된다.

근거: engine/src/backend/opencl.rs (UMA 자동 감지, L1487–1503), engine/src/backend/opencl/memory.rs::OpenCLMemory::alloc (L108–164)

실측 교훈 (Adreno, Galaxy S25 — 코드로는 확인 불가한 운영 지식, CLAUDE.md):
  • CL_MEM_ALLOC_HOST_PTR(UnifiedBuffer)는 Adreno에서 zero-copy 효과가 없다 — driver가 내부 pool에 할당하므로 진짜 zero-copy 원천이 아니다. 진짜 원천은 rpcmem DMA-BUF interop(§3.2)이다.
  • 다만 Decode TBT 실측에서는 --backend opencl ≈ --backend opencl --opencl-rpcmem ≈ 32 ms/tok (Qwen2.5-1.5B Q4_0, n=3 median)으로 두 경로의 종단 성능이 같았다.
  • ARM UMA 캐시 비일관성 때문에 mapped 포인터 직접 접근은 stale cache를 읽을 수 있다 — CPU↔GPU 데이터 교환은 read_buffer/write_buffer 경유가 필수다(tensor partition 경로에서 실측).

3.2 갈래 2 — rpcmem DMA-BUF + CL_MEM_USE_HOST_PTR alias

--opencl-rpcmem 활성 시(OpenCLBackend::new_with_options(_, true)), RpcmemAllocator가 Qualcomm libcdsprpc.so를 dlopen하고 정확히 3개 심볼만 캐싱한다: rpcmem_alloc(heap_id, flags, size), rpcmem_free(ptr), rpcmem_to_fd(ptr). 할당은 RPCMEM_HEAP_ID_SYSTEM=25, RPCMEM_DEFAULT_FLAGS=1 고정이고 size는 ABI 제약상 i32::MAX 이하여야 한다. 반환된 host_ptr은 DMA-BUF 페이지라 CPU/GPU/DSP가 같은 물리 페이지를 공유하고, rpcmem_to_fd로 fd도 뽑아 보관한다(현 production 경로는 cl_mem alias만 사용, fd는 future graph 주입용). 호스트(non-Android) 빌드에서 new()는 즉시 Err — backend가 받아서 rpcmem을 끄고 경고 1회 출력한다.

근거: engine/src/memory/rpcmem/allocator.rs::RpcmemAllocator::new (L96–136), ::alloc (L176–213), 상수 L43–45

KV cache 할당 경로 OpenCLMemory::alloc_kv_rpcmem은 rpcmem 영역 위에 clCreateBuffer(CL_MEM_USE_HOST_PTR, size, host_ptr)로 같은 페이지를 가리키는 alias cl_mem을 만들고 둘을 RpcmemKvBuffer로 묶는다. 이 버퍼는 cl_mem()=alias(GPU 커널·read/write_buffer 호환), as_ptr()=host_ptr(eviction 등 CPU 정책 호환)을 동시에 내준다 — UnifiedBuffer와 달리 map/unmap 상태 기계가 없다(sync_device()도 no-op: Adreno UMA에서 같은 물리 페이지라 cache flush 불필요라는 게 코드의 전제). alias 생성이 실패하면 그 자리에서 rpcmem_free로 롤백하고 Err를 돌려 버퍼 단위로 UnifiedBuffer 경로로 fallback한다(INV-RPCMEM-003).

근거: engine/src/backend/opencl/memory.rs::alloc_kv_rpcmem (L65–104), ::alloc_kv (L166–218), engine/src/memory/rpcmem/kv_buffer.rs::RpcmemKvBuffer (L39–148)

Drop 순서가 안전성의 핵심이다. RpcmemKvBuffer는 Rust의 "필드 선언 순서 = drop 순서"를 이용해 ① alias cl_mem_obj가 먼저 release되고, ② 그 다음 Drop impl이 allocator.free(host_ptr)를 호출하며, ③ Arc<RpcmemAllocator> 보유로 buffer가 살아있는 동안 dlclose가 일어나지 않음(INV-RPCMEM-005)을 타입 시스템으로 강제한다. weight swap 쪽 RpcmemAliasBuffer는 반대로 host 메모리를 소유하지 않고, per-layer rpcmem region을 Arc<dyn RpcmemRegionGuard>(strong)로 pin하고 secondary store는 Weak로만 본다 — store가 alias를 캐싱하면서 생기는 Arc 자기순환을 끊기 위해서다.

근거: engine/src/memory/rpcmem/kv_buffer.rs (Drop 순서 주석 L16–26, Drop L99–108), engine/src/memory/rpcmem/opencl_alias.rs::RpcmemAliasBuffer (cycle-break 주석 L10–18, 필드 L52–59)

정리하면 zero-copy 의사결정은 두 층이다. 층 1(어디서): alloc은 UMA 감지 결과만 보고 UnifiedBuffer/OpenCLBuffer를 고른다 — rpcmem은 활성화 텐서에 절대 쓰이지 않는다(INV-RPCMEM-007, alloc이 rpcmem_allocator 필드를 무시). 층 2(KV만): alloc_kv만 rpcmem 분기를 갖는다. KV + weight-swap secondary 두 소비자가 동일한 Arc<RpcmemAllocator> 인스턴스를 주입받는다(INV-RPCMEM-002).

4. Quantization 블록 레이아웃 — 바이트 단위

도메인 환기 한 줄: block quantization은 연속 N개의 f32를 "공유 scale(+min) + 저비트 정수"로 압축하는 표현이다. 이 코드베이스에서 블록 타입은 전부 engine/src/quant.rs에 #[repr(C)]로 정의되고, 크기는 const _: () = assert!(...)로 컴파일 타임에 고정된다. 모든 블록의 원소 수는 32다 (QK4_0 = QK4_1 = QK8_0 = QK2_0 = QKKV = 32).

4.1 Q4_0 블록 18B 해부

byte offset내용
0–1d: scale, f16 little-endian. d = max|x| / 7
2–17qs[0..16]: 32개 4-bit 값. byte qs[i]의 low nibble = 원소 i, high nibble = 원소 i+16 (인접 쌍이 아니라 전반부/후반부 interleave)
 
// quantize (대칭): q ∈ [-8, 7] → +8 해서 unsigned nibble로 저장
let d  = max_abs / 7.0;
let v0 = (src[i]      * id).round().clamp(-8.0, 7.0) as i8;
let v1 = (src[i + 16] * id).round().clamp(-8.0, 7.0) as i8;
qs[i]  = ((v0 + 8) as u8) | (((v1 + 8) as u8) << 4);

// dequantize
let v0 = (b & 0x0F) as i8 - 8;   // 원소 i
let v1 = (b >> 4)   as i8 - 8;   // 원소 i+16
out[i]      = v0 as f32 * d;
out[i + 16] = v1 as f32 * d;

Q8_0은 34B = scale 2B + signed i8 32개, x ≈ qs[i] * d로 단순하다. Q4_1은 min을 더한 20B로 x ≈ nibble * d + m(unsigned nibble).

함정: nibble 배치 규약이 블록 패밀리마다 다르다. BlockQ4_0/BlockQ4_1은 llama.cpp 호환의 반분할 interleave(low=elem i, high=elem i+16)지만, KIVI 계열 BlockKVQ4는 인접 쌍 패킹이다: qs[i] = lo(src[2i]) | hi(src[2i+1]). BlockQ2_0도 인접 4개를 한 byte에 LSB부터 채운다(q << (j*2)). Q4_0 코드 패턴을 KIVI 블록에 복붙하면 원소 순서가 뒤섞인다.

4.3 Descriptor 기반 일반화: KVLayoutDesc와 OpaqueBuffer

위 블록 타입들이 "닫힌 enum 세계"라면, 동적 .so format(플러그인 KV format)을 위한 열린 세계가 KVLayoutDesc다. 4개 필드 — block_elems(블록 원소 수), bits(원소당 비트), scale_layout(None / PerBlockF16 / PerBlockF16WithMin), packing(Dense / Nibble / Byte) — 의 어휘로 블록 패밀리를 기술하고, block_bytes() = scale_bytes + quant_bytes가 바이트 회계의 단일 진실원천이 된다(q4_0 descriptor → 2 + 16 = 18B로 BlockQ4_0과 정확히 일치). 이 descriptor를 sidecar로 붙인 raw 바이트 컨테이너가 OpaqueBuffer다: Buffer::dtype()은 DType::U8을 반환해 typed 재해석을 차단하고(backend match의 U8 arm은 전부 floor/Err), 진짜 의미는 descriptor()가 운반하며 식별은 as_any().downcast_ref로 한다.

성능 보정 장치가 하나 있다: layout_desc_to_builtin_dtype()은 동적 format의 descriptor가 내장 DType과 bit-equivalent하면 opaque floor 대신 typed fast path로 라우팅한다. 인식 집합은 {F32, F16, Q4_0}뿐이다 — Q8_0/Q4_1은 typed attention이 보장되지 않아 의도적으로 제외 (opaque floor가 typed Q4_0 NEON 대비 ~1.34배 느리다는 Galaxy S25 실측이 주석에 기록돼 있다). encode_via_descriptor의 q4_0 출력은 BlockQ4_0::quantize와 byte-exact임이 테스트로 강제된다.

근거: crates/technique-api/src/lib.rs::KVLayoutDesc (L913–922, block_bytes L930), engine/src/buffer/opaque.rs::OpaqueBuffer (L29–59), engine/src/format/dtype_layout.rs::layout_desc_to_builtin_dtype (L82–87), ::dtype_to_layout_desc (L28–70)

5. Weight 경로: 디스크 → mmap → backend

5.1 GGUF 로드 (기본 경로)

GGUF 파일은 MmapOptions::new().map(&file)로 통째로 mmap되고 Arc<memmap2::Mmap>으로 보관된다. 텐서별 로드(load_raw)는 4갈래 분기다:

  1. K-quant 등 미지원 타입 → needs_dequant_fallback이면 로드 시 F32(embed/lm_head는 F16)로 dequant — 복사 발생.
  2. F32로 저장된 embed/lm_head → 로드 시 F16 downcast (GPU 업로드 절반화, RSS 최적화) — 복사 발생.
  3. Llama Q/K weight → llama.cpp의 convert reshape를 되돌리는 unpermute_qk_rows 후 SharedBuffer::from_vec로 소유 버퍼 생성 — 복사 발생.
  4. 그 외 전부(Q4_0/Q8_0 native 블록, F16/F32 norm 등) → MmapBuffer::new(mmap, abs_offset, len, dtype) zero-copy. abs_offset = tensor_data_offset + info.offset.

즉 Q4_0 weight의 18B 블록 바이트는 디스크 포맷 그대로가 CPU 텐서다. GPU backend가 대상이면 마지막에 backend.copy_weight_from(cpu_tensor)가 호출되고, 기본 구현은 copy_from 위임 — OpenCL에서는 OpenCLMemory::alloc(UMA면 UnifiedBuffer)으로 목적지를 만들고 device copy 또는 write로 업로드한다. dtype은 ggml_type_to_dtype으로 매핑되며 Q5_x, K-quant 등은 명시적 Err 또는 dequant fallback이다.

5.3 Adreno SOA 변환과 NoshuffleWeightBuffer

Adreno GEMV 커널은 AOS(18B 블록 연속)가 아니라 SOA noshuffle 레이아웃을 읽는다: nibble을 풀어 재배열한 q_buf(블록당 16B)와 scale만 모은 d_buf(블록당 2B)를 각각 column-major로 transpose한 두 버퍼다. 변환은 3단계 — (1) 블록 내 nibble bit unshuffle (q[i] = (x0 & 0x0F) | ((x1 & 0x0F) << 4), q[i+8] = ((x0 & 0xF0) >> 4) | (x1 & 0xF0)), (2) q의 ushort 단위 2D transpose, (3) d의 half 단위 2D transpose. 총 바이트는 블록당 16+2=18B로 AOS와 동일 (테스트로 불변식 강제). 이 변환은 AUF 빌드 타임(auf/q4_0_soa.rs, CPU 순수 구현)과 런타임 GPU 커널 (cvt.cl) 두 구현이 byte-identical하다.

변환 후에는 원본 AOS cl_mem이 죽은 짐이 된다(Llama 3.2 1B 기준 Q4_0 텐서 112개에 ~523 MiB — 모듈 주석 수치). 그래서 weight 텐서의 backing을 NoshuffleWeightBuffer로 교체한다: SOA 3핸들(q_buf/d_buf/q_img)을 소유하고, cl_mem()은 d_buf를 반환해 backend의 SOA registry lookup key(포인터 주소)로 쓰이게 하며, dtype()은 여전히 Q4_0을 보고해 matmul dispatch를 바꾸지 않는다. 교체 순간 옛 buffer의 Arc가 0이 되어 AOS 할당이 release된다.

근거: engine/src/auf/q4_0_soa.rs::q4_0_aos_to_adreno_soa (L27–81), ::q4_0_aos_to_soa_unshuffled (L96–116), engine/src/memory/opencl/noshuffle.rs::NoshuffleWeightBuffer (module doc L1–32, cl_mem L142–147)

6.2 Section table과 reader 검증

Section entry는 48B 고정: [0..24) tag(ASCII, NUL 패딩 — WEIGHTS_ADRENO_SOA 18자 때문에 spec의 16B에서 24B로 확장), [24..32) offset u64, [32..40) size u64, [40..44) flags u32(REQUIRED/STRIPPABLE/COMPRESSED), [44..48) version u32. 필수 section은 META(JSON), TOKENIZER, TENSOR_INDEX 3종 + backend가 요구하는 WEIGHTS_* 1종 (WEIGHTS_ADRENO_SOA / WEIGHTS_CPU_AOS / WEIGHTS_CUDA_AOS). payload는 WEIGHTS_ALIGNMENT = 65536(64KB) 정렬이다.

auf::reader::open(path, BackendTag)은 파일을 mmap하고 ①헤더 magic/major/capability 검증 → ②section table 경계·유니크 tag·flag·range 검증 → ③필수 section 확인 → ④해당 backend의 WEIGHTS_* 존재 확인 → ⑤META/TOKENIZER/TENSOR_INDEX 파싱 순으로 진행해 AufView를 만든다. 7. KV cache에서의 quant format 사용

KV 캐시의 layout(KVLayout::HeadMajor 고정)·eviction·pressure 파이프라인은 07장의 영역이다. 여기서는 바이트 회계 관점의 세 가지만 짚는다.

7.1 typed Q4_0 KV: 블록 단위 회계가 강제다

--kv-type q4_0이면 K/V 버퍼의 dtype이 DType::Q4_0이 되고, 쓰기 경로는 StandardFormat::write_inner의 비-F32 cast 분기다: F32 입력을 scratch 텐서로 backend.cast()(F32→Q4_0 양자화)한 뒤 KVCache::update로 넘긴다 — KVCache::update 자체는 cast하지 않고 입력이 이미 cache dtype이라고 전제한다(이 전제를 어기면 silent garbage라고 주석이 경고). scratch는 dynamic cache(memory=Some)에서만 lazy 할당 가능하다.

이후의 모든 이동 — update의 HeadMajor scatter, grow, shrink_to_fit — 은 backend.copy_slice(src, dst, src_off, dst_off, count)를 쓰는데, Q4_0에서는 count/offset이 element가 아니라 블록 단위다(블록당 18B). 예컨대 HeadMajor scatter는 blocks_per_head = head_dim / QK4_0, dst_head_stride = capacity * head_dim / QK4_0로 계산한다.

실제 사고 사례: shrink_to_fit()이 element 단위 count를 그대로 넘겨 Q4_0에서 ~32배(QK4_0) over-read → OOB → SIGSEGV가 났던 버그가 있었다(2026-06-09 수정). 현재 코드의 L418–420 주석이 이 사실을 박제하고 있고 SeqMajor/HeadMajor 양쪽에 블록 변환이 들어가 있다. Q4_0 KV를 만지는 모든 새 copy_slice 호출은 "이 count가 블록 단위인가"를 리뷰에서 반드시 확인하라.

근거: engine/src/pressure/standard_format.rs::write_inner (cast 분기 L259–324), engine/src/pressure/kv_cache.rs::update (Q4_0 블록 산식 L557–601), ::shrink_to_fit (블록 변환 + OOB 주석 L418–445)

7.2 GPU에는 Q4_0 KV fast-path가 없다

StandardFormat의 attention 디스패치는 kv_dtype == Q4_0 && backend.is_gpu()이면 GPU attention_gen으로 보내지 않는다 — GPU에 Q4_0 dequant-attention 커널이 없어 BlockQ4_0 raw 바이트를 float로 오독해 garbage가 되기 때문에 CPU fallback 경로를 탄다. typed-attention 완전지원 집합이 {F32, F16, Q4_0(CPU NEON)}으로 정의된 것(§4.3)과 같은 맥락이다.

근거: engine/src/pressure/standard_format.rs (L459–464 주석+분기)

7.3 opaque KV와 KIVI

동적 .so format KV는 KVCache 버퍼가 OpaqueBuffer일 때 update_opaque가 F32 입력을 encode_via_descriptor로 직접 양자화해 HeadMajor byte offset에 scatter한다(U8 type_size=1이므로 copy 단위가 byte). KIVI는 또 다른 합성이다 — KiviCache가 BlockQ2_0/BlockKVQ4/BlockKVQ8 블록 저장 + F32 residual 버퍼([kv_heads][res_cap][head_dim], res_cap은 QKKV=32의 배수)를 들고, Key는 per-channel(토큰 축으로 32개씩), Value는 per-token(head_dim 축으로 32개씩) 양자화한다. 운영 흐름은 07장 참조.

근거: engine/src/pressure/kv_cache.rs::update_opaque (L698–723), engine/src/pressure/kivi_cache.rs::KiviCache (struct doc + 필드, L182–260)
