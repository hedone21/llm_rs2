## Phase 3: 메모리 안정성 (Memory Stability)

긴 시퀀스 생성과 KV 캐시 eviction 정책의 안정성을 검증합니다.

### 결과

| 테스트 | Backend | Decode | Eviction | 완료 | RSS Start | RSS Peak | RSS End | Delta |
|--------|---------|--------|----------|------|-----------|----------|---------|-------|
| no_eviction_opencl | OpenCL | 1024 | none | O | 918MB | 4952MB | 1764MB | +846MB |
| eviction_sliding_opencl | OpenCL | 2048 | sliding | X (exit 1) | 1000MB | 4644MB | 1764MB | +764MB |
| eviction_large_prefill | OpenCL | 1024 | sliding | X (exit 1) | 697MB | 4357MB | 1788MB | +1090MB |
| no_eviction_cpu | CPU | 1024 | none | O | 1230MB | 4330MB | 1737MB | +506MB |

### 분석

**RSS delta가 크게 나타나지만 이는 메모리 누수가 아닙니다.** RSS(Resident Set Size)에는 모델 가중치(~1.1GB), KV 캐시, OpenCL 버퍼(`CL_MEM_ALLOC_HOST_PTR`로 매핑된 GPU 버퍼)가 모두 포함됩니다. RSS Start가 모델 로딩 전이고 RSS End가 모델 로딩 후이므로 delta는 정상적인 메모리 할당을 반영합니다.

**핵심 발견사항**:
- **eviction이 없는 OpenCL/CPU 추론은 정상 완료** (exit 0)
- **eviction 정책(sliding)이 적용된 2개 테스트는 크래시** (exit 1) — 이는 eviction 로직에 잠재적 버그가 있음을 시사
- 크래시에도 불구하고 시스템 OOM은 발생하지 않았으며 디바이스는 안정적
- 모든 테스트에서 RSS Peak ~4.3-5.0GB로, 12GB RAM 대비 충분한 여유

**주의**: sliding window eviction의 크래시는 별도 조사가 필요합니다.
